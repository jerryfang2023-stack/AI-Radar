**中文翻译**

> 我最近一直在思考一件事：过去，我认识的最优秀的工程师会花大量时间以各种方式自动化他们的工作。比如优化 vim/emacs 的自动化设置，编写 lint 规则来捕捉重复的代码问题，构建一套端到端测试，这样就不用手动进行冒烟测试。这类事情是工程师能做的最高杠杆活动，因为它们能倍增自己的产出，进而让他们能构建更多东西。
>
> 我认为，如今许多这类自动化变得更加重要了。这有几个原因。第一，基础设施和开发者体验的自动化能让你提速。如果你在运行一个代理大军，每个代理也会因此提速。更多的自动化 == 单位时间内更多的产出。第二，将事务转化为代码能提高效率。你的代理可以在每次看到某个问题时去修复它，但这会消耗 token，还可能遗漏一些情况。如果让 Claude 编写一条 lint 规则、一个 CI 步骤或一个例行程序，那整类问题就可以被永久性地完全自动化。这其实就是人们谈论“循环”时所指的——是自动化掉整类繁琐工作，而不是逐一解决。这根本不是什么新想法。工程师们已经这样做很久了！
>
> 第三，也是最重要的一点，自动化让其他人能更轻松地为代码库做出贡献。我越来越多地看到，工程师在第一天就能为代码库做贡献，因为 Claude 可以为他们导航代码库；而非工程师也能像工程师一样有效地为代码库做贡献。阻碍这两者的，是存在于人们头脑中而非自动化中的领域知识——也就是你过去在适应阶段必须学习的东西。

> 原始观点/摘要：Something I have been thinking about: in the past, the best engineers I knew spent a lot of time automating their work in various ways. Better vim/emacs automations, writing lint rules to catch repeat code issues, building up a suite of e2e tests so they don't need to smoke test the app manually. These kinds of things were the highest leverage activities an engineer could do, because it multiplied their own output, which in turn meant they could build more things. I think many of these automations have become even more important now. This is true for a number of reasons. First, infra and DevX automation speeds you up. And if you are running an army of agents, each of those agents will be sped up also. More automation == more output per unit of time. Second, moving things to code improves efficiency. Your agent could fix an issue every time it sees that issue happen, but that uses tokens and might miss cases. If Claude instead writes a lint rule, CI step, or routine, that class of issue can be fully automated forever. This is really what people are talking about when they talk about loops -- it's about automating entire types of busywork rather than solving them one off. This isn't a new idea at all. Engineers have been doing this for a long time! Third and most importantly, automation makes it possible for others to contribute to the codebase more easily. Increasingly what I am seeing is engineers are contributing to codebases on day one because Claude can navigate the codebase for them, and that non-engineers are able to contribute to a codebase as effectively as engineers can.
