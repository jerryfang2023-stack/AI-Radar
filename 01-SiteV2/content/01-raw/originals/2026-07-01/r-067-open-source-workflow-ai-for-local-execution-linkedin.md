At what point are you comfortable with deploying an AI workflow to production?
This is a question that every tech leader needs to be able to answer, but it's not an easy one. For starters, ask yourself these questions:
- How consistent are my outputs (i.e. if the workflow runs multiple times, do the outputs change)? Are any inconsistencies within appropriate bounds?
- Does this workflow consume an appropriate amount of resources (tokens, GPU, wall clock time, etc)?
- Do I have error handling for all edge cases (invalid input, API errors, heavy load, etc)?
- Are there validation mechanisms on your inputs and outputs? How do you know that you aren't at risk for prompt injection, exposure of PII, or just delivering a bad user experience?
If you can't answer one or more of these questions, then the chances are pretty good that your workflow is not ready to ship.

Released glm5.1-distill on Hugging Face .
It's a 1.2B parameter chat model built on Liquid AI LiquidAI/LFM2.5-1.2B-Base, supervised-fine-tuned on 50k conversations from Jackrong/GLM-5.1-Reasoning-1M-Cleaned. The goal was to bring some of the conversational reasoning behavior of larger Z.ai GLM-5.1 teacher models into a small, edge-friendly student that fits on a single consumer GPU.
Training setup:
- Method: LoRA SFT via Unsloth AI , then merged back to a single 16-bit checkpoint
- Rank 16, alpha 16, train_on_responses_only
- 1 epoch, effective batch size 32, packed at seq length 2048
- AdamW, lr 2e-5 linear with 50 warmup steps, BF16
- Single GPU, fits comfortably on a free Colab T4
The LFM2 hybrid architecture (short conv blocks plus grouped query attention) keeps inference fast and memory low, which is why I picked it as a base instead of a standard transformer.
This is a starting checkpoint, not a finished assistant. It needs evaluation on real benchmarks (MMLU, GSM8K, IFEval, MT-Bench) and probably another fine-tuning pass for any specific domain. Treat it as a base to build on.
Model: https://lnkd.in/d3TCk936

Your LLMs are silently degrading, and your APM won't catch it. Welcome to "AI Shrinkflation."
Providers are continuously tweaking reasoning depths, safety layers, and routing pools to manage their own compute costs. The API still returns a 200 OK, but behind the scenes, your workload is secretly degrading.
Case in point: A recent analysis of 234,000+ tool calls showed a sudden 67% drop in an LLM's thinking depth. The result? A 122x cost multiplier due to agent thrashing, retries, and premature stops.
> Better prompts won't fix this. You need to architect for drift:
- Stop direct SDK calls: Build strict abstraction layers so you can swap models instantly.
- Golden Evals: Run representative workflows daily to catch behavioral degradation before users do.
- Harness Guards: Enforce read-before-edit rules and strict cost-per-task alerts to catch spinning agents.
Treat LLMs as swappable, volatile dependencies, not permanent fixtures.

Recently, I’ve been using OpenClaw to gradually delegate repetitive and time-consuming tasks to AI.
As I automated tasks one by one, I realized that many workflows that previously took 1–2 hours can now be handled effectively by today’s AI.
By automating tasks such as news aggregation, scheduling, and Excel data management, I’ve been able to free up roughly 3 hours per day.
Initially, I experimented with subscription-based APIs, but with a usage of 1–2 million tokens per day, the cost quickly became unsustainable. $20 a day, no joke. Unfortunately, I don't have that kind of money.
This led me to shift toward a local setup. Considering power consumption and noise constraints, I chose a DGX Spark-based environment. I don't want to give my colleagues the gift of hot wind and noise.
And I ultimately adopted the MSI EdgeExpert, which uses the same nvidia's gb10 chipset but offers better thermal control and a significantly lower price point. I mean cheaper, that's the point. I am not a rich person who spends an extra $1.5k to increase it by 3TB. And the EdgeExpert 1TB is also $3,000. If I hadn't make money from stocks, I wouldn't even dare to dream of that price. Thank you, Samsung and SK Hynix.
Put the chatter aside, After purchasing the PC and setting up OpenClaw, I installed GPT OSS 120B. I chose this model because I don't mind if the response is a bit slow, but I want to be less dumb.
While running GPT OSS 120B locally has some performance drawbacks compared to commercial models, a major advantage is that it can be operated without ongoing costs.
Additionally, utilizing CUDA was beneficial because it enabled large-scale data preprocessing, particularly through tools like cuDF. Since I also handle tasks analyzing reusable cup and bowl's status data at work, processing tens of millions of rows of data accumulated over four years used to take far too long with conventional CPU computations.
I’m still in the learning phase, but I believe AI-driven workflow automation has the potential to create a shift comparable to the introduction of Excel.

From LLM 'Brain' 🧠 to the 'Nervous System'
In the shift from chatbots to autonomous agents - one major hurdle is Resource Management. If you run several agents at once, they are likely to lose memory, fight for tokens, and crash the context window.
The paper AIOS: LLM Agent Operating System: https://lnkd.in/gejFFX2S offers a brilliant fix. It treats LLM like a CPU and builds an AI Kernel around it. I found this quite insightful as it applies classic OS logic (scheduling, memory management, and storage) to AI. The AIOS Kernel:
✅ Schedules tasks so no single agent hogs the LLM
✅ Context-switches smoothly, preserving 'thoughts' without wasting tokens
✅ Secures the boundaries between different AI tools
Architecture like AIOS is what will make autonomous agents highly reliable.
If you’re interested in the plumbing that will power AGI, this is a must-read.

- Method: LoRA SFT via Unsloth AI , then merged back to a single 16-bit checkpoint
- Rank 16, alpha 16, train_on_responses_only
- 1 epoch, effective batch size 32, packed at seq length 2048
- AdamW, lr 2e-5 linear with 50 warmup steps, BF16
- Single GPU, fits comfortably on a free Colab T4
The LFM2 hybrid architecture (short conv blocks plus grouped query attention) keeps inference fast and memory low, which is why I picked it as a base instead of a standard transformer.
This is a starting checkpoint, not a finished assistant. It needs evaluation on real benchmarks (MMLU, GSM8K, IFEval, MT-Bench) and probably another fine-tuning pass for any specific domain. Treat it as a base to build on.
