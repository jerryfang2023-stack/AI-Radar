Most network teams think they have a documentation problem. What they actually have is a source of truth problem, and the two are not the same.
Documentation is something you write once and stop trusting six months later. A source of truth is the system the network is built from. The one place where the IP plan, the VLANs, the circuits, the device roles and the cabling actually live, and where automation reads its intent before anything touches a router.
When that single source does not exist, you get the failure mode every operator knows. Two engineers hold different versions of reality in their heads. The spreadsheet says one thing, the device says another, and the wiki was last accurate in 2023. Every change becomes an investigation before it becomes a change.
How many of your incidents actually start as a routing or hardware fault, versus a wrong assumption about what was deployed?
For years we have built our automation around NetBox Labs platform as that source of truth. Python and Jinja2 render config from it, CI/CD pipelines validate against it, monitoring syncs from it. It is the layer that lets a multi-vendor network behave like one system instead of a pile of snowflakes.
So this step is overdue. ITcare

The second MCP pilot team is where enterprise rollout usually breaks.
The first team got the servers connected.
The second team asks for the same access across a shared workflow.
That is when the story stops being about integration and starts being about authorization.
If every employee still has to authorize every MCP server one by one, the organization is trying to scale a consumer consent pattern into shared enterprise infrastructure.
The new enterprise-managed authorization work in MCP matters for exactly that reason.
It moves server access into the identity layer the business already uses for onboarding, offboarding, groups, roles, and conditional-access policy.
But this is the important distinction:
centralized authorization does not remove approval boundaries.
OpenAI still defaults MCP and connector calls to approval before data is shared with a remote server.
That means one control decides who may reach the server at all.
Another control still decides whether a specific action should proceed.
Before broad shared rollout, require one authorization rollout sheet for each production workflow:
- client surface
- MCP server set
- IdP and group mapping
- approval boundary
- revocation owner
- audit sink
- unsupported client fallback
If the organization cannot show that evidence, it has connected more tools.
It has not finished governing shared access.
