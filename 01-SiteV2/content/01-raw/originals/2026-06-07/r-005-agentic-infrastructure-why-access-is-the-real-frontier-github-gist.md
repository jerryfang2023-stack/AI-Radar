Agentic Infrastructure: Why Access Is the Real Frontier
The agentic AI market tripled to $3.8 billion in 2024. The number of agent companies exploded from roughly 300 to thousands in under twelve months. One in five new unicorns now develops agents. Broader AI investment hit $225.8 billion in 2025, doubling the prior year. The sheer weight of capital flowing into this space creates an impression of maturity. It should create suspicion instead.
A structural problem persists beneath the funding charts: agents that reason brilliantly but cannot act. An agent can draft a marketing campaign with precise targeting, audience segmentation, and creative copy. It cannot send the SMS to execute it. It can design a landing page with optimized conversion elements. It cannot provision the image generation API to populate it. It can plan infrastructure deployment with detailed architecture diagrams. It cannot buy the compute to run it.
The equation governing agentic value is multiplicative, not additive:
Agentic Value = Reasoning x Access x Efficiency
The industry has invested billions into the first variable. Almost nothing has gone into the second. And the third, efficiency, determines whether access infrastructure is economically viable at scale.
This week delivered a signal that investors have noticed the imbalance. Sapiom, an access layer for autonomous agents, announced a $15.75M seed round led by Accel, with participation from Anthropic, Coinbase, Okta, Menlo Ventures, and strategic angels from OpenAI, GitHub, Shopify, and Vercel. In the same week, Linkup raised funding for real-time web search infrastructure purpose-built for agents. Two companies, both building agent access infrastructure, both raising in the same window. The signal is not subtle.
Knowledge graphs sit at the center of this transformation. They are the technology that makes both internal and external access efficient, structured, and reliable enough for autonomous action. Understanding why requires examining what "access" actually means, where it breaks down, and how graph-based architectures solve problems that no other approach can.
The Access Gap
Ilan Zerbib, Sapiom's founder, frames the problem with a metaphor that has stuck across the industry: "Intelligence without access is a brain in a jar." The metaphor captures something precise. Current AI agents operate in a paradox. They possess sophisticated reasoning capabilities trapped behind human-mediated bottlenecks at every point where reasoning meets the real world.
The bottlenecks are specific and enumerable. API key provisioning requires a human to create credentials, share them with an agent, define access rules, manage permissions, and configure billing. OAuth flows were designed for human browser sessions, not machine-to-machine authentication at scale. Purchase order approval chains assume a human decision-maker at each financial commitment. Vendor onboarding processes require paperwork, compliance checks, and contractual review that agents cannot navigate.
Sapiom's thesis treats capability as the product of reasoning and access. If access stays near zero, trapped behind static integrations and human approval flows, effective capability stays near zero regardless of how sophisticated the reasoning becomes. A model that scores 95% on reasoning benchmarks but can only access 3 out of 100 services it needs delivers 2.85% of its potential value.
The obvious counter-argument: the integration industry is not small. Zapier, Make, MuleSoft, and hundreds of integration platforms have raised billions collectively. LangChain, CrewAI, and a growing ecosystem of orchestration frameworks provide tool-use abstractions. Anthropic's Model Context Protocol standardizes how agents connect to data sources. This represents real investment in connecting systems.
The distinction that matters is between integration and access. Integration is human-triggered: a developer configures a connection, sets up authentication, defines data mappings, and deploys a workflow. Access is agent-autonomous: an agent discovers a service it needs, authenticates to it, negotiates terms, pays for usage, and manages the relationship, all without human intervention at each step. Integration solves the problem of connecting known systems in advance. Access solves the problem of agents operating in environments where the required services are unknown until runtime.
This distinction reveals two dimensions of the access problem. External access concerns how agents reach the broader digital economy: web data, third-party APIs, payment systems, and commercial services. Internal access concerns how agents navigate enterprise data: knowledge bases, operational systems, CRM records, and organizational context. Both dimensions require infrastructure. Both remain largely unsolved.
External Data Infrastructure
The external data problem bifurcates into two distinct markets that enterprises too often conflate. Agents that need current information with acceptable accuracy require real-time search. Agents that need verified information for business-critical decisions require pre-verified data infrastructure. Treating these as the same market leads to solutions that satisfy neither requirement.
Linkup occupies the real-time search position. Built from scratch for agent consumption rather than adapted from human search, the company provides an API and MCP server that delivers structured, agent-optimized search results. Linkup achieves a 91% SimpleQA Verified F-Score, matching the top tier of competitors. The architecture uses custom ML classifiers for page refresh frequency and domain indexing decisions, embedding models for vectorization, and a real-time indexing pipeline that processes web content for agent retrieval. Enterprise customers include KPMG, University of Glasgow, and Chanel. Integration requires five lines of code. The company projects compute costs approaching $1 million monthly by end of 2026, reflecting the infrastructure-scale commitment required.
Sacha Uzan, Linkup's CEO, frames the company's thesis around a foundational claim: "The internet will remain a primitive." Despite the rise of proprietary data stores, RAG systems, and knowledge graphs, the open internet remains the most comprehensive real-time knowledge source. Agents will treat web search as a first-class capability.
Cala AI attacks the same problem from the opposite direction. Elisenda Bou-Balust just launched Knowledge Query, a product that redefines how agents access external facts. Bou-Balust's trajectory explains why Cala exists. She founded Billings in 2012 — a pre-LLM era company that built custom NLP, computer vision, and speech-to-text models for multimodal content understanding. Billings built knowledge graphs from YouTube videos and pivoted to media companies, securing data access agreements with NBC, Fox, CBS, and ABC News. Apple acquired Billings in 2020. Bou-Balust led development of Apple's knowledge graph, the third largest globally. She founded Cala in October 2024 with co-founder Issey Masuda Mora, bringing over a decade of production knowledge graph experience to the agent infrastructure problem.
Her thesis: the web was built for humans who click around. Agents need to walk through a graph, not dig through a graveyard of HTML. Knowledge Query makes this concrete. One query: OpenAI.competitors.funding_rounds.lead_investors . Or startups.location=Spain.funding>10M.founders.previous_experience — multi-hop filtered relational traversal returning structured, schema-validated results in seconds. No scraping, no parsing, no stitching 15 sources together.
The technical architecture separates online and offline processing with a specific rationale. Graph databases handle complex offline operations: fact-checking, computing relationships, validating implications across hundreds of related triples. This is where the expensive multi-hop reasoning happens — entity disambiguation across sources, cross-hopping analysis that navigates knowledge graph relationships, and ontological constraint verification. Elasticsearch and S3 handle fast online serving with pre-computed query patterns, avoiding expensive graph database operations for common 1-2 hop queries. This separation allows unlimited time for thorough verification offline while serving pre-validated results at millisecond latency. Most data sources refresh on a 24-hour cycle, with critical sources updating hourly.
Cala's two-agent verification system provides the quality layer that distinguishes pre-verified data infrastructure from search. Agent 1 performs reputational scoring — like PageRank but for facts, not links. It counts how many facts from a source match already-verified knowledge in the graph, producing a trustworthiness score for each source. Agent 2 conducts deep graph-based validation, temporarily inserting new facts into the knowledge graph and checking implications across hundreds of connected triples for probabilistic consistency. When sources conflict, the system handles the ambiguity explicitly: less reputable sources get corrected, but when multiple reputable sources disagree, both versions are stored with metadata tracking recency, source authority, and contextual factors. This is not binary fact-checking. It is probabilistic truth maintenance with provenance.
The knowledge graph itself uses dynamic property graphs with domain-dependent hybrid ontologies rather than a fixed schema. Self-learning ontologies through curriculum learning start with foundational domain knowledge, progressively learn more complex relationships, and automatically compute trustworthiness for new sources. Each new source is easier to evaluate because the existing knowledge base provides reference points. The system compounds: more verified facts make it more accurate at verifying the next fact.
When a compliance team needs verified regulatory data, they do not want an agent pulling a stale webpage. They want pre-verified, graph-validated facts with full provenance and confidence scores.
The practical difference between the two approaches maps to enterprise decision types. A sales team researching competitor activity tolerates 91% accuracy in exchange for real-time results. A compliance team validating regulatory requirements for automated workflows demands higher confidence with full provenance tracking. An agent populating CRM records with company data needs verified facts from primary sources, cross-checked across multiple databases, with confidence scores for each field. An agent monitoring reputation risk needs hourly updates from the broadest possible web surface.
Most enterprises will need both. The single-vendor assumption is a trap. A unified interface that routes queries based on characteristics (high-stakes to verified sources, high-recency to real-time sources) with fallback logic and confidence-weighted merging represents the practical architecture.
Sapiom adds the transactional layer on top of both. It is one thing for an agent to discover and retrieve information. It is another for that agent to autonomously authenticate to a service, negotiate access terms, and pay for API usage. Sapiom abstracts API key management, permission scoping, billing configuration, and vendor onboarding into a single interface designed for machine-speed operation. The investor syndicate tells the story: Accel (enterprise SaaS), Anthropic (foundation models), Coinbase (crypto payments), Okta (identity and authentication), Shopify and Vercel and GitHub (developer platforms). Each maps to a specific capability an access layer requires.
Internal Data Infrastructure
External data is half the equation. The other half lives inside the enterprise, fragmented across operational systems that were designed for human navigation. Glean's approach to this problem demonstrates why knowledge graphs are foundational infrastructure, not optional enhancements.
Arvind Jain called context graphs a "trillion-dollar opportunity" this week. He is building on that thesis. Glean's context graph connects people, content, and systems to time-ordered traces of actual work. Not final states in a CRM. The temporal sequence of edits, messages, status changes, and meetings that reveal how an organization actually operates.
David Huynh, Pradeep Vaghela, and Julie Mills just published the engineering blueprint in detail. Their architecture is a 5-step pipeline that reveals how context graphs work at production scale.
Step 1: Deep connectors observe every enterprise system — CRM, ticketing, chat, docs, email, calendars, code repositories, dashboards, and internal apps. The system captures all change events, not just document snapshots, normalized as temporal traces. Step 2: An ML pipeline infers higher-level entities (projects, customers, products, teams) and performs cross-system entity resolution — recognizing that "ACME Inc" in CRM and "ACME" in support tickets refer to the same entity. Activity signals like views, edits, and comments reveal actual usage patterns beyond what document metadata captures. Step 3: Personal graphs construct a chronological sequence of actions per employee across all tools. LLMs group low-level events into semantic tasks ("investigating an alert," "drafting a spec") using shared signals like titles, links, and meeting invites. Step 4: Personal graphs are anonymized and aggregated into organizational patterns using k-anonymity thresholds — patterns must appear across at least k distinct users and n independent traces before entering the collective graph. Each pattern captures action type (view, edit, comment, escalate), tool family (docs, chat, tickets, code), knowledge graph entities (incident type, product, service), derived process tags from LLMs or heuristics, and timing features with outcomes. Step 5: Agent execution traces feed back into the graph. Every tool call, its order, inputs, outputs, success metrics, and user feedback become new traces. Offline replay and alternative route scoring reinforce optimal patterns. The result is a joint model of human and agent behavior that improves with every execution.
The architecture uses a hybrid storage model: free-form text broken into chunks with embedded entity IDs ( incident_id , channel_id , action_type ). This provides clear guideposts for agents to walk process steps while maintaining reasoning flexibility — a pragmatic compromise between pure graph traversal and free-form retrieval.
Two critical design choices emerge from the blueprint. First, actions are first-class entities with timestamps and metadata, not merely tracking what exists. Traditional knowledge graphs model state. Context graphs model change. Second, the context graph and orchestration layer must be co-owned — separating them creates drift between what the graph knows and what the agent does.
Before shipping externally, Glean tested the system on their own workflows with employee opt-in: AE mid-market deal cycles, SE proofs-of-concept, on-call incident response, and PM feature launches. The dogfooding revealed specific workflow types where the gap between perceived and actual process is measurable — and where emergent playbooks from the graph outperform manually authored SOPs.
The results from Glean's Agentic Engine 2 quantify what graph-powered context provides. The system achieves 94.1% task completeness. Performance on challenging queries improved by over 20 percentage points, from 68.2% to 88.8%. The engine makes 2.5x more calls to the Enterprise Graph per query than its predecessor. Legal contract redlining went from two weeks to one minute plus two hours of human review.
The performance data reveals a specific insight about agent failures. Agents fail at planning, not execution. They are given a goal and a set of tools but lack the deep contextual understanding to create an effective plan. They execute faithfully against a plan built on an incomplete model of the enterprise. The graph fixed that by providing emergent playbooks: patterns that arose from thousands of observed interactions, not from manual programming. Agents that consistently consult the knowledge graph create plans grounded in how the organization actually works.
A data flywheel compounds the advantage. Every human interaction teaches the graph which documents answer which questions, who the domain experts are, what content is authoritative. More agent usage generates more graph improvements, which generate better agent performance, which drives more usage. The 21% increase in user adoption following the graph-powered engine reflects this directly: reliability drives usage.

Billings built knowledge graphs from YouTube videos and pivoted to media companies, securing data access agreements with NBC, Fox, CBS, and ABC News. Apple acquired Billings in 2020. Bou-Balust led development of Apple's knowledge graph, the third largest globally. She founded Cala in October 2024 with co-founder Issey Masuda Mora, bringing over a decade of production knowledge graph experience to the agent infrastructure problem.
Her thesis: the web was built for humans who click around. Agents need to walk through a graph, not dig through a graveyard of HTML. Knowledge Query makes this concrete. One query: OpenAI.competitors.funding_rounds.lead_investors . Or startups.location=Spain.funding>10M.founders.previous_experience — multi-hop filtered relational traversal returning structured, schema-validated results in seconds. No scraping, no parsing, no stitching 15 sources together.
The technical architecture separates online and offline processing with a specific rationale. Graph databases handle complex offline operations: fact-checking, computing relationships, validating implications across hundreds of related triples. This is where the expensive multi-hop reasoning happens — entity disambiguation across sources, cross-hopping analysis that navigates knowledge graph relationships, and ontological constraint verification. Elasticsearch and S3 handle fast online serving with pre-computed query patterns, avoiding expensive graph database operations for common 1-2 hop queries. This separation allows unlimited time for thorough verification offline while serving pre-validated results at millisecond latency. Most data sources refresh on a 24-hour cycle, with critical sources updating hourly.
Cala's two-agent verification system provides the quality layer that distinguishes pre-verified data infrastructure from search. Agent 1 performs reputational scoring — like PageRank but for facts, not links. It counts how many facts from a source match already-verified knowledge in the graph, producing a trustworthiness score for each source. Agent 2 conducts deep graph-based validation, temporarily inserting new facts into the knowledge graph and checking implications across hundreds of connected triples for probabilistic consistency. When sources conflict, the system handles the ambiguity explicitly: less reputable sources get corrected, but when multiple reputable sources disagree, both versions are stored with metadata tracking recency, source authority, and contextual factors. This is not binary fact-checking. It is probabilistic truth maintenance with provenance.
The knowledge graph itself uses dynamic property graphs with domain-dependent hybrid ontologies rather than a fixed schema. Self-learning ontologies through curriculum learning start with foundational domain knowledge, progressively learn more complex relationships, and automatically compute trustworthiness for new sources. Each new source is easier to evaluate because the existing knowledge base provides reference points. The system compounds: more verified facts make it more accurate at verifying the next fact.
When a compliance team needs verified regulatory data, they do not want an agent pulling a stale webpage. They want pre-verified, graph-validated facts with full provenance and confidence scores.
The practical difference between the two approaches maps to enterprise decision types. A sales team researching competitor activity tolerates 91% accuracy in exchange for real-time results. A compliance team validating regulatory requirements for automated workflows demands higher confidence with full provenance tracking. An agent populating CRM records with company data needs verified facts from primary sources, cross-checked across multiple databases, with confidence scores for each field. An agent monitoring reputation risk needs hourly updates from the broadest possible web surface.
Most enterprises will need both. The single-vendor assumption is a trap. A unified interface that routes queries based on characteristics (high-stakes to verified sources, high-recency to real-time sources) with fallback logic and confidence-weighted merging represents the practical architecture.
Sapiom adds the transactional layer on top of both. It is one thing for an agent to discover and retrieve information. It is another for that agent to autonomously authenticate to a service, negotiate access terms, and pay for API usage. Sapiom abstracts API key management, permission scoping, billing configuration, and vendor onboarding into a single interface designed for machine-speed operation. The investor syndicate tells the story: Accel (enterprise SaaS), Anthropic (foundation models), Coinbase (crypto payments), Okta (identity and authentication), Shopify and Vercel and GitHub (developer platforms). Each maps to a specific capability an access layer requires.
Internal Data Infrastructure
External data is half the equation. The other half lives inside the enterprise, fragmented across operational systems that were designed for human navigation. Glean's approach to this problem demonstrates why knowledge graphs are foundational infrastructure, not optional enhancements.
Arvind Jain called context graphs a "trillion-dollar opportunity" this week. He is building on that thesis. Glean's context graph connects people, content, and systems to time-ordered traces of actual work. Not final states in a CRM. The temporal sequence of edits, messages, status changes, and meetings that reveal how an organization actually operates.
David Huynh, Pradeep Vaghela, and Julie Mills just published the engineering blueprint in detail. Their architecture is a 5-step pipeline that reveals how context graphs work at production scale.
Step 1: Deep connectors observe every enterprise system — CRM, ticketing, chat, docs, email, calendars, code repositories, dashboards, and internal apps. The system captures all change events, not just document snapshots, normalized as temporal traces. Step 2: An ML pipeline infers higher-level entities (projects, customers, products, teams) and performs cross-system entity resolution — recognizing that "ACME Inc" in CRM and "ACME" in support tickets refer to the same entity. Activity signals like views, edits, and comments reveal actual usage patterns beyond what document metadata captures. Step 3: Personal graphs construct a chronological sequence of actions per employee across all tools. LLMs group low-level events into semantic tasks ("investigating an alert," "drafting a spec") using shared signals like titles, links, and meeting invites. Step 4: Personal graphs are anonymized and aggregated into organizational patterns using k-anonymity thresholds — patterns must appear across at least k distinct users and n independent traces before entering the collective graph. Each pattern captures action type (view, edit, comment, escalate), tool family (docs, chat, tickets, code), knowledge graph entities (incident type, product, service), derived process tags from LLMs or heuristics, and timing features with outcomes. Step 5: Agent execution traces feed back into the graph. Every tool call, its order, inputs, outputs, success metrics, and user feedback become new traces. Offline replay and alternative route scoring reinforce optimal patterns. The result is a joint model of human and agent behavior that improves with every execution.
The architecture uses a hybrid storage model: free-form text broken into chunks with embedded entity IDs ( incident_id , channel_id , action_type ). This provides clear guideposts for agents to walk process steps while maintaining reasoning flexibility — a pragmatic compromise between pure graph traversal and free-form retrieval.
Two critical design choices emerge from the blueprint. First, actions are first-class entities with timestamps and metadata, not merely tracking what exists. Traditional knowledge graphs model state. Context graphs model change. Second, the context graph and orchestration layer must be co-owned — separating them creates drift between what the graph knows and what the agent does.
Before shipping externally, Glean tested the system on their own workflows with employee opt-in: AE mid-market deal cycles, SE proofs-of-concept, on-call incident response, and PM feature launches. The dogfooding revealed specific workflow types where the gap between perceived and actual process is measurable — and where emergent playbooks from the graph outperform manually authored SOPs.
The results from Glean's Agentic Engine 2 quantify what graph-powered context provides. The system achieves 94.1% task completeness. Performance on challenging queries improved by over 20 percentage points, from 68.2% to 88.8%. The engine makes 2.5x more calls to the Enterprise Graph per query than its predecessor. Legal contract redlining went from two weeks to one minute plus two hours of human review.
The performance data reveals a specific insight about agent failures. Agents fail at planning, not execution. They are given a goal and a set of tools but lack the deep contextual understanding to create an effective plan. They execute faithfully against a plan built on an incomplete model of the enterprise. The graph fixed that by providing emergent playbooks: patterns that arose from thousands of observed interactions, not from manual programming. Agents that consistently consult the knowledge graph create plans grounded in how the organization actually works.
A data flywheel compounds the advantage. Every human interaction teaches the graph which documents answer which questions, who the domain experts are, what content is authoritative. More agent usage generates more graph improvements, which generate better agent performance, which drives more usage. The 21% increase in user adoption following the graph-powered engine reflects this directly: reliability drives usage.
The semantic layer extends this architectural pattern beyond enterprise data. A unified theory of semantic layer necessity identifies three tiers required for agentic systems. The Foundation Tier provides meaning infrastructure: ontological frameworks, taxonomies, relationship models encoding significance beyond mere connectivity. The Mediation Tier handles knowledge processing: entity extraction, relationship inference, context interpretation, and selective activation of relevant knowledge. The Application Tier provides the agentic interface: prompt augmentation, response verification against semantic constraints, and learning loops that enrich the foundation.
A six-level maturity scale tracks organizational progress from basic metadata management through controlled vocabulary integration, relationship mapping, ontological framework development, and contextual knowledge integration, culminating in full meaning infrastructure where agents contribute to and evolve semantic understanding. Most organizations today sit at Level 1 or 2. Reaching Level 5 or 6 requires treating meaning as the foundation of computing, not an afterthought applied to data storage.
Graphs as the Efficiency Layer
The third variable in the agentic value equation is efficiency. Efficiency is the economic viability of access: cost per query, accuracy per dollar, time to integrate. Graphs are the architecture that makes access efficient enough to deploy at scale.
Three specific properties of graphs create this efficiency.
Relationship traversal. Flat data structures treat each fact independently. A company's revenue, employee count, industry classification, and competitor relationships exist as isolated data points. A graph encodes these as connected entities with typed, weighted relationships. When an agent needs to validate a revenue figure, it traverses the graph to check whether the number is consistent with employee count, industry benchmarks, historical growth rates, and competitor positions. This multi-hop consistency checking across hundreds of related triples catches errors that isolated fact retrieval misses. The graph structure itself becomes a source of evidence.
Structured context for planning. Glean's 2.5x increase in graph calls per query demonstrates the pattern. Agents that can traverse structured context produce better plans. Better plans mean fewer failed execution steps. Fewer failed steps mean lower compute costs, faster task completion, and higher user trust. The 21% increase in user adoption following Glean's graph-powered engine reflects this directly: reliability drives usage.
Recursive enhancement. This is the property that makes graphs uniquely powerful for agentic infrastructure. Better graphs produce better agents. But agents, through their interactions, also improve graphs. An agent that identifies a gap in the knowledge graph (a missing relationship, an outdated fact, an inconsistent triple) can flag or directly enrich the graph. This creates a compounding loop: more agent usage generates more graph improvements, which generate better agent performance, which drives more usage. The semantic layer maturity scale captures this progression formally. At Level 6, agents contribute to and evolve semantic understanding autonomously, and knowledge structures adapt to emergent meanings. The feedback loop becomes self-sustaining.
Enterprise knowledge graphs (Glean) power this loop for internal data. Fact-level knowledge graphs (Cala) power it for external data. Semantic layers (the three-tier Foundation-Mediation-Application model) provide the meaning infrastructure that ties both together. The result is an efficiency layer that reduces the marginal cost of each additional agent query, each additional data source, each additional integration.
This efficiency is what transforms access from a cost center into an infrastructure advantage. Without efficiency, scaling access means scaling cost linearly. With graph-based efficiency, scaling access means improving quality while cost grows sub-linearly. That is the difference between infrastructure that works at prototype scale and infrastructure that works at enterprise scale.
Workflow Gravity as the Moat
OpenAI's own Frontier stack diagram makes the architectural argument visible. Gokul Rajaram flagged where Systems of Record sit in that diagram: at the bottom, with 3-4 layers of context and intelligence between them and the end application. The SoR layer is color-coded white. Value is migrating upward toward AI orchestration, reasoning, and action layers. This is not speculation. It is OpenAI publishing where they believe strategic value accrues.
Saanya Ojha articulates why this matters for the companies building access infrastructure. "Obvious ideas are dead on arrival," she argues. Any pitch reducible to "What if AI did X for everyone?" sits in the path of foundation model labs that must move up-stack to justify their capital burn. Speed is table stakes, not an advantage. The key unlock is compounding insight faster than models compound capability.
Her sharpest insight: "Your moat is workflow gravity. If ripping you out requires retraining humans, rewiring incentives, or breaking routines, you have a chance." Features evaporate. Workflow integration sticks. Real gravity forms when a product becomes the system of record, the coordination layer, or the place decisions get made.
This framing explains why access infrastructure companies like Glean, Cala, and Alpic have defensibility that pure-play model wrappers do not. Glean's context graph becomes the coordination layer for enterprise knowledge. Cala's verified fact graph becomes the system of record for external business data. Alpic's MCP deployment becomes the access layer for legacy system connectivity. Each embeds into workflows at a level where removal requires rewiring how work gets done.
John Foreman, Clio's CPO, demonstrates the pattern in legal. "AI that lives only in chat boxes is a wallflower, waiting to be asked to dance," he argues. Clio's AI product Vincent spans the entire client journey — communication, scheduling, billing, timekeeping, signing, filing, data collection, document management — because it is plugged into the platform that holds the full operational context. Drafting without workflow context is commoditizing. Foundation models will produce increasingly good language. They will not do the hard work of embedding into real legal operations and the full client journey. The "last mile" of operational integration is where access infrastructure creates defensible value.
The biggest investing mistake in the current cycle, Ojha warns, is confusing inevitability with investability. Yes, AI will transform work. Yes, agents will be everywhere. But inevitability compresses returns unless paired with asymmetric ownership of a bottleneck: distribution, data, workflow control, or trust. Access infrastructure is that bottleneck.
Real-World Evidence
Theory matters less than deployment. Alpic provides a concrete case study in access infrastructure economics.
Alpic deploys MCP servers that make any API agent-callable. The value proposition is precise: legacy B2B applications in travel and hospitality have years of customer history locked behind APIs designed for human consumption. AI-native competitors built on Claude and ChatGPT are eating their market. Building custom AI integrations costs six or more months and requires DevOps expertise these companies do not have. Without agent accessibility, they become invisible to AI-powered discovery.
The numbers tell the story. Cloudbeds, a property management system with over 40,000 properties and 300-plus API endpoints, represents the archetypal target. The company has explored AWS Agentcore for agent infrastructure. The question for Cloudbeds is not whether AI agents will become a distribution channel for travel inventory. The question is how quickly they can expose that inventory through agent-callable interfaces.
Alpic reduces the answer from months to hours. Import a GitHub repository containing MCP server code, automatically build Docker containers, deploy to production with a public domain URL, and provide a JSON-RPC endpoint any LLM can call. The competitive benchmark: Merge.dev charges $65 per connection per month and requires significant configuration. Custom development takes six-plus months and requires a DevOps team. API gateways expose REST APIs but are not MCP-compatible. LangChain tools require code changes tied to a specific framework.
The pattern repeats across the travel and hospitality vertical. Shiji Group operates across 91,000 hotels with 1,200-plus API endpoints and $31 billion in transaction volume. Apaleo, an API-first hospitality cloud based in Munich, already deployed its own MCP server in September 2025. Selectour, a French travel agency network with 1,200-plus agencies, is developing MCP integration with a partner called Kleio.
The observation from field work with these companies: 60 to 80 percent of engineering effort at B2B AI startups goes toward building custom integration layers. Companies spend six months on custom integrations when infrastructure solutions exist that reduce the timeline to weeks. The economics of access infrastructure are not theoretical. They represent direct cost avoidance and competitive acceleration.
The access infrastructure pattern extends beyond travel. Any vertical with legacy systems holding valuable data faces the same dynamic. Property management, event ticketing, restaurant reservations, car rental and mobility platforms. The agent economy creates a new distribution channel. Companies that make their systems agent-accessible capture demand from the fastest-growing consumer segment in enterprise software. Companies that do not become invisible.
The Emerging Agentic Stack
The convergence of Sapiom, Linkup, Cala, Glean, and Alpic reveals a stack decomposing into distinct infrastructure layers. Each layer spawns its own category of companies. Each layer requires its own capital, its own technical expertise, and its own competitive dynamics.
Perception layer. How agents observe the world. Real-time web search (Linkup), fact-level knowledge graphs (Cala), and enterprise search (Glean) provide structured observation capabilities. Agents need to perceive both external environments and internal organizational contexts.
Cognition layer. How agents reason. Foundation models from Anthropic, OpenAI, Google, and Meta provide the reasoning engine. This layer has received the most investment and attention. It is also the layer most subject to diminishing returns without corresponding infrastructure investments.
Memory layer. How agents retain and organize knowledge. Enterprise knowledge graphs (Glean's two-layer architecture), vector databases, and persistent context systems provide structured memory. The distinction between enterprise-wide and personal memory (Glean's Enterprise Graph plus Personal Graph) reflects how human organizations actually work.
Access layer. How agents reach services and data. Sapiom provides transactional access (discovery, authentication, payment). MCP provides protocol-level access to tools and data sources. API gateways and integration platforms provide connectivity. This is the layer with the largest gap between what agents need and what infrastructure currently provides.
Action layer. How agents execute tasks in external systems. Browser automation, API calls, file operations, and system commands translate reasoning into real-world effects. This layer exists but remains fragile, with error rates compounding across multi-step action sequences.
Transaction layer. How agents manage economic relationships. Payment processing, contract management, usage metering, and billing represent the financial infrastructure required for agent-to-agent and agent-to-service commerce. Sapiom's investor base (Coinbase, Circle, Mercury) signals that this layer may involve both traditional payment rails and crypto-native approaches.
The investment signal from Sapiom and Linkup raising in the same week goes beyond coincidence. It reflects investor conviction that agent infrastructure is a distinct investment category, separate from model training, fine-tuning, and prompt engineering. The stack is differentiating. Capital is flowing to each layer independently.
The stack will not stabilize soon. Consolidation is coming. Companies operating at the access and transaction layers will merge as the boundaries between discovery, authentication, and payment blur. Perception-layer companies will face pressure from foundation model providers building native search capabilities. Memory-layer companies will compete with enterprise software incumbents extending their platforms.
The winners will be companies that become deeply embedded in agent workflows before consolidation reshapes the landscape. The parallel to cloud computing's maturation is instructive. AWS, Azure, and GCP eventually absorbed many early cloud infrastructure companies. But companies like Stripe (payments), Cloudflare (networking), and Datadog (observability) built enough adoption and switching costs to maintain independent positions. The same dynamic will play out in agent infrastructure.
Conclusion
The constraint on agentic AI is no longer model intelligence. The frontier has moved. The binding constraint is connecting models to the world they operate in: to data, to services, to payment systems, to organizational context, to the accumulated knowledge and decision logic that makes action meaningful.
The agentic value equation makes this concrete. Reasoning has reached levels where marginal improvements in model capability produce marginal improvements in real-world outcomes. Access remains near zero for most deployed agents. If access stays near zero, effective capability stays near zero regardless of how many billions flow into training the next foundation model. The multiplier cannot produce value when one of its terms is negligible.
Knowledge graphs provide the architecture that makes access efficient. Enterprise graphs (Glean) enable internal access with 94.1% task completeness. Fact-level graphs (Cala) enable external access with verified, pre-processed data that agents can trust for business-critical decisions. Semantic layers provide the meaning infrastructure that grounds both internal and external access in structured, evolving understanding.
The companies building this infrastructure today (Sapiom for transactional access, Linkup for real-time knowledge, Cala for verified facts, Glean for enterprise context, Alpic for legacy system connectivity) are laying the foundation for every agentic workflow that follows. The opportunity is not to build a smarter brain. The opportunity is to build the nervous system, the sensory organs, and the hands that let the brain interact with reality.
Build the access infrastructure now. The graph is the architecture that makes it efficient.
