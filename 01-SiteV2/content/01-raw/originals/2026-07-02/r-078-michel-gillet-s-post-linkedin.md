# Michel Gillet's Post - LinkedIn

## clean_text

Using AI to write code is the easy part. The hard and interesting part is what this means for software architecture.
Like many, I’ve been playing with AI for quite a long time. More recently, I had more time to really dig into it and use it systematically in actual development work.
What stands out to me is that the most impactful shift is not at the coding level but at the architectural level.
We’re moving from designing systems and components to designing how software gets built in the first place, with AI as part of that process.
That means thinking about how AI fits into the toolchain, how we structure tasks, how we capture knowledge, and how we enforce rules. A lot of what we used to handle through guidelines, reviews, or documentation is now moving directly into the workflow, with checks and guardrails applied continuously.
You can already see it in areas like SoC build and integration flows, where AI-driven workflows are putting real pressure on the traditional approach.
The gap between using AI occasionally and actually structuring development around it is growing fast.
If you haven’t really explored this yet, now is probably a good time.
I’ve been building a few small public projects to understand this in practice better. I’ll share links in the comments for anyone interested.
Curious how others are seeing this evolve.
