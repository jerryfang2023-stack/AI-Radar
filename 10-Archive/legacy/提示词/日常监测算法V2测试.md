# 日常监测算法 V2 测试提示词

更新时间：2026-05-05  
状态：测试管线，不替换生产 `ai-2`

## 1. 使用边界

本提示词只用于 `06-content/` 测试管线。网页展示不是当前优先级。成熟前不得写入正式 `01-Signals/`、`02-Scoring/`、`03-Trends/AI趋势总表.md` 或 `07-Opportunities/`。

## 1.1 总流程

日常监测算法 V2 必须按以下判断资产循环执行：

```text
Trend -> Signal -> Insight -> Opportunity -> MVP -> Database -> Content -> User -> Feedback -> Signal
```

对应含义：

- Trend：行业变化 / 资本流向 / 技术拐点。
- Signal：每日精选 3 条。
- Point：一线观点输入，用于共识、分歧和边界，不作为事实证据直接加权。
- Insight：观澜核心判断。
- Opportunity：1 张机会卡，回答谁付钱 / 怎么做 / 能否迁移。
- MVP：7-14 天测试机会是否成立。
- Database：机会库 / 场景库 / 项目库 / 风险库。
- Content：内参 / 朋友圈 / 海报 / 短视频。
- User：老板 / 合伙人 / 操盘手。
- Feedback：行业问题 / 企业需求 / 项目线索，回流到 Signal。

每个精选 Signal、Point 和深挖内参都必须说明自己处于这条链路的哪个位置，以及向下一层输出什么。

## 2. 每日输出

1. 原始采集 30-50 条：`06-content/01-raw/YYYY-MM-DD-raw-candidates.md`，每条必须保存本地原文到 `06-content/01-raw/originals/YYYY-MM-DD/`
2. 初筛入池 10-15 条：`06-content/02-pool/YYYY-MM-DD-signal-pool.md`
3. 结构化入库 5-8 条：`06-content/03-structured-signals/YYYY-MM-DD-structured-signals.md`
4. 精选 3 条 Signal：`06-content/04-selected-signals/YYYY-MM-DD-front-signals.md`
5. 趋势归类：`06-content/05-trend-chain/YYYY-MM-DD-trend-classification.md`
6. Insight 判断层：`06-content/06-insights/YYYY-MM-DD-insights.md`
7. Point 观点输入：可写入 `06-content/07-points/YYYY-MM-DD-points.md`
8. 深挖内参最多 1 篇：`06-content/08-opportunities/deep-dive/YYYY-MM-DD-opportunity-deep-dive.md`
9. MVP 验证计划：可写入 `06-content/09-mvp-validation/YYYY-MM-DD-mvp-validation.md`
10. 长期数据库：`06-content/10-databases/`
11. 分发派生：可写入 `06-content/11-content-distribution/`
12. 反馈回流：可写入 `06-content/12-feedback/`

## 2.1 每日交付物与字数

| 交付物 | 数量 | 建议字数 |
|---|---:|---:|
| Raw Candidates | 30-50 条 | 每条 30-80 字，另存本地原文 |
| Signal Pool | 10-15 条 | 每条 120-250 字 |
| Structured Signals | 5-8 条 | 每条 700-1200 字 |
| Selected Signals | 3 条 | 每条 1200-1800 字 |
| Insight | 1-3 条 | 每条 300-600 字 |
| Point / Builder 共识 | 0-N 条 | 每条 150-300 字 |
| Trend Classification | 全量结构化项 | 800-1500 字 |
| Opportunity Deep Dive | 0-1 篇 | 3000-6000 字 |
| MVP Validation | 0-1 个 | 600-1200 字 |
| Feedback | 可选 | 每条 100-300 字 |

深挖内参不是每日硬凑项。证据不足时必须写：

```text
今日暂无足够证据支撑深挖内参。
```

## 3. 关键规则

- 精选 3 条必须二次搜索，每条至少 3 个 S/A/B 来源。
- 深挖内参必须二次搜索，至少 5 个来源，其中至少 2 个 S 级或一手来源。
- 每条结构化 Signal 必须完成 6 维度分析。
- C 级来源只触发检索，不直接支撑高价值 Signal。
- Builder 观点只能作为观点边界，不冒充事实证据。
- Opportunity 标题不能写公司名。
- 不输出投资建议、经营命令或确定性机会判断。
- 正式标签必须使用 `agent-workflow/product/tag-taxonomy.md` 中已有 `tag_id`，新词先进入 `candidate_tags`。
- `primary_trend_label` 用于趋势标题归类，`formal_tags` 用于迁移和关系网络，两者不得混用。
- Raw 阶段必须保存本地原文；只有链接、没有本地原文档案的线索，不得进入 `01-raw`。

检索强度：

```text
初筛靠广度，精选靠评分，深挖靠二次搜索和交叉验证。
```

## 3.1 Raw 原文保存格式

Raw 列表每条必须引用本地原文档案：

```markdown
### R-001｜标题或事件名

- 原文档案：`06-content/01-raw/originals/YYYY-MM-DD/R-001-source-title.md`
- 出处：来源名称 + URL
- 来源类型：官网 / 媒体 / 投资方 / 客户案例 / 社媒 / 开源 / 招聘 / 政策 / 数据库
- 采集理由：30-80 字，说明它可能影响收入、成本、流程、岗位、资本或迁移判断。
```

本地原文档案必须包含：

```markdown
---
raw_id:
source_name:
source_url:
source_type:
captured_at:
language:
copyright_note: local research archive only
---

# 原文标题

## 原文 / 可用正文

保存网页正文、官方公告正文、社媒原帖文本、播客转写片段、招聘 JD 正文或合法可保存摘录。

## 中文摘录 / 机器翻译

可选。

## 采集备注

说明是否为一手来源、是否需要二次搜索、是否存在可信度问题。
```

## 4. Structured 入库要求

Structured 入库不是填字段。每条 5-8 条候选都必须写成可复盘的判断素材。

每条必须包含：

1. 一句话判断。
2. 事实背景。
3. 来源摘要：逐条说明来源提供的事实。
4. 商业信号解释。
5. 6 维度分析。
6. 反证 / 风险：至少 2 个。
7. 证据缺口。
8. 关联 Trend / Opportunity / Judgment Node 候选。
9. 是否进入精选 3 条，以及原因。

每条建议 700-1200 字。若只能写成几句泛泛解释，说明证据不足，应降级观察。

## 5. 精选 3 条 Signal 要求

精选 3 条不是卡片摘要，而是短深度信号文章。每条建议 1200-1800 字。

### Signal Selector Prompt

```text
You are an AI business signal selector.

From the filtered candidate signals, select the TOP 3 signals that matter for business decision-makers.

Selection rules:

- Must strongly impact revenue, cost, or core business workflow
- Must clearly indicate a trend (not a one-off event)
- Must be explainable to a business owner
- Prefer signals with strong funding or adoption evidence
- Avoid selecting 3 signals from the same sub-category

For each selected signal, output:

### Signal Title

- What happened (≤80 Chinese characters)
- Why it matters (≤100 Chinese characters)
- Business implication (≤100 Chinese characters)
- Who should care (老板 / 运营 / 投资人等)
- Source

Then add:

### Guanlan Insight（观澜判断）
1-2 sentences explaining:
"What this signal REALLY means for business"

Language: Chinese
Tone: decision-oriented, not descriptive
```

每条必须包含：

1. 标题：事件 + 商业含义。
2. 导语：150-250 字。
3. 发生了什么。
4. 为什么不是普通新闻。
5. 二次搜索补强：至少 3 个 S/A/B 来源，并说明每个来源提供的增量事实。
6. 观澜判断：指向 Trend / Opportunity / Judgment Node。
7. 反证与边界：至少 2-3 个。
8. 后续 7 / 30 / 90 天观察点。
9. 未入选候选说明。
10. 向下游输出：
    - 对应 Insight。
    - 可能支撑的 Opportunity。
    - 是否需要进入 MVP 验证。
    - 应沉淀到哪个 Database。

精选 3 条合起来应像一份“今日信号内参”，不是 3 张可视化卡片。

### 精选信号轻量二次搜索

每条入选的 3 个 Signal，都至少补充查 3 类信息：

1. 公司 / 产品官网  
确认它到底做什么，不只看媒体转述。

2. 融资或投资人信息  
确认金额、轮次、投资方、资本为什么关注。

3. 客户 / 案例 / 定价信息  
确认有没有真实商业化，不是只停留在发布会。

完成三类补充后，才可写入精选 Signal。

### Signal 输出示例

```markdown
## Signal：AI SDR继续获得资本下注

### 事实
某AI销售Agent公司融资。

### 6维度分析
- 解决问题：
- 目标客户：
- 替代流程：
- 商业模式：
- 为什么现在：
- 迁移判断：

### 趋势归属
AI Agent 正在重构销售前端。

### 相关历史信号
- 11x.ai
- Artisan
- Regie.ai

### 观澜判断
销售岗位不是消失，而是被拆成“AI执行 + 真人成交”。

### 机会卡关联
私域AI销售Agent
```

## 5.1 Insight 判断层要求

每日至少输出 1 条 Insight，最多 3 条。Insight 是观澜自己的核心判断，不是新闻总结。

每条 Insight 必须包含：

1. `insight_id`
2. `title`
3. `related_signal_ids`
4. `related_trend_ids`
5. `related_opportunity_ids`
6. `judgment`：观澜判断。
7. `supporting_evidence`：支撑事实。
8. `counter_evidence`：反证或削弱变量。
9. `what_would_confirm`
10. `what_would_weaken`
11. `next_7d / next_30d / next_90d`

Insight 必须连接 Signal 和 Opportunity。没有 Insight 的精选 Signal，不能直接跳到深挖机会。

## 5.1A Point 观点层要求

Point 是一线观点输入，不是事实证据。它用于辅助 Insight、Trend 和 Opportunity 的判断边界。

输出模板：

```markdown
## Point #001｜某某认为 Agent 将先改造企业内部流程

- 来源人物：某某
- 身份标签：创业者 / 投资人 / 研究者 / 大厂负责人
- 原始观点：一句话概括
- 观澜提炼：这句话真正指向什么趋势
- 商业含义：影响哪个行业、岗位或流程
- 关联趋势：AI Agent / AI Coding / 企业工作流
- 机会关联：是否能进入机会卡
```

使用规则：

- Point 可支持 Insight，但不能单独让一个 Signal 入选 TOP 3。
- Point 可作为 Opportunity 假设来源，但必须经过 Signal 和二次搜索验证。
- 谨慎或反对观点应写入反证和边界。

### Builder 共识聚类

当至少 3 个独立 Point 指向同一趋势时，输出 Builder 共识：

```markdown
## 今日Builder共识｜AI正在重构销售岗位

来源：
- VC A
- 创业者 B
- YC项目 C

共识提炼：
销售前端（获客、筛选、跟进）正在被AI统一替代

观澜判断：
这是一个强趋势，而不是单点观点
```

共识聚类只能增强 Insight 的观点上下文，不能替代事实证据。

## 5.2 Trend Classifier Prompt

```text
You are an AI trend classifier.

Given a signal, assign it to ONE primary trend:

- AI Sales Automation
- AI Customer Support
- AI Knowledge Base
- AI Workflow Automation
- AI Coding / Vibe Coding
- One-person Company
- AI Content Growth
- Other (specify)

Then output:

- Trend name
- Why this signal belongs here
- Related past signals (if possible)
```

每条 Signal 只能有一个 primary trend。若需要补充其他关联趋势，放在 secondary notes，不改变主归属。

## 5.3 Signal -> Trend 自动生成

核心原则：

```text
Signal 是点，Trend 是由多个高质量 Signal 反复验证出来的方向。
```

流程：

```text
Signal 入库
↓
趋势标签识别
↓
相似信号聚类
↓
趋势强度评分
↓
判断：挂载旧 Trend / 新建 Trend / 仅入库观察
↓
更新趋势阶段、热度、置信度
```

每条 Signal 入库后必须打 6 个标签：

1. 场景标签：销售 / 客服 / Coding / 知识库 / 内容增长 / 法务 / 招聘
2. 技术标签：Agent / RAG / Voice AI / Workflow / AI Coding / Multimodal
3. 商业标签：融资 / 产品发布 / 客户案例 / 定价 / 并购 / 大厂动作
4. 影响对象：老板 / 销售团队 / 客服团队 / 开发者 / 运营 / HR / 法务
5. 价值维度：增收 / 降本 / 提效 / 替代岗位 / 新产品机会
6. 迁移判断：中国可迁移 / 部分可迁移 / 暂不适合

新建 Trend 条件：

```text
过去30天相关高质量Signal >= 3条
来源类型 >= 2种
平均Signal评分 >= 7.5
能明确说明正在重构哪个岗位/流程
```

Trend Score：

```text
Trend Score =
信号数量 × 20%
+ 信号质量 × 20%
+ 来源多样性 × 15%
+ 商业化强度 × 20%
+ 迁移可行性 × 15%
+ 增长速度 × 10%
```

Trend 生命周期：

- 萌芽期。
- 起步期。
- 加速期。
- 爆发期。
- 成熟期。
- 衰退 / 泡沫期。

最终标准：

```text
它是否能帮助老板提前判断一个岗位、流程、行业或商业模式正在发生结构性变化。
```

如果不能，就不要叫 Trend，只能叫标签。

## 6. 深挖内参要求

深挖每天最多 1 篇。目标形态是内参级深度报道，不是泛泛机会卡。

### Opportunity Analyst Prompt

```text
You are an AI opportunity analyst.

From the selected signals, choose the ONE best opportunity for deep analysis.

Selection criteria:

- Repeated signal (trend, not isolated)
- Strong commercial potential
- Matches real business scenarios
- Can be tested quickly (7-14 days)

Then generate a structured Opportunity Card:

---

## Opportunity Name

## Target Customer
Be specific (industry + role)

## Core Pain Point
Concrete, not abstract

## AI Solution
Explain clearly what the AI does

## Business Model
(subscription / usage / outcome-based / service fee)

## Why Now
What changed recently that makes this possible?

## China / Chinese Market Migration
- Can it be replicated?
- If not, what can be adapted?

## MVP Plan (7-14 days)
- Step 1
- Step 2
- Step 3

## Key Risk
What might fail?

## Opportunity Score
(1-10)

---

Important:
- Avoid generic ideas
- Must be specific and actionable
- Focus on business, not technology

Language: Chinese
Tone: pragmatic, founder-level thinking
```

只有满足以下条件才写：

- 至少 1 条精选 Signal 或 2 条 Structured Signals 指向同一机会方向。
- 至少 5 个来源，其中至少 2 个 S 级 / 一手来源。
- 能明确目标客户、替代流程、付费方式和进入壁垒。
- 能写出反证与证据缺口。
- 能形成 7 / 30 / 90 天观察框架。

不满足时写：

```text
今日暂无足够证据支撑深挖内参。
```

深挖内参固定结构：

1. 标题。
2. 摘要：300-500 字。
3. 一、事件背景。
4. 二、产业语境。
5. 三、客户与痛点。
6. 四、替代流程。
7. 五、解决方案形态。
8. 六、商业模式。
9. 七、市场与竞争。
10. 八、中国市场迁移。
11. 九、反证与风险。
12. 十、行动地图。
13. 附录：来源表。
14. 证据缺口。
15. MVP 验证计划。
16. Database 沉淀清单。
17. Content 分发建议。

每篇建议 3000-6000 字。必须有事实、场景、冲突、反证和未解问题，不能只写方向判断。

### 机会卡评分系统（10分制）

核心原则：

```text
不是评估“这个机会看起来大不大”，而是评估“这个机会值不值得老板关注、是否值得进入项目库”。
```

总分公式：

```text
机会总分 = 市场价值 × 20%
        + 痛点强度 × 15%
        + 付费意愿 × 15%
        + AI适配度 × 15%
        + 迁移可行性 × 15%
        + MVP可验证性 × 10%
        + 风险可控性 × 10%
```

评级：

| 总分 | 等级 | 判断 |
|---:|---|---|
| 9.0-10 | S级机会 | 值得重点跟踪，可进入项目库 |
| 8.0-8.9 | A级机会 | 值得关注，可做机会卡 |
| 7.0-7.9 | B级机会 | 入库观察，不急于深挖 |
| 6.0-6.9 | C级机会 | 作为趋势信号保留 |
| <6.0 | D级机会 | 不建议进入机会库 |

评分模板：

```markdown
## 机会评分

| 维度 | 权重 | 分数 | 加权分 | 判断理由 |
|---|---:|---:|---:|---|
| 市场价值 | 20% |  |  |  |
| 痛点强度 | 15% |  |  |  |
| 付费意愿 | 15% |  |  |  |
| AI适配度 | 15% |  |  |  |
| 迁移可行性 | 15% |  |  |  |
| MVP可验证性 | 10% |  |  |  |
| 风险可控性 | 10% |  |  |  |
| **总分** | **100%** |  |  |  |
```

反向扣分项，每命中一个扣 0.3-1 分：

1. 强监管风险。
2. 平台封禁风险。
3. 交付过重。
4. 客户预算不确定。
5. 同质化严重。

前台只展示总分、等级和一句判断；详情页展示完整评分表。

## 6A. Signal 详情页决策路径

Signal 详情页后续应从“信息”变成“决策工具”。内容组织先按这条路径：

```text
看标题
↓
看评分
↓
理解趋势
↓
判断影响
↓
看到机会
↓
产生行动
```

对应内容：

- 标题：这是什么变化。
- 评分：它值不值得优先看。
- 趋势：它是否属于连续变化。
- 影响：它影响收入、成本、岗位、流程还是渠道。
- 机会：它能否转成可验证机会。
- 行动：7 / 14 / 30 天继续看什么。

### 深挖机会卡重搜索

每天那 1 张机会卡，至少做 5 类交叉验证：

1. 代表公司  
这个方向是否有 3 家以上公司在做。

2. 融资信号  
最近 6-12 个月是否持续融资。

3. 客户场景  
它服务的是谁，客户是否真实付费。

4. 定价 / 商业模式  
是订阅、按席位、按量、按结果，还是服务费。

5. 中国迁移卡点  
监管、平台、数据、支付、客户预算、交付难度分别是什么。

### 深挖证据链模板

每张机会卡最后必须加：

```markdown
## 证据链

- 公司证据：
- 融资证据：
- 客户证据：
- 商业模式证据：
- 迁移可行性证据：

## 反向证据

- 哪些因素说明这个机会可能不成立：
```

证据链必须写具体事实、来源和链接，不得只写结论。

## 6.1 MVP 验证要求

深挖后必须输出 7-14 天 MVP 验证计划。

MVP 可以是：

- 访谈 5-10 个目标客户。
- 做一页需求验证问卷。
- 做低保真原型或服务包说明。
- 检索采购、招标、招聘和客户案例。
- 找潜在合作方验证可行性。
- 用朋友圈、社群或私域内容测试真实反馈。

MVP 必须写：

```yaml
mvp_id:
opportunity_id:
validation_window:
hypothesis:
target_users:
test_method:
success_signal:
failure_signal:
feedback_items:
decision:
```

## 6.2 Database 沉淀要求

- 场景库条目。
- 项目库条目。
- 风险库条目。

## 6.3 Content 分发要求

每篇深挖内参必须标注可派生内容：

- 内参版：3000-6000 字。
- 朋友圈版：300-600 字。
- 海报版：1 个判断 + 3 个证据 + 1 个反证。
- 短视频版：60-120 秒脚本。

分发内容必须保留证据边界，不能把观察写成确定。
