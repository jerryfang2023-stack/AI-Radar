# Guanlan Copy Style｜观澜 AI 基础文案规范 Skill

## 一、职责边界

本 Skill 负责观澜 AI 全站通用文案，包括：

- 网站栏目名；
- 页面标题；
- 模块标题；
- 卡片标题；
- 内容摘要；
- 按钮文案；
- 空状态文案；
- 引导提示；
- 推荐语；
- 标签文案；
- 页面说明；
- 运营短文案；
- 非专题类内容的默认文案风格。

本 Skill 不负责：

- 撰写「观澜每日观察」完整文章；
- 进行每日观察选题；
- 进行每日观察 QC；
- 替代专题栏目对应的专门 Skill 或 Agent；
- 输出长篇深度分析；
- 为了显得高级而生成大量抽象概念文案。

优先级规则：

> 如果某个栏目、专题或任务已有专门 Skill / Agent，优先使用对应 Skill / Agent。
> 本 Skill 只提供全站默认文案风格和基础防护规则。

例如：

- 「观澜每日观察」正文：优先调用 `guanlan-daily-observation-writer`。
- 「观澜每日观察」选题：优先调用 `guanlan-daily-observation-pitch`。
- 「观澜每日观察」质检：优先调用 `guanlan-daily-observation-qc`。
- 普通页面标题、栏目名、说明文案、卡片摘要：调用本 Skill。

## 二、观澜基础文案定位

观澜 AI 的文案不追求“AI 感”“科技感”“宏大感”，而追求：

> 克制、清楚、有判断、有商业感、读起来不像机器生成。

观澜文案应该像一个长期观察 AI 商业变化的人在说话：

- 不喊口号；
- 不堆术语；
- 不用行业黑话制造专业感；
- 不把趋势说满；
- 不把弱信号写成强结论；
- 不用空泛修饰填充页面；
- 尽量把抽象概念落到具体对象、动作、成本、流程、组织或竞争上。

## 三、最高原则

1. 文案要清楚，不要装深。
2. 文案要克制，不要喊趋势。
3. 文案要有判断，但不要过度断言。
4. 文案要像人写的，不要像模板生成。
5. 不用技术术语充当高级感。
6. 不用“赋能、重塑、生态、闭环”等词填空。
7. 栏目名和模块名要能让用户一眼知道它提供什么价值。
8. 短文案也要有信息量，不写“欢迎使用”“探索更多”这类无效句。
9. 遇到 AI 技术概念，要转译成用户能理解的业务问题。
10. 专题内容优先服从对应专题 Skill，本 Skill 只提供底层风格约束。

## 四、默认语气

观澜文案的默认语气：

- 克制；
- 清醒；
- 简洁；
- 具体；
- 有判断；
- 不端着；
- 不热血；
- 不卖弄；
- 不像研报；
- 不像公关稿；
- 不像 AI 生成的行业洞察。

适合的语气：

```text
看清 AI 商业变化里的真实变量。
```

```text
不是所有热点都值得写，观澜只挑真正改变预算、流程和竞争的信号。
```

```text
把模型、产品和公司动作，翻译成商业问题。
```

不适合的语气：

```text
探索 AI 时代的无限可能。
```

```text
赋能千行百业，构建智能未来。
```

```text
洞察趋势，链接生态，共创价值。
```

## 五、栏目名规则

栏目名必须：

- 简洁；
- 有辨识度；
- 有具体指向；
- 不像模板；
- 不堆抽象词；
- 不滥用“洞察、观察、前沿、趋势、生态、智库”等泛词；
- 不用晦涩技术词做栏目核心，除非该栏目就是面向专业读者。

栏目名优先表达：

- 观察对象；
- 内容用途；
- 判断角度；
- 用户为什么要看。

避免：

```text
AI 前沿洞察
智能生态观察
产业趋势聚合
AI 商业新范式
创新价值发现
生态能力中心
```

更好的方向：

```text
每日观察
公司信号
产品变化
Builder 说法
预算线索
采用现场
模型之外
工具变局
谁在买单
流程里的 AI
```

栏目名可以短，但不能空。
如果一个栏目名换到任何 AI 网站都能用，通常不合格。

## 六、标题规则

标题必须尽量包含：

- 一个变化；
- 一个矛盾；
- 一个代价；
- 一个问题；
- 一个商业变量；
- 一个具体对象。

避免标题像：

```text
AI Agent 发展趋势
大模型应用持续深化
企业 AI 化加速
智能体生态迎来新阶段
AI 工具创新观察
```

更好的标题方向：

```text
Agent 进企业，先要有人能叫停它
```

```text
模型降价之后，应用公司反而要面对毛利压力
```

```text
企业买的不是更长上下文，而是更少人工返工
```

```text
AI 写代码越快，测试环境越不能省
```

```text
平台争夺的不是接口，而是谁能留住客户数据
```

标题不要只告诉用户“主题是什么”，还要暗示“为什么值得看”。

## 七、模块说明文案规则

模块说明用于帮助用户理解一个栏目或功能的价值。

好的模块说明应该回答：

- 这里放什么；
- 为什么值得看；
- 用户能获得什么判断；
- 它和普通 AI 新闻有什么不同。

避免：

```text
这里汇聚 AI 行业最新动态，帮助用户洞察前沿趋势。
```

```text
精选优质内容，助力用户把握 AI 时代发展机遇。
```

```text
全面呈现 AI 生态变化，赋能商业决策。
```

更好的写法：

```text
每天从 AI 商业信息池里挑出一个最值得看的变化，说明它影响的是预算、流程、组织，还是竞争边界。
```

```text
记录 builder 对产品、模型和工作流的真实判断，不把社交媒体热度直接当成行业趋势。
```

```text
这里不追所有更新，只看哪些变化可能进入企业采购和真实流程。
```

## 八、摘要与卡片文案规则

摘要和卡片文案不能只是重复标题。
它应该补充一个具体信息：

- 证据；
- 变化；
- 对象；
- 商业含义；
- 判断边界。

避免：

```text
本文分析了 AI Agent 的发展趋势及其商业影响。
```

```text
本期关注大模型应用落地的最新变化。
```

```text
深入探讨 AI 技术如何赋能企业创新。
```

更好的写法：

```text
几条平台和工具更新共同指向一个问题：Agent 想进入企业系统，权限、审计和回滚机制会先进入采购清单。
```

```text
模型能力不是唯一变量。企业是否愿意买单，越来越取决于 AI 能否接入真实流程并被管住。
```

```text
这不是一个单纯的工具更新，而是企业把 AI 从试用带入流程时必须面对的成本问题。
```

摘要要短，但要有判断。
不要写成“本文将介绍……”。

## 八点一、判断资产卡片文案规则

本规范不仅覆盖页面壳、栏目名和按钮，也覆盖判断资产里的模型生成文案。当前阶段适用对象包括：

- 商业信号卡；
- 前沿观点卡；
- 变化候选；
- 场景候选；
- 趋势候选；
- 成熟变化短专题；
- 商业信号发布索引中的展示文案。

覆盖字段包括但不限于：

- 标题、摘要、一句解释；
- 发生了什么、为什么值得看、商业含义；
- 技术路线 / 方法变化；
- 观澜解读、继续观察、证据边界；
- 前台 `frontend` 对象中的标题、列表摘要、详情页模块文案和展示说明。

这些字段都属于“模型生成的解释性 / 判断性 / 展示性文案”，必须清楚、克制、具体，先写事实，再写判断，再写边界。

本规范不覆盖、也不得改写：

- Raw 原文；
- source quote；
- 原始摘录；
- URL；
- source metadata；
- HTML / Markdown / JSON 证据快照；
- `key_excerpts.text` 中保存的原文摘录。

这些内容是证据，不是文案。可以在旁边补一段“观澜解读”或“中文转述”，但不能为了统一文风改写原始证据。

当前资产类型的表达边界：

| 类型 | 后台生成文案 | 前台展示文案 | 必须保护 |
|---|---|---|---|
| 商业信号卡 | 说清谁做了什么、发生在哪里、为什么进入观察、影响什么商业变量 | 标题必须是事件型；摘要先讲事件，再说明影响客户、流程、预算、组织、责任、风险或竞争 | Raw、来源链接、关键摘录、数据来源 |
| 前沿观点卡 | 说清谁以什么身份在何处说了什么，以及观澜如何解读 | 原文或原文摘录优先；观澜解读必须说明这是观点参照，不是事实主证据；前台展示前必须完成四档评级 | 原文、发表时间、原文链接、capture_scope、opinion_tier、display_lane |
| 变化候选 | 说清变化假设、支撑信号、前后对比、商业变量和观察窗口 | 当前不作为普通前台卡片；成熟后写成变化短专题 | 支撑信号、反证、缺口、来源关系 |
| 场景候选 | 说清行业 / 部门、岗位 / 使用者、流程 / 任务和 AI 改变的步骤 | 当前不作为前台主卡；成熟后支撑相关场景或热力图节点 | 案例原文、客户说法、流程证据 |
| 趋势候选 | 说清底层变化、场景支撑、观点参照、证据缺口和反证 | 前台只能展示已达门槛的趋势报告或热力图判断；候选状态用自然语言降级 | 变化 / 场景 / Raw 来源关系和反证 |

卡片资产通过文案标准，必须同时满足：

1. 事实清楚：能看出具体对象、动作、场景和来源。
2. 判断克制：不把弱信号写成确定趋势，不把单一材料写成变化。
3. 商业变量明确：至少落到客户、流程、预算、组织、责任、风险或竞争之一。
4. 低 AI 味：不使用模板化趋势句、报告腔和公关腔。
5. 无内部生产词：前台不出现 Raw、Pool、gate、入库、补证、强证据等内部语言。
6. 证据边界不包装：信息不足时写“暂无公开信息”“暂未监测到同类案例”，不得补戏。

## 九、按钮和操作文案规则

按钮文案要清楚告诉用户动作，不要装饰。

优先使用：

```text
查看今日观察
阅读完整分析
生成选题 Brief
开始质检
继续返修
查看证据
展开材料
复制结果
```

避免：

```text
探索更多
开启智能之旅
立即体验未来
一键洞察
发现价值
开启赋能
```

按钮文案应该像产品动作，不像营销口号。

## 十、空状态文案规则

空状态文案要说明：

- 当前为什么为空；
- 用户可以做什么；
- 不要使用情绪化废话。

避免：

```text
暂无内容，敬请期待。
```

```text
这里空空如也，快去探索吧。
```

```text
更多精彩内容即将到来。
```

更好的写法：

```text
今天还没有生成观察。导入 raw 池后，可以先生成选题 Brief。
```

```text
还没有可用材料。补充 clean text 或 full text 后，再判断是否能形成主信号。
```

```text
当前没有通过 QC 的稿件。可以查看返修建议，或回到 writer 重新修改。
```

## 十一、技术语言转译规则

观澜文案可以出现技术词，但不能让技术词承担主要表达。

遇到技术词时，优先转译为：

- 业务动作；
- 成本变化；
- 流程变化；
- 组织责任；
- 采购理由；
- 风险边界；
- 竞争位置；
- 客户价值。

常见转译：

```text
RAG → 把企业内部知识接进 AI，减少胡编乱答。
```

```text
MCP → Agent 调用外部工具和数据时，需要更稳定的连接方式。
```

```text
context window → 模型一次能处理多少资料。
```

```text
inference cost → 每次调用 AI 的成本，影响毛利和定价。
```

```text
benchmark → 测试分数不能直接证明客户愿意付费。
```

```text
multimodal → AI 能处理图像、音频、视频，但价值取决于能否进入真实流程。
```

不要写：

```text
MCP 正在推动 Agentic Workflow 范式演进。
```

更好：

```text
Agent 要进入企业系统，难点不只是会调用工具，而是谁允许它碰哪些数据、执行哪些动作。
```

## 十二、去 AI 味规则

### 1. 避免空泛词

以下词语不是绝对禁用，但不能作为填充词高频出现：

- 值得关注；
- 可以看到；
- 这背后反映出；
- 从某种意义上说；
- 一方面……另一方面……；
- 总体来看；
- 未来值得持续关注；
- 正在重塑；
- 赋能；
- 闭环；
- 生态；
- 抓手；
- 底座；
- 深度融合；
- 提质增效；
- 千行百业；
- 打开想象空间；
- 进入新阶段；
- 迎来新拐点。

如果删掉这些词，句子意思没有损失，就删掉。

### 2. 避免机械结构

避免：

- 每段都很平均；
- 所有标题都是四字词；
- 每个模块都叫“洞察 / 趋势 / 生态 / 能力”；
- 每句都像标准答案；
- 每个标题都像“AI + 抽象名词”；
- 每段结尾都拔高一次。

### 3. 避免分析提示语过多

少用：

- 这说明……
- 这意味着……
- 这背后的变量是……
- 这句话的商业含义是……
- 这不是一个单纯的……故事
- 从这个角度看……

优先直接写出具体变化。

不要写：

```text
这句话的商业含义比技术含义更重要。
```

改成：

```text
这类工具真正靠近预算的地方，是它能不能让 AI 进入客户已有系统。
```

## 十三、观澜式表达检查

生成任何文案后，都要问：

1. 这句话是否具体？
2. 是否有商业变量？
3. 是否像真实的人写的？
4. 是否过度像 AI 行业文案？
5. 是否用了不必要的技术词？
6. 是否用了空泛大词？
7. 用户看完是否知道这里有什么价值？
8. 如果删掉这句话，信息是否有损失？

如果没有信息损失，删掉。
如果只是为了显得高级，重写。
如果任何 AI 网站都能用，不合格。

## 十四、专题 Skill 优先级

当任务属于明确专题时，优先调用专题 Skill。

优先级示例：

1. 每日观察选题
   使用 `guanlan-daily-observation-pitch`

2. 每日观察正文撰写
   使用 `guanlan-daily-observation-writer`

3. 每日观察质检
   使用 `guanlan-daily-observation-qc`

4. 其他专题栏目
   使用该栏目对应 Skill / Agent

5. 没有专题 Skill 的普通文案
   使用本 Skill

本 Skill 不覆盖专题内容的专业判断，只负责默认风格和通用文案质量。

## 十五、输出要求

根据用户任务输出对应文案，不默认解释规则。

如果用户要求多个备选，输出 3-8 个即可。
每个备选要有明显差异，不要只改几个词。

如果是栏目名，输出格式：

```md
## 栏目名建议

1. 名称：
   - 适合原因：
   - 风格说明：

2. 名称：
   - 适合原因：
   - 风格说明：
```

如果是页面文案，输出格式：

```md
## 页面标题

{}

## 页面说明

{}

## 模块文案

{}
```

如果是短文案优化，输出：

```md
## 优化后

{}

## 修改理由

-
```

如果用户只要求结果，可以只输出结果，不解释过程。

## 十六、最终标准

合格的观澜基础文案必须做到：

1. 清楚；
2. 克制；
3. 不喊口号；
4. 不堆技术词；
5. 不像 AI 生成模板；
6. 不像公关稿；
7. 有具体对象或具体动作；
8. 尽量有商业变量；
9. 能让用户知道这个栏目或功能有什么价值；
10. 与专题 Skill 不冲突；
11. 能体现观澜的基本气质：看准、讲清、少一点虚词，多一点判断。

最终原则：

> 观澜基础文案不是为了显得 AI、科技、宏大，而是为了让用户更快看清：这里提供什么判断，解决什么问题，为什么值得看。
