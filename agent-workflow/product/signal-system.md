# Signals 定义、监测与运营模型

更新时间：2026-05-02

## 1. 核心定义

Signal 不是新闻，不是公司动态，也不是工具列表。

Signal 是：

> 一个已经出现商业证据、可能改变赛道判断或机会优先级的 AI 商业变化。

进入观澜 AI 的 Signal 必须同时具备三类信息：

1. 事件：融资、产品发布、客户采用、收入增长、并购、监管、渠道变化、招投标、重大合作。
2. 证据：客户、收入、融资、部署、复购、合作、政策、行业采用、明确产品更新。
3. 商业含义：说明某个方向正在升温、分化、成熟、受阻，或出现可验证机会。

没有商业含义的新闻，不进入 Signals。

## 2. Signal 与新闻的边界

| 类型 | 是否进入 Signals | 原因 |
|---|---:|---|
| 单纯发布新闻 | 否 | 只有事件，没有商业含义 |
| 热门观点文章 | 否 | 只有观点，缺少证据 |
| 融资但无场景说明 | 待观察 | 有资本信号，但机会不清晰 |
| 产品发布且有客户采用 | 是 | 有事件、有证据、有商业含义 |
| 监管变化影响赛道 | 是 | 可能改变机会优先级或风险判断 |
| 招投标 / 客户部署 | 是 | 代表真实需求和落地路径 |

## 3. 可运营算法模型

Signals 系统采用“规则评分 + 人工复核 + 运营反馈”的混合模型。

模型目标不是替人做最终判断，而是帮助团队完成四件事：

1. 从每日监测中筛出高价值候选。
2. 让今日观察选择更稳定。
3. 让 Trend Reports 和 趋势候选s 有来源与事实支撑。
4. 反向优化关键词、网站和数据源。

## 4. Signal Score

Signal Score 使用 100 分制。

| 维度 | 权重 | 判断问题 |
|---|---:|---|
| 商业证据强度 | 25 | 是否出现客户、收入、部署、复购、招投标、监管、并购等证据 |
| 商业含义清晰度 | 20 | 是否能说明某方向升温、分化、成熟、受阻或出现机会 |
| 来源可信度 | 15 | 是否来自一手来源、高质量媒体或可靠产业数据 |
| 赛道相关性 | 15 | 是否落在观澜 AI 当前重点关注赛道或机会方向 |
| 新变化程度 | 10 | 是否代表新阶段、新客户、新动作、新变量 |
| 关联价值 | 10 | 是否能关联已有 Trend Report、Trend 或高价值标签 |
| 反证与风险提示 | 5 | 是否包含会削弱判断的风险变量或反证线索 |

## 5. 运营状态

| 分数 | 状态 | 处理方式 |
|---:|---|---|
| 85-100 | `core_signal` | 进入 Daily Brief 候选，必须人工复核 |
| 70-84 | `valid_signal` | 进入 Signals 库，可关联 Trend Report 或 Trend |
| 55-69 | `watch_signal` | 进入观察池，等待更多证据 |
| 40-54 | `weak_signal` | 暂存，不进入前台 |
| 0-39 | `noise` | 标记噪音，用于优化关键词和来源 |

注意：高分只代表“值得关注或复核”，不代表确定性结论。

## 7. 监测来源分层

### S 级：一手证据源

优先监测：

- 公司官网、产品更新、博客、开发者文档。
- 客户案例、合作公告、上线公告。
- 监管文件、政策文件、招投标公告。
- 招聘页、产品文档、价格页、API 文档变化。
- GitHub、云市场、插件市场、App Store、Chrome Web Store。

### A 级：高质量商业媒体

保留稳定监测：

- 海外：WSJ、FT、Bloomberg、The Information、TechCrunch、Axios、CNBC。
- 中文：36氪、晚点、甲子光年、钛媒体、财新、机器之心、量子位。

### B 级：产业垂直媒体与数据库

按赛道监测：

- 医疗、工业、金融、营销、企业服务等行业媒体。
- 融资数据库、并购公告、工商变更。
- G2、Product Hunt、云市场、应用商店评价。

### C 级：社媒与聚合站

只作为线索，不直接作为高分证据。

### D 级：噪音源

长期低效、重复转载、标题党或不可验证来源，应降权或移除。

## 8. 关键词体系

关键词不是静态清单，而是可优化的监测资产。

| 层级 | 示例 | 作用 |
|---|---|---|
| 赛道词 | AI Agent、AI营销、AI治理、端侧推理 | 定位关注方向 |
| 证据词 | 融资、客户、收入、上线、部署、复购、并购 | 判断是否有商业证据 |
| 场景词 | 客服、销售、法务、财务、供应链、医疗、工业 | 判断落地场景 |
| 风险词 | 监管、诉讼、暂停、裁员、亏损、安全、隐私 | 捕捉反证与风险 |
| 资源词 | 合作、采购、招标、试点、集成、渠道 | 判断可连接资源 |

## 9. 每日监测优化规则

每日监测任务需要记录每个关键词和来源的表现。

关键词指标：

- 命中数量。
- 有效 Signal 数量。
- 平均 Signal Score。
- 噪音率。
- 重复率。
- 进入 Daily Brief 的数量。
- 转化为 Trend Report 或 Trend 的数量。
- 主要来源域名。

来源指标：

- 有效 Signal 产出率。
- 一手证据比例。
- 平均分。
- 首发或早发现能力。
- 重复转载率。
- 噪音率。
- 覆盖赛道。

优化规则：

1. 连续多日产生高分 Signal 的关键词进入 `high_yield_keywords`。
2. 连续多日只产生噪音或重复内容的关键词降权。
3. 高分 Signal 中出现的新公司、新产品、新场景、新客户类型进入 `candidate_keywords`。
4. 新趋势出现时，扩展中英文同义词、场景组合词和证据词。
5. 长期低效来源进入 `source_downgrade_candidates`。
6. 一手证据产出率高的来源进入 `priority_sources`。

## 10. 每日监测输出

每日监测除了生成：

```text
YYYY-MM-DD-AI商业雷达.md
```

还应生成或更新：

- 当日高分 Signal 清单。
- 噪音来源清单。
- 重复候选清单。
- 新增候选关键词。
- 降权关键词。
- 高价值来源。
- 需要人工复核的边界案例。

## 11. 阶段性反馈机制

### 每日复核

检查：

- 今日高分 Signal 是否真的进入 Daily Brief 候选。
- 是否存在公司名标题、重复 Signal、无商业含义内容。
- 是否有裸 Markdown 链接或来源不可读问题。

### 每周优化

输出：

```text
agent-workflow/reports/signal-quality-weekly-YYYY-WW.md
```

内容：

- 高产关键词。
- 低效关键词。
- 新增关键词建议。
- 待移除关键词。
- 高价值网站或数据源。
- 噪音来源。
- Signal Score 分布。
- Daily Brief 转化率。

### 每月校准

输出：

```text
agent-workflow/reports/signal-model-calibration-YYYY-MM.md
```

内容：

- 哪些高分 Signal 最终形成 Trend Report 或 Trend。
- 哪些高分 Signal 被证明只是热度。
- 哪些低分 Signal 后续变重要。
- 是否需要调整权重。
- 是否需要新增赛道或标签。

### 季度复盘

目标：

- 调整重点赛道。
- 清理长期低效来源。
- 升级关键词体系。
- 检查模型是否过度偏向融资、媒体热度或单一赛道。

## 12. 对外展示规则

- Signal 标题必须是“事件 + 商业含义”，不能只是公司名。
- 公司名、产品名是证据字段，不是判断本身。
- 每个 Signal 要说明“为什么值得关注”。
- 每个 Signal 尽量关联 Trend Report 或 Trend。
- 重复 Signal 不新增第二条，应合并更新或补充证据。
- 前台不展示完整算法细节，只展示证据强度、趋势关联和判断依据。
