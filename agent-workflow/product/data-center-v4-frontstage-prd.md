# 观澜数据中心 V4 前台产品需求文档

状态：已确认  
版本：`SITE-V4.2.0-entity-history-prd-v1.3`
日期：2026-07-16  
适用范围：观澜内部只读网站的数据中心与应用中心  
上游事实标准：`SITE-V4.0-data-center`、`RAW-V3.0`、`EVENT-V1.1`、`FDE-V2.0`、`HARDWARE-V1.0`、`TAG-V4.0`

## 1. Problem Statement

观澜现有前台仍以商业信号、卡片、趋势和报告等历史内容产品为主要组织方式，无法直接、稳定地承载 V4 已建立的结构化事实链，也容易让数据事实、社群材料、人物观点和下游分析混在同一层级。

新的前台需要正式定位为 AI 行业数据中心，为内部研究、AIP 构建和后续决策支持应用提供统一的数据访问入口。它必须以事件为核心，以 FDE 实施、AI 硬件、社群情报和一线观点为并列数据栏目，并通过实体索引统一访问公司、产品、人物、技术、场景与行业记录，支持浏览、搜索、筛选和回溯，同时严格保持数据层中立。

数据中心只提供结构化、体系化、可关联的数据基座，不替用户做决策，不判断商业价值，不推荐行动，不进行趋势定性，也不教育用户应当得出什么结论。

## 2. Solution

将网站划分为两个一级中心：

1. **数据中心**：承载事件及六类关联数据，是网站默认入口和事实基础。
2. **应用中心**：承载机会地图、周报、月报等下游分析应用。

数据中心以事件为核心，采用统一的“搜索与筛选 + 精简列表 + 数据详情 + 关联跳转”访问模式。应用中心可以分析、判断和形成机会，但必须引用并链接回数据层记录，且不得把分析结论写回数据层。

首期面向内部只读使用，不建设复杂账号体系、运营后台、编辑能力、导出能力、开放 API 或智能问答。

## 3. Product Positioning And Boundaries

### 3.1 产品定位

观澜数据中心是结构化、体系化、来源可回溯的 AI 行业数据基座，服务于：

- 内部研究人员查找和串联 AI 行业事实；
- AIP 产品读取和调用标准化数据；
- 机会地图、周报、月报等应用生成下游分析；
- 创业者或业务研究者通过应用层使用数据，但由用户自行决策。

### 3.2 数据层允许的能力

- 浏览结构化数据；
- 按关键词搜索；
- 按必要字段筛选；
- 按日期排序和分页；
- 查看公司、产品、事件及其他数据之间的事实关联；
- 查看中文翻译与保留的原文；
- 查看原始来源入口；
- 显示“未披露”和事实冲突；
- 通过 Tag 进行跨类型检索和关联。

### 3.3 数据层禁止的能力

- 重要性、价值、影响、机会和趋势成熟度评分；
- 商业意义、推荐理由、关注理由和行动建议；
- 热度榜、推荐榜和主观排序；
- 自动总结“意味着什么”；
- 将社群材料或人物观点直接当作事实事件；
- 用 Tag 决定数据是否成立、是否入库或如何排名；
- 把应用层结论回写为事实；
- 设置“来源与证据”“质量与覆盖”等独立前台模块或仪表盘。

原始来源链接作为记录详情的必要回溯入口保留，但不扩展为前台证据评分、来源评价或覆盖率展示。

## 4. Users And Primary Tasks

### 4.1 主要用户

- 观澜内部研究人员；
- 观澜 AIP 产品构建人员；
- 需要使用结构化行业数据的内部业务人员。

### 4.2 首要任务

用户进入网站后的首要任务是查找和浏览事件，而不是阅读数据概览或查看统计仪表盘。

网站根入口默认打开“商业事件”，并只显示当天事件。栏目标题下方仅保留一行紧凑的搜索与筛选工具栏，随后直接进入事件列表。统计数字不能占据页面核心位置。

## 5. Information Architecture

### 5.1 导航分组

左侧栏分为：

- 数据中心；
- 应用中心。

### 5.2 数据中心栏目

1. 商业事件
2. FDE 实施
3. AI 硬件
4. 社群情报
5. 一线观点
6. 实体索引

进入数据中心时默认打开“商业事件”。

实体索引统一收录公司、机构、模型、产品、服务、人物、AI 技术、场景与行业，位于“一线观点”之后。公司、产品和人物使用稳定实体 ID；技术、场景和行业使用稳定分类节点 ID。它们不再作为独立侧边栏栏目，但继续保留独立详情记录与稳定链接。

### 5.3 应用中心栏目

1. 行业报告

行业报告沿用现有页面，内部承载机会地图、周报和月报，不再把三者拆成独立侧边栏入口。

### 5.4 导航原则

- 顶栏只展示正式横版 Logo；
- 顶栏不展示数据中心/应用中心切换或全局搜索；
- 左侧栏分组展示数据中心六个栏目和应用中心一个行业报告入口；
- 数据中心默认路由指向事件列表；
- 行业报告保留现有机会地图、周报和月报的主要结构与交互，不展示关联路径模块。

## 6. Data Layer Requirements

### 6.1 商业事件

事件是数据中心的核心事实单位。每条事件只表达一个主要事实。默认事件页只显示当天事件，同一天内使用稳定顺序，必要时按更新顺序稳定排列。历史事件通过栏目搜索和筛选查询。

事件日期只显示到天，格式为 `YYYY-MM-DD`，不显示时、分、秒。

商业事件必须通过 AI 产业范围门禁：原始标题或已接受 Claim 必须证明 AI 是事件直接涉及的技术、产品、能力、组织、政策、交易、部署、硬件或研究对象。AI 频道、RSS、搜索词、页面导航和偶然出现的 AI 词不构成准入证据。仅以“智慧 / 大模型”包装垂直行业本地系统、但没有明确 AI 产业主体、产品身份、市场动作或可复用技术发布的材料不进入前台事件列表。

商业事件默认陈列顺序为：融资与案例类优先，包括“融资与并购、商业合作、部署与案例”；其次为“模型、产品与服务”；最后陈列组织政策、研究、AI 硬件等其他内容。各层内部保持原有稳定顺序，搜索和筛选结果沿用同一排序。

#### 前台事件分组

| 前台分组 | EVENT-V1.1 精确类型 |
|---|---|
| 模型、产品与服务 | `model_release`、`product_release`、`service_change`、`pricing_change` |
| 融资与并购 | `funding`、`acquisition`、`ipo_listing`、`capital_investment`、`financial_performance` |
| 商业合作 | `partnership`、`procurement_contract` |
| 部署与案例 | `deployment` |
| 组织、政策与法律 | `organization_people`、`organization_restructuring`、`market_expansion`、`policy_regulation`、`lawsuit_settlement`、`certification_compliance`、`standard_specification`、`security_incident` |
| 研究结果 | `research_result` |
| AI 硬件 | `hardware_product`、`hardware_capacity`、`hardware_supply`、`hardware_deployment` |

事件列表展示：

- 中文事件标题；
- 一个前台事件类型；
- 最多两个 Tag。

如果原始标题为英文，必须翻译为中文用于主标题。公司、产品、金额等信息已包含在标题中时，不重复展示。事件默认页不显示独立日期行或逐条日期；当前数据日期以小字显示在栏目标题旁。列表不展示摘要、状态、来源、ID、评分、内部字段或其他元数据。

事件详情展示：

1. 中文标题；
2. 原始标题；
3. 日期、EVENT-V1.1 精确事件类型、最多五个 Tag；
4. 事件事实：主体、动作、对象、已披露数据；
5. 相关公司和产品；
6. 相关 FDE 实施和 AI 硬件记录；
7. 相关社群情报和一线观点；
8. 原始来源链接；
9. 相关事件。

缺失信息显示“未披露”。发生事实冲突时显示“存在冲突”，并并列呈现不同版本及其原始链接，不提供置信度评分或商业判断。

### 6.2 实体索引：公司

实体索引提供“全部、公司与机构、产品模型服务、人物、AI 技术、场景与行业”切换，并支持关键词与 Tag 筛选。公司仍是独立详情记录，不建设百科式介绍、公司评价、竞争分析或投资判断。

公司索引项展示：

- 公司名称；
- 主要类型；
- 最多两个 Tag。

公司详情展示：

- 公司名称和基本身份；
- 最多五个核心 Tag；
- 相关产品；
- 按日期倒序的相关事件；
- 相关 FDE 实施；
- 相关 AI 硬件；
- 相关社群情报；
- 相关一线观点。

每个关联模块先展示少量近期记录，并提供“查看全部”。

### 6.3 实体索引：产品

产品与公司共用实体索引列表。产品仍是独立详情记录，不提供产品评测、推荐、排名或竞争优劣判断。

产品索引项展示：

- 产品名称；
- 所属公司；
- 产品类型；
- 最多两个 Tag。

产品详情展示：

- 产品名称；
- 所属公司；
- 产品类型；
- 最多五个 Tag；
- 按日期倒序的相关事件；
- 相关 FDE 实施；
- 相关社群情报；
- 相关一线观点。

### 6.3.1 人物与分类节点

人物使用稳定实体 ID，详情页汇总人物身份、所属机构、公开观点与有事实依据的相关事件。观点仍属于独立数据层，不能作为商业事件或关系证据。

AI 技术、场景与行业使用稳定分类节点 ID，详情页仅汇总带对应 TAG-V4 Assertion 或 FacetAssertion 的事件、实体、FDE 与硬件记录，不输出热度、趋势、排名或推荐。

### 6.4 FDE 实施

FDE 实施是从已接受事件投影形成的企业实施数据，不是独立事件类型，也不得绕过事件事实链直接生成。

FDE 列表展示：

- 中文事件标题；
- 日期；
- 实施阶段；
- 最多两个 Tag。

FDE 详情展示：

1. 标题、日期、实施阶段、最多五个 Tag；
2. 客户、服务商、行业、使用场景；
3. 已披露需求；
4. 交付组成；
5. 已披露结果，未披露时明确显示“未披露”；
6. 相关实体、产品和原始事件；
7. 原始来源链接。

不得展示实施效果评价、价值判断或行动建议。

### 6.5 AI 硬件

AI 硬件是从已接受硬件事件投影形成的结构化数据，不得绕过事件事实链直接生成。

AI 硬件列表展示：

- 中文事件标题；
- 日期；
- 硬件类型；
- 最多两个 Tag。

AI 硬件详情展示：

1. 标题、日期、硬件事件类型、最多五个 Tag；
2. 部件、算力层级、供应方、客户；
3. 已披露制程、容量、地点、金额、出货日期；
4. 相关实体、产品和原始事件；
5. 原始来源链接。

不得展示性能评价、市场影响判断或采购建议。

### 6.6 社群情报

社群情报是独立数据类型，与事件处于同一数据层级，但不能自动升级为事实事件。它可以通过实体、事件和 Tag 建立关联。

社群情报列表展示：

- 中文标题；
- 日期；
- 最多两个 Tag。

社群情报详情展示：

1. 中文标题；
2. 日期和最多五个 Tag；
3. 原始内容；
4. 作者和社群；
5. 相关公司和产品；
6. 相关事件；
7. 原始入口。

列表不展示摘要、热度、评分、推荐、内部状态或其他辅助字段。

### 6.7 一线观点

一线观点是独立的人物观点数据，与事件处于同一数据层级，但不能作为事实事件或商业结论。观点之间的分歧不属于事实冲突。

一线观点列表展示：

- 中文观点标题；
- 人物；
- 日期；
- 最多两个 Tag。

一线观点详情展示：

1. 中文观点标题；
2. 中文翻译全文；
3. 人物、角色、机构、日期和最多五个 Tag；
4. 原文；
5. 相关公司和产品；
6. 相关事件；
7. 原始入口。

不得添加编辑解读、立场判断、观点评分、热度或推荐。

## 7. Search, Filters, Tags And Pagination

### 7.1 栏目搜索

顶栏不设置全局搜索。六个数据栏目分别提供栏目内关键词搜索，搜索结果使用所属栏目的精简列表字段，不生成 AI 答案、自动总结或相关度评分。

事件默认页只显示当天记录；事件栏目搜索与“更多筛选”可用于查询历史事件。

### 7.2 搜索范围

- 事件：中文标题、原始标题、主体、对象、产品；
- 实体索引：公司名称与别名、产品名称、所属公司、模型或产品类型；
- FDE 实施：标题、客户、服务商、行业、使用场景；
- AI 硬件：标题、部件、供应方、客户；
- 社群情报：标题、正文关键词、公司、产品、人物、行业、Tag；
- 一线观点：中文观点、原文关键词、人物、机构、公司、产品、Tag。

中文翻译与原文都必须可搜索。专有名词使用官方名称，不进行不必要的翻译。

### 7.3 栏目基础筛选

| 栏目 | 默认基础筛选 |
|---|---|
| 事件 | 事件类型、Tag；日期收纳在“更多筛选” |
| 实体索引 | 实体类型切换、Tag |
| FDE 实施 | 日期、实施阶段、行业、Tag |
| AI 硬件 | 日期、硬件类型、Tag |
| 社群情报 | 日期、Tag |
| 一线观点 | 日期、人物、Tag |

所有栏目均支持关键词搜索。非高频条件收纳在“更多筛选”中，避免首屏控件过多。

### 7.4 Tag

Tag 只用于结构化描述、筛选和关联：

- 每个技术 Tag 必须符合 `TAG-V4.0` 并具有 Claim 事实依据；产品形态、应用场景、行业、部署形态和目标用户使用独立 FacetAssertion，不进入技术 Tag 命名空间；
- 列表最多显示两个 Tag；
- 详情最多显示五个核心 Tag；
- Tag 不决定入库、真伪、关系方向或排序；
- 不显示 Tag 热度、趋势、评分或推荐；
- Tag 可点击。

点击 Tag 后进入统一标签结果页，默认显示“全部”，按六个前台栏目分组展示，并支持切换栏目和关键词搜索。

### 7.5 分页

- 事件默认页不显示数字分页；
- 其他完整数据列表、栏目搜索结果和 Tag 结果采用数字分页；
- 每页默认二十条；
- 形式为“上一页 1 2 3 4 … 20 下一页”；
- 翻页时保留关键词、筛选条件和当前数据类型；
- 首期不提供每页条数切换；
- 不使用无限滚动作为主分页方式。

## 8. Application Layer Requirements

### 8.1 首期应用

- 机会地图；
- 周报；
- 月报。

### 8.2 应用边界

应用中心可以进行归纳、分析、判断和机会表达，但必须：

- 明确属于下游应用，而不是数据事实；
- 引用或链接回支撑结论的数据层记录；
- 与数据层分开存储；
- 不修改数据层事实字段；
- 不将分析标签写入事实 Tag；
- 不将应用结论自动生成 CanonicalEvent。

### 8.3 现有页面兼容

首期沿用现有应用页面的主要样式、结构和核心交互：

- 机会地图沿用现有行业报告的地图结构和交互；
- 周报、月报沿用现有报告列表和详情结构；
- 只调整全站导航、数据回链和数据引用关系；
- 本期不重做应用中心核心布局。

## 9. Shared Interaction And Display Rules

- 六个数据栏目共用“搜索与筛选 + 精简列表”的基础模式；
- 不同栏目只改变必要字段与筛选条件；
- 各类详情页共用“基本信息 + 关联数据”的阅读逻辑；
- 日期、Tag、空状态、分页和跳转行为保持一致；
- 数据列表以记录本身为核心，辅助标签弱化展示；
- 前台遵循“非必要不展示”原则；
- 桌面端优先，目标宽度约为 1440 像素；
- 手机和平板只保证可读、可搜索和基础导航，不建设复杂移动交互；
- 首期避免宽表格、复杂关系图和移动端专属页面；
- 视觉配色使用观澜 VI 配色；
- 页面具体布局、密度和组件摆放在下一阶段“栏目—页面架构”中确认。

## 10. User Stories

1. 作为内部研究人员，我希望进入网站后直接看到当天事件，以便快速开始当天的信息检索。
2. 作为内部研究人员，我希望默认页不混入历史日期，以便聚焦当天发生的事实。
3. 作为内部研究人员，我希望事件列表以中文标题为核心，以便快速浏览外文信息。
4. 作为内部研究人员，我希望在事件详情中保留原始标题，以便核对翻译与原始表达。
5. 作为内部研究人员，我希望通过搜索和更多筛选查询历史事件，以便在需要时查看某一天或某一时间段的数据。
6. 作为内部研究人员，我希望按事件类型筛选，以便集中查看融资、合作、部署或硬件等事实。
7. 作为内部研究人员，我希望通过关键词同时搜索中文标题和原始标题，以便不受语言限制。
8. 作为内部研究人员，我希望查看事件的主体、动作、对象和披露数据，以便获得最小但完整的事实结构。
9. 作为内部研究人员，我希望缺失字段明确显示“未披露”，以便区分未获取与实际不存在。
10. 作为内部研究人员，我希望冲突事实并列显示不同版本及来源，以便自行判断而不接受系统代替判断。
11. 作为内部研究人员，我希望从事件跳转到公司和产品，以便沿实体继续检索相关历史。
12. 作为内部研究人员，我希望公司页聚合其产品和全部历史事件，以便形成连续的数据视图。
13. 作为内部研究人员，我希望产品页聚合版本、发布和部署事件，以便查看产品演进事实。
14. 作为企业 AI 研究人员，我希望单独浏览 FDE 实施记录，以便查找企业采用和交付数据。
15. 作为企业 AI 研究人员，我希望 FDE 结果未披露时被明确标记，以免把需求或试点误认为已取得成果。
16. 作为硬件研究人员，我希望单独浏览硬件产品、容量、供应和部署数据，以便查找产业链事实。
17. 作为硬件研究人员，我希望硬件详情只展示已披露的制程、容量、地点、金额等字段，以免系统推断缺失信息。
18. 作为内部研究人员，我希望社群情报保持独立数据类型，以便使用一线材料而不把它误当作正式事件。
19. 作为内部研究人员，我希望社群情报支持正文关键词和 Tag 搜索，以便从非新闻材料中找到相关线索。
20. 作为内部研究人员，我希望一线观点保持独立数据类型，以便阅读建设者观点而不将其当作事实结论。
21. 作为内部研究人员，我希望一线观点同时保留中文翻译和原文，以便准确理解人物表达。
22. 作为内部研究人员，我希望搜索人物、机构、公司、产品和 Tag，以便定位相关一线观点。
23. 作为数据使用者，我希望在当前栏目直接搜索，以便搜索结果保持明确的数据类型。
24. 作为数据使用者，我希望顶栏保持简洁，以便页面空间优先用于当前栏目数据。
25. 作为数据使用者，我希望点击 Tag 后查看跨类型关联记录，以便围绕一个主题继续检索。
26. 作为数据使用者，我希望 Tag 不显示热度和评分，以便保持数据层中立。
27. 作为数据使用者，我希望筛选项保持精简，以便页面首先呈现数据而不是大量控件。
28. 作为数据使用者，我希望在完整列表和搜索结果中使用数字分页并保留搜索条件，以便稳定返回和重复定位结果。
29. 作为 AIP 构建人员，我希望网站字段与 V4 数据合同保持一致，以便前台和后续应用共享同一事实基础。
30. 作为 AIP 构建人员，我希望数据记录之间使用稳定关联，以便应用可以回溯事件、实体和投影数据。
31. 作为应用阅读者，我希望机会地图中的分析能够跳回支撑它的数据记录，以便核验分析基础。
32. 作为应用阅读者，我希望周报和月报中的结论能够链接相关事件，以便继续查看原始数据。
33. 作为数据治理人员，我希望应用层不能修改数据层事实，以便避免分析污染事实数据库。
34. 作为数据治理人员，我希望社群和观点不能自动生成正式事件，以便保持事实边界。
35. 作为内部用户，我希望首期网站保持只读，以便优先完成稳定的数据访问体验。
36. 作为内部用户，我希望网站不要求复杂登录和权限操作，以便快速进入日常检索。
37. 作为内部用户，我希望日期统一精确到天，以便与当前每日生产节奏一致。
38. 作为桌面端用户，我希望页面针对约 1440 像素工作环境优化，以便高效浏览较多数据。
39. 作为移动端用户，我希望能够完成阅读、搜索和基础导航，以便临时查阅数据。
40. 作为产品负责人，我希望数据中心和应用中心在导航上明确分层，以便用户知道何处是事实、何处是分析。

## 11. Implementation Decisions

### 11.1 前台壳层模块

建设统一的网站壳层：顶栏只展示正式横版 Logo；左侧栏分组承载数据中心和应用中心栏目，并显示当前栏目状态。该模块不包含数据业务判断。

### 11.2 查询与筛选模块

建设栏目查询接口，接收关键词、栏目基础筛选、Tag 和页码，返回稳定排序的结果。事件默认查询只返回当天记录；历史查询通过搜索或“更多筛选”触发。搜索匹配翻译字段与原文字段，但不生成搜索摘要或相关度评分。

### 11.3 精简列表模块

建设可配置的统一列表容器，复用标题、弱化 Tag、空状态和必要时的数字分页。事件默认页不显示独立日期行和分页，数据日期在栏目标题旁辅助显示。不同数据类型通过字段配置展示各自的最小必要信息，不建设六套互不一致的列表系统。

### 11.4 详情与关联模块

建设统一详情容器，支持基本信息、事实字段、缺失状态、冲突状态、实体关联、事件关联和原始入口。各数据类型提供自己的详情字段定义。

### 11.5 实体聚合模块

公司、产品和人物页通过稳定实体标识聚合事件、FDE、硬件和观点记录；技术、场景和行业使用分类节点页。关系只来自带 Event、Claim 与 SourceArtifact 引用的 RELATION-V2 记录，Tag 共现不能推断关系，也不使用主观推荐算法。

### 11.6 Tag 结果模块

建设统一 Tag 结果页，按六个前台栏目分组并支持类型切换、关键词搜索和数字分页。公司与产品在 Tag 结果中合并为实体索引。Tag 只触发过滤，不触发热度计算、趋势计算或推荐排序。

### 11.7 应用数据回链模块

为机会地图、周报和月报增加统一的数据引用结构，使应用条目可以链接事件、公司、产品、FDE 或硬件记录。应用数据与事实数据保持单向引用关系。

### 11.8 数据适配模块

前台读取 V4 规范化数据及社群、观点的独立数据集。适配层负责将规范字段转换为页面所需的最小展示模型，不允许从历史页面摘要、旧 Card 评分或应用结论反向补全事实字段。

### 11.9 路由模块

首期路由至少覆盖：

- 数据中心六个栏目列表与七类详情记录；
- 栏目搜索结果；
- Tag 统一结果；
- 应用中心机会地图、周报和月报列表/详情；
- 公司、产品及各类关联记录之间的稳定跳转。

具体 URL、页面模板数量和模块摆放在下一阶段确认。

## 12. Testing Decisions

测试只验证用户可观察行为和数据边界，不锁定内部实现细节或具体 DOM 结构。

### 12.1 数据合同测试

- 继续沿用现有 Data Center V4 数据构建与完整性测试；
- 验证事件、FDE、硬件和 Tag 均来自允许的事实链；
- 验证禁止字段不会进入数据层前台模型；
- 验证缺失字段和冲突不会被静默补全。

### 12.2 查询行为测试

- 验证中文标题与原始标题均可命中；
- 验证栏目筛选与关键词组合生效；
- 验证数字分页保留关键词和筛选状态；
- 验证事件默认页只显示当天记录并保持稳定顺序；
- 验证事件历史记录可通过搜索和更多筛选查询。

### 12.3 列表与详情行为测试

- 验证各列表只出现 PRD 允许的字段；
- 验证列表 Tag 数量不超过两个；
- 验证详情 Tag 数量不超过五个；
- 验证原始标题、中文翻译、日期和必要关联可用；
- 验证“未披露”和“存在冲突”的显示行为；
- 验证相关记录及“查看全部”跳转正确。

### 12.4 数据边界测试

- 验证社群情报和一线观点不会混入事件列表；
- 验证应用结论不会出现在数据层事实字段；
- 验证 Tag 不影响入库、事实真伪和默认排序；
- 验证 FDE 与硬件记录均可回溯原始事件；
- 验证应用引用可以回到数据层，但不能修改数据层。

### 12.5 页面回归测试

- 验证桌面端约 1440 像素下的侧边栏导航、紧凑工具栏、浏览、搜索、筛选和必要分页；
- 验证手机和平板可以阅读、搜索和完成基础导航；
- 验证数据中心和应用中心导航状态正确；
- 验证机会地图、周报和月报保留现有主要结构与核心交互。

## 13. Acceptance Criteria

首期完成必须同时满足：

1. 数据中心六个栏目均可浏览；
2. 六个栏目均支持关键词搜索和基础筛选，完整列表与搜索结果支持数字分页；
3. 每类数据均有可打开的详情页；
4. 公司、产品、事件、FDE、硬件、社群和观点之间的必要关联可跳转；
5. 事件默认页只展示当天事件，日期在栏目标题旁辅助显示；
6. Tag 统一结果页可用；
7. 事件以中文标题为核心，外文记录保留原始标题或原文；
8. 日期统一精确到天；
9. 缺失与冲突按规则展示；
10. 机会地图、周报和月报继续可用；
11. 应用内容可回溯到数据层记录；
12. 应用层不能修改或污染数据层；
13. 数据层不出现评分、推荐、热度、商业判断或教育性结论；
14. 网站满足内部只读使用；
15. 现有 V4 数据完整性测试继续通过；
16. 新增前台外部行为测试通过。

## 14. Out Of Scope

首期不包括：

- 对外开放和公开用户体系；
- 复杂登录、角色、权限和付费；
- 数据运营后台；
- 前台创建、编辑、删除、纠错、批注和审核；
- 数据导出和开放 API；
- 智能问答和 AI 搜索答案；
- 推荐系统、相关度评分、热度榜和趋势榜；
- 公司、产品、机会或事件评分；
- 数据概览仪表盘；
- 独立“来源与证据”页面；
- 独立“质量与覆盖”页面；
- 复杂关系图谱；
- 宽表格型数据工作台；
- 移动端专属功能；
- 重做机会地图、周报和月报的核心页面结构；
- 将应用结论回写为数据事实；
- 将社群情报或一线观点自动升级为事件。

## 15. Further Notes

- 本 PRD 确认产品定位、边界、栏目范围、数据字段、搜索筛选规则、实施模块和验收标准。
- 本 PRD 不确认页面视觉稿和具体模块布局。
- PRD 确认后，下一阶段按顺序确认：
  1. 栏目与页面清单；
  2. 每个页面的模块结构；
  3. 页面之间的跳转关系；
  4. 桌面端信息密度与视觉层级；
  5. 最终 V4 页面示意图。
- 页面视觉需使用观澜 VI 配色，但不强制沿用此前提出的 Typography-first 或额外字体审查要求。
- 在页面方案最终确认前，现有前台继续保持冻结兼容状态。
