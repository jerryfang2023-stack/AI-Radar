# 日常监测算法 v2 测试管线

## 1. 定位

日常监测算法 v2 是观澜AI的测试管线，用于把每日 AI 商业雷达从“一次生成 Signals”拆成可复核的分阶段漏斗。

本管线只进入 `06-content/`。网页展示不是当前优先级；成熟前不替换 `01-Signals/`、`signals.html` 或 `ai-3` 生产同步闸门。

2026-05-05 追加原则：

- v2 的核心不是多生成页面，而是建立从入库到深挖的内容判断标准。
- 精选 3 条 Signals 必须形成可阅读、可复核、可进入内参候选的完整文字，不得只是卡片摘要。
- 深挖机会每天最多 1 篇，目标形态是“内参级深度报道”，不是泛泛机会卡。
- 任何缺少事实层、产业层、客户层、商业模式层、反证层和后续观察层的深挖，不得标记完成。

## 1.1 PM Agent / Data Agent 联合结论

### PM Agent 结论

v2 管线的商业价值不在“更多条目”，而在“更少但更深”的判断资产。

PM 口径：

- Raw 和 Pool 是内部筛选资产，不面向外部用户。
- Structured 是判断素材库，必须足够完整，便于后续复盘。
- 精选 3 条 Signals 是内参候选，不是普通资讯摘要；每条都要能让用户理解“发生了什么、为什么值得看、证据边界在哪里、后续看什么”。
- 深挖机会是未来付费内参的雏形，每篇都应该像一篇小型深度报道，能支撑闭门简报、行业报告、销售沟通或资源连接判断。
- 如果当天没有足够证据支撑深挖，宁可不写深挖，也不能硬凑一篇泛泛机会分析。

PM 对深挖内参的通过线：

1. 读者不需要回看原新闻，也能理解事件背景和商业问题。
2. 读者能看见多方证据，而不是只看一家公司的叙述。
3. 读者能理解目标客户、替代流程、付费方式和进入难点。
4. 读者能看到反证和不确定性，而不是被推向一个结论。
5. 读者能获得后续观察框架，但不被下经营、投资或合作指令。

### Intelligence Data Agent 结论

Data 口径：

- Structured 入库不是“写满字段”，而是形成可追溯的判断单元。
- 精选 3 条的核心是二次搜索后的证据增强，不是从 8 条里挑标题更好看的 3 条。
- 深挖机会必须具备多源证据结构：一手来源、行业/客户来源、资本或商业来源、反证来源、可迁移性来源。
- Builder 观点、社媒讨论、GitHub 热度、榜单和聚合站只能作为线索或观点边界，不得作为事实主证据。
- 每篇深挖必须记录“未找到的关键证据”，否则容易把未知误写成确定。

Data 对深挖内参的通过线：

1. 至少 5 个来源，其中至少 2 个 S 级或一手来源。
2. 至少 1 个反证或削弱判断的来源 / 事实 / 变量。
3. 明确区分事实、推断、观澜判断和待验证问题。
4. 明确关联 Trend / Opportunity / Judgment Node 候选。
5. 可被后续 7 / 30 / 90 天回测。

## 1.2 判断资产循环

PM Agent 与 Intelligence Data Agent 共同确认：日常监测算法 v2 不只是内容生产流程，而是判断资产循环。

核心链路：

```text
趋势层 Trend
（行业变化 / 资本流向 / 技术拐点）
        ▲
        │
        ├───────────────┐
        │               │
信号层 Signal       判断层 Insight
（每日筛选 3 条）   （观澜核心判断）
        │               │
        ▼               ▼
机会层 Opportunity
（1 张机会卡：谁付钱 / 怎么做 / 能否迁移）
        │
        ▼
验证层 MVP
（7-14 天测试：是否成立）
        │
        ▼
沉淀层 Database
（机会库 / 场景库 / 项目库 / 风险库）
        │
        ▼
分发层 Content
（内参 / 朋友圈 / 海报 / 短视频）
        │
        ▼
用户层 User
（老板 / 合伙人 / 操盘手）
        │
        ▼
需求反馈 Feedback
（行业问题 / 企业需求 / 项目线索）
        │
        ▼
回流到 Signal
```

### 层级定义

| 层级 | 作用 | 输出 |
|---|---|---|
| Trend | 判断行业变化、资本流向和技术拐点是否形成连续方向 | 趋势状态、证据阶梯、反证变量 |
| Signal | 每日从噪音中筛出 3 条高价值变化 | 短深度 Signal 文章 |
| Insight | 形成观澜自己的核心判断 | 判断句、证据边界、反证 |
| Opportunity | 将判断转成可讨论机会 | 谁付钱、怎么做、能否迁移 |
| MVP | 7-14 天验证机会是否成立 | 访谈、搜索、原型、客户验证 |
| Database | 沉淀长期资产 | 机会库、场景库、项目库、风险库 |
| Content | 把判断转成可传播内容 | 内参、朋友圈、海报、短视频 |
| User | 面向真实商业用户 | 老板、合伙人、操盘手 |
| Feedback | 捕捉真实需求与项目线索 | 行业问题、企业需求、项目机会 |

### PM 规则

- 每日 3 条 Signal 必须能服务一个更高层 Trend 或 Insight，否则只是新闻筛选。
- 每张 Opportunity 必须能回答“谁付钱 / 怎么做 / 能否迁移”，否则不得进入深挖。
- 每篇深挖内参必须为 MVP 验证留出问题和路径，否则不能沉淀为商业资产。
- 分发层不是简单改写，而是从同一判断资产生成不同表达形态。
- 用户反馈必须回流到 Signal 和 Trend，用于调整关键词、来源和机会优先级。

### Data 规则

每个层级都要有稳定 ID 和关系：

```text
trend_id -> signal_id -> point_id -> insight_id -> opportunity_id -> mvp_id -> database_record_id -> content_id -> feedback_id
```

短期可先从以下字段开始：

- `trend_ids`
- `signal_ids`
- `point_ids`
- `insight_id`
- `opportunity_id`
- `mvp_validation_status`
- `database_targets`
- `content_derivatives`
- `feedback_items`

关系检查优先级：

1. 精选 Signal 必须关联 Trend 或 Insight。
2. Point 可关联 Trend / Insight / Opportunity，但不得作为事实证据直接加权。
3. 深挖 Opportunity 必须关联至少 1 条 Signal 和 1 个 Trend / Insight。
4. MVP 必须关联 Opportunity，并有 7-14 天验证结果。
5. Database 沉淀必须记录来自哪个 Signal / Point / Opportunity / MVP。
6. Feedback 必须能回流为新 Signal、Trend 反证或 Opportunity 修正。

## 1.7 Point 观点层

Point 是一线观点输入层，连接 The Point 与 v2 监测管线。

Point 不等于事实证据。它的作用是：

- 提供趋势判断线索。
- 暴露行业共识、分歧和边界。
- 帮助形成 Insight。
- 为 Opportunity 提供假设和反证问题。

### Point 写入模板

```markdown
## Point #001｜某某认为 Agent 将先改造企业内部流程

- 来源人物：某某
- 身份标签：创业者 / 投资人 / 研究者 / 大厂负责人
- 原始观点：一句话概括
- 观澜提炼：这句话真正指向什么趋势
- 商业含义：影响哪个行业、岗位或流程
- 关联趋势：AI Agent / AI Coding / 企业工作流
- 机会关联：是否能进入机会卡
```

### Point 入链规则

- 如果 Point 与当日 Signal 指向同一趋势，可进入 Insight 的 supporting context。
- 如果 Point 提供谨慎、质疑或边界，应进入 counter_evidence 或 what_would_weaken。
- 如果 Point 指向明确客户、流程或付费问题，可作为 Opportunity 假设来源。
- 如果 Point 只是观点热度，没有事实支撑，不得推动精选 Signal 或 Opportunity 升级。

### Point 字段

```yaml
point_id:
date:
person:
person_role:
original_view:
guanlan_extract:
business_meaning:
related_trend_ids:
related_signal_ids:
related_insight_ids:
related_opportunity_ids:
stance: bullish | cautious | critical | explanatory
evidence_role: context | hypothesis | counter_evidence | boundary
```

## 1.8 Builder 共识聚类

The Point 不只沉淀单条观点，还要形成 Builder 共识聚类。

共识聚类用于判断一个观点是否从个人判断升级为趋势线索。它仍然不是事实证据，但可以增强 Insight 的观点上下文。

### Builder 共识模板

```markdown
## 今日Builder共识｜AI正在重构销售岗位

来源：
- VC A
- 创业者 B
- YC项目 C

共识提炼：
销售前端（获客、筛选、跟进）正在被AI统一替代

观澜判断：
这是一个强趋势，而不是单点观点
```

### 共识入链规则

- 至少 3 个独立 Point 指向同一趋势，才可形成 Builder 共识。
- 来源角色应尽量覆盖不同视角：投资人、创业者、研究者、大厂负责人、客户或生态方。
- 共识只能增强 Trend / Insight 的可信度，不能替代 Signal 的事实证据。
- 如果共识与事实信号相反，应写入分歧和反证观察，而不是硬合并。

## 1.9 Signal 详情页决策路径

Signal 详情页的核心逻辑要从“信息展示”升级为“决策工具”。

用户阅读路径：

```text
看标题
↓
看评分
↓
理解趋势
↓
判断影响
↓
看到机会
↓
产生行动
```

### 信息结构

| 顺序 | 模块 | 用户要完成的判断 |
|---|---|---|
| 1 | 标题 | 这是什么变化 |
| 2 | 评分 | 它值不值得优先看 |
| 3 | 趋势 | 它是否属于连续变化 |
| 4 | 影响 | 它影响收入、成本、岗位、流程还是渠道 |
| 5 | 机会 | 它是否能转成可验证机会 |
| 6 | 行动 | 7 / 14 / 30 天应继续看什么 |

页面可以后续再改，但内容层必须先按这条路径组织。

## 1.3 Insight 判断层

v2 增加 `Insight` 作为 Signal 与 Opportunity 之间的判断层。

Insight 不是一句总结，也不是营销观点。它是观澜对一组 Signal / Trend 的核心判断。

### Insight 必须回答

1. 这组变化说明了什么？
2. 它改变了哪个行业、客户、成本、渠道、流程或监管判断？
3. 它支持哪个 Opportunity？
4. 哪些事实会削弱这个判断？
5. 未来 7 / 30 / 90 天看什么？

### Insight 写法

推荐：

```text
企业客服 Agent 的变化不在“能不能回答问题”，而在它开始承接品牌语气、工单升级、合规边界和运营指标。机会更可能出现在 Agent 运营控制层，而不是单一聊天机器人。
```

避免：

```text
AI 客服市场前景广阔，值得重点关注。
```

### Insight 字段

```yaml
insight_id:
title:
date:
related_signal_ids:
related_trend_ids:
related_opportunity_ids:
judgment:
supporting_evidence:
counter_evidence:
what_would_confirm:
what_would_weaken:
next_7d:
next_30d:
next_90d:
```

## 1.4 MVP 验证层

MVP 不是开发产品，而是在 7-14 天内验证这个机会是否值得继续投入判断资源。

### MVP 可选形式

- 访谈 5-10 个目标客户。
- 做一页需求验证问卷。
- 做一个低保真原型或服务包说明。
- 检索采购、招标、招聘和客户案例。
- 找 3 个同行或潜在合作方验证可行性。
- 用朋友圈、社群或私域内容测试真实反馈。

### MVP 输出

```yaml
mvp_id:
opportunity_id:
validation_window: 7d | 14d
hypothesis:
target_users:
test_method:
success_signal:
failure_signal:
findings:
decision: continue | revise | pause | archive
feedback_items:
```

MVP 的结果必须回流：

- 继续：进入 Opportunity / Database。
- 修正：更新 Insight 和 Opportunity。
- 暂停：写入风险库。
- 归档：进入项目库或素材库，不继续占用每日深挖名额。

## 1.5 Database 沉淀层

深挖内参之后必须沉淀，不允许只生成文章。

至少沉淀四类库：

| 库 | 内容 |
|---|---|
| 机会库 | 已形成方向、客户、付费方式和验证路径的机会 |
| 场景库 | 客户类型、业务流程、痛点、预算方和替代流程 |
| 项目库 | 可被资源连接、服务化或产品化的项目线索 |
| 风险库 | 反证、监管、成本、竞争、交付和平台依赖 |

每篇深挖内参必须在结尾列出：

- 进入机会库的条目。
- 进入场景库的条目。
- 进入项目库的条目。
- 进入风险库的条目。

## 1.6 Content 分发层

分发层不是降低质量，而是把同一判断资产改写成不同传播形态。

同一深挖内参可派生：

- `内参版`：3000-6000 字。
- `朋友圈版`：300-600 字，保留判断和边界。
- `海报版`：1 个判断 + 3 个证据 + 1 个反证。
- `短视频版`：60-120 秒脚本，按问题、变化、机会、风险组织。

禁止：

- 为传播牺牲证据边界。
- 把“观察”写成“确定”。
- 把“可验证问题”写成“行动指令”。

| 数据与运营依赖 | 每日 raw 候选、二次搜索、来源分层、去重、趋势归类和 QA 复核 |

| 前台 / 后台边界 | 测试页不进入普通主导航；不展示 prompt、编辑、同步、JSON 或生产后台词 |
| 非目标 | 不替换正式 Signals；不新增一级栏目；不输出确定性建议 |
| 决策 | 原型验证 / 测试管线 |
| 下一步 owner | Data 维护模型，Dev 维护同步脚本，QA 连续 7 天复核 |

成熟条件：

- 连续 7 天稳定产出。
- 每日前台 3 条均有二次搜索和至少 3 个 S/A/B 来源。
- 深挖机会卡每天最多 1 条，且行动地图完整。
- QA 证明前台 3 条质量优于现有 Signals。
- PM / Data / QA 决定是否进入正式入站任务。

## 3. 模块决策表

| 决策项 | 结论 |
|---|---|
| 模块名称 | Signal Lab |
| 用户是谁 | 观澜AI内容与判断团队；后续可作为会员内参样张候选 |

| 非目标 | 不做新闻流、公司榜、投资建议或正式导航 |

## 4.1 Structured 入库写作标准

Structured 不是前台文章，但必须具备可复盘的完整度。每条 5-8 条入库 Signal 必须包含以下内容。

### 必填结构

1. `一句话判断`：说明这条变化可能改变哪个商业判断。
2. `事实背景`：谁、在什么时间、做了什么，事件类型是什么。
3. `来源摘要`：来源分别提供了哪些事实，不合并成一句笼统来源。
4. `商业信号解释`：这不是普通新闻的原因。
5. `6 维度分析`：
   - 解决什么具体问题？
   - 目标客户是谁？
   - 替代或优化什么流程？
   - 商业模式是什么？
   - 为什么现在值得关注？
   - 是否可迁移中国市场？
6. `反证 / 风险`：至少写 2 个可能削弱判断的变量。
7. `证据缺口`：说明目前缺什么证据。
8. `关联候选`：Trend / Opportunity / Judgment Node 候选。
9. `是否进入精选 3 条`：写入是 / 否，并说明原因。

### 文字要求

- 每条 Structured 建议 700-1200 字。
- 不写“提升效率”“市场空间巨大”这类空泛句，除非后面跟具体客户、流程或成本。
- 不能只把 6 个问题各写一句；每一项至少要解释一个真实业务关系。
- 对 C 级来源触发的线索，必须说明已经回找或尚未回找到哪些 S/A/B 来源。

## 4.2 精选 3 条 Signal 写作标准

精选 3 条是未来前台或内参的候选文字，不是列表卡片。每条必须是“短深度信号文章”。

### Signal Selector 角色

从 filtered candidate signals 中选出最值得商业决策者关注的 TOP 3。

选择规则：

- 必须强影响收入、成本或核心业务流程。
- 必须清楚指向趋势，而不是孤立事件。
- 必须能用老板听得懂的语言解释。
- 优先选择有融资、客户采用、收入、采购、产品落地等强证据的信号。
- 避免 3 条都来自同一子类，例如全部是 AI SDR、全部是模型发布或全部是融资。

### 每条必须包含

1. `标题`：事件 + 商业含义，不以公司名作为标题核心。
2. `发生了什么`：不超过 80 个中文字符。
3. `为什么重要`：不超过 100 个中文字符。
4. `商业含义`：不超过 100 个中文字符。
5. `谁应该关注`：老板 / 运营 / 销售 / 投资人 / 产品 / 渠道 / 合伙人等。
6. `来源`：至少列出主要来源。
7. `二次搜索补强`：
   - 至少 3 个 S/A/B 来源。
   - 每个来源写清提供了什么增量事实。
   - C 级来源只记录为线索，不写成支撑证据。
8. `Guanlan Insight（观澜判断）`：1-2 句解释这个信号对商业真正意味着什么，必须是决策导向，不是描述性总结。
9. `反证与边界`：写清至少 2-3 个削弱判断的变量。
10. `未入选对比`：说明为什么其他结构化候选没有进入今日 3 条。
11. `后续观察`：7 / 30 / 90 天看什么。

### 精选 3 条输出模板

```markdown
### Signal Title

- What happened：≤80 中文字符
- Why it matters：≤100 中文字符
- Business implication：≤100 中文字符
- Who should care：老板 / 运营 / 投资人等
- Source：

### Guanlan Insight（观澜判断）

1-2 句说明这个信号对商业真正意味着什么。
```

### 长度和质量

- 每条精选 Signal 可先用上方短模板做决策级筛选，再扩展为 1200-1800 字的短深度信号文章。
- 3 条合计应形成一份可阅读的“今日信号内参”，不是泛泛列表。
- 如果某条无法说明收入、成本或核心业务流程变化，通常不应入选 3 条。
- 精选 3 条必须有明确主次：第一条可作为今日主线，第二、第三条服务不同赛道或反证维度，避免都集中在同一类融资新闻。

### 示例：AI SDR 资本下注

```markdown
## Signal：AI SDR 继续获得资本下注

### 事实
某 AI 销售 Agent 公司融资。

### 6 维度分析
- 解决问题：
- 目标客户：
- 替代流程：
- 商业模式：
- 为什么现在：
- 迁移判断：

### 趋势归属
AI Agent 正在重构销售前端。

### 相关历史信号
- 11x.ai
- Artisan
- Regie.ai

### 观澜判断
销售岗位不是消失，而是被拆成“AI 执行 + 真人成交”。

### 机会卡关联
私域 AI 销售 Agent
```

## 4.3 深挖内参写作标准

深挖机会每天最多 1 篇。它不是“机会卡”，而是内参级深度报道草稿。

### Opportunity Analyst 角色

从精选 3 条 Signals 中选择 1 个最值得深挖的机会。

选择标准：

- 重复信号：来自趋势，而不是孤立事件。
- 商业潜力强：能影响收入、成本、转化、交付或客户留存。
- 匹配真实业务场景：有明确行业、角色和流程。
- 能快速验证：7-14 天内可以访谈、测试、搜索或做低成本原型。

### 选题门槛

满足以下条件才允许深挖：

- 至少 1 条精选 Signal 或 2 条 Structured Signals 指向同一机会方向。
- 能补到至少 5 个来源，其中至少 2 个 S 级 / 一手来源。
- 能明确目标客户和替代流程。
- 能写出反证，而不是只有正向论述。
- 能形成 7 / 30 / 90 天观察框架。

不满足时，当日写：

```text
今日暂无足够证据支撑深挖内参。
```

### 深挖内参固定结构

1. `标题`：机会方向 + 当前证据阶段，不写公司名标题。
2. `摘要`：300-500 字，交代核心判断、证据来源和不确定性。
3. `一、事件背景`：从触发 Signal 写起，说明它不是孤立新闻。
4. `二、产业语境`：这个方向之前如何演进，现在出现了什么变化。
5. `三、客户与痛点`：目标客户是谁，痛点如何发生，预算或责任人是谁。
6. `四、替代流程`：替代什么旧流程，旧流程成本、低效或风险在哪里。
7. `五、解决方案形态`：产品或服务可能如何交付，哪些能力是必要组件。
8. `六、商业模式`：怎么收费，谁付费，按什么指标证明价值。
9. `七、市场与竞争`：已有玩家、替代方案、平台挤压和进入壁垒。
10. `八、中国市场迁移`：本土客户、渠道、监管、数据、交付和竞争差异。
11. `九、反证与风险`：至少 5 条，不只写“竞争激烈”。
12. `十、行动地图`：
    - 可验证问题。
    - 应访谈客户。
    - 应观察指标。
    - 应继续搜索的来源。
    - 7 / 30 / 90 天观察点。
13. `附录：来源表`：列出来源、层级、URL、提供的事实和可信度备注。
14. `证据缺口`：明确哪些关键事实尚未找到。

### Opportunity Card 先行模板

正式深挖前，先输出一张结构化 Opportunity Card。卡片通过后，再扩展为 3000-6000 字内参。

```markdown
## Opportunity Name

## Target Customer
具体到行业 + 角色。

## Core Pain Point
具体痛点，不写抽象概念。

## AI Solution
清楚说明 AI 做了什么。

## Business Model
subscription / usage / outcome-based / service fee

## Why Now
最近发生了什么变化，让这件事现在可行。

## China / Chinese Market Migration
- 能否复制？
- 如果不能复制，能改造成什么？

## MVP Plan (7-14 days)
- Step 1
- Step 2
- Step 3

## Key Risk
最可能失败在哪里。

## Opportunity Score
1-10
```

质量要求：

- 避免泛泛想法。
- 必须具体、可执行。
- 关注商业，不以技术能力为中心。
- 语气务实，像创始人做机会判断。

### 长度和质量

- 每篇深挖内参建议 3000-6000 字。
- 必须像深度报道：有背景、有场景、有冲突、有反证、有未解问题。
- 不能只有“问题、客户、商业模式、风险”几个字段式段落。
- 不写“建议立即做”“应该入局”“确定机会”等行动指令。
- 允许形成判断，但必须保留证据边界。

## 5.6 机会卡评分系统（10 分制）

Opportunity 层使用 10 分制。核心原则：

```text
不是评估“这个机会看起来大不大”，而是评估“这个机会值不值得老板关注、是否值得进入项目库”。
```

### 总分公式

```text
机会总分 = 市场价值 × 20%
        + 痛点强度 × 15%
        + 付费意愿 × 15%
        + AI适配度 × 15%
        + 迁移可行性 × 15%
        + MVP可验证性 × 10%
        + 风险可控性 × 10%
```

满分 10 分。

### 7 个评分维度

| 维度 | 权重 | 9-10 分 | 7-8 分 | 5-6 分 | 3-4 分 | 1-2 分 |
|---|---:|---|---|---|---|---|
| 市场价值 | 20% | 大市场，高增长，多个行业可复用 | 细分市场较大，有增长空间 | 市场存在但规模一般 | 小众需求，增长有限 | 市场过窄 |
| 痛点强度 | 15% | 高频刚需，直接影响收入或成本 | 明显痛点，替代方案不满意 | 有痛点但不紧急 | 效率优化，不是强需求 | 伪需求 |
| 付费意愿 | 15% | 明确预算方，成熟付费习惯 | 有付费可能，预算需教育 | 认可价值但付费不稳定 | 想用但不愿付费 | 无付费方 |
| AI适配度 | 15% | 高频、重复、标准化、数据丰富 | AI 明显增强流程，仍需人工 | AI 只能辅助 | 依赖复杂人工判断 | AI 只是噱头 |
| 迁移可行性 | 15% | 国内需求强，合规低，渠道可落地 | 可迁移，需本地化改造 | 部分模块可借鉴 | 监管、数据、支付或平台限制明显 | 基本不可迁移 |
| MVP可验证性 | 10% | 7-14 天可验证，成本低，指标清晰 | 30 天内可验证 | 需客户配合，周期较长 | 依赖复杂系统或大量数据 | 短期无法验证 |
| 风险可控性 | 10% | 监管、交付、技术、平台风险较低 | 有风险但可控 | 明显风险，需要谨慎 | 高监管、高交付、高信任风险 | 不适合作为当前机会 |

### 评级体系

| 总分 | 等级 | 判断 |
|---:|---|---|
| 9.0-10 | S级机会 | 值得重点跟踪，可进入项目库 |
| 8.0-8.9 | A级机会 | 值得关注，可做机会卡 |
| 7.0-7.9 | B级机会 | 入库观察，不急于深挖 |
| 6.0-6.9 | C级机会 | 作为趋势信号保留 |
| <6.0 | D级机会 | 不建议进入机会库 |

```markdown
## 机会评分

| 维度 | 权重 | 分数 | 加权分 | 判断理由 |
|---|---:|---:|---:|---|
| 市场价值 | 20% |  |  |  |
| 痛点强度 | 15% |  |  |  |
| 付费意愿 | 15% |  |  |  |
| AI适配度 | 15% |  |  |  |
| 迁移可行性 | 15% |  |  |  |
| MVP可验证性 | 10% |  |  |  |
| 风险可控性 | 10% |  |  |  |
| **总分** | **100%** |  |  |  |
```

### 反向扣分项

每命中一个，扣 0.3-1 分：

1. 强监管风险：金融、医疗、教育承诺、投资建议等。
2. 平台封禁风险：依赖抖音、小红书、微信等平台高敏动作。
3. 交付过重：必须深度定制、长期陪跑才能出效果。
4. 客户预算不确定：客户感兴趣但没有明确预算。
5. 同质化严重：工具容易被抄，缺少数据或场景壁垒。

### 展示边界

前台只展示：

```markdown
机会评分：8.3 / 10
等级：A级机会
判断：值得关注，可进入机会卡库，但需控制合规和话术风险。
```

详情页展示完整评分表。

一句话原则：

```text
高分机会 = 大市场 + 真痛点 + 有人付钱 + AI真能替代 + 能迁移 + 能快速验证 + 风险可控。
```

## 6. 去重规则

- 使用 `company + product + event_type` 三元去重。
- 同一融资新闻多来源合并到一个 pool 项。
- 同一产品发布和客户案例若说明同一商业变化，应合并并记录新增证据。
- Builder 观点只能作为线索或观点边界，不冒充事实证据。
- 若已进入 `01-Signals/`，在 v2 管线标记 `already_in_production`。
- 重复但有新增证据的内容进入“已跟踪信号补充”，不新增前台样张。

## 7. 二次搜索规则

进入前台 3 条和深挖机会卡前必须执行二次搜索：

- 回找公司官网、产品页、公告、客户案例、价格页或开发者文档。
- 回找投资方、融资数据库、监管、采购、垂直行业来源。
- 回找反证，包括客户流失、毛利压力、诉讼、安全、隐私、监管和平台依赖。
- 每条前台 Signal 至少 3 个来源，C 级来源只能触发检索。
- 深挖机会卡至少 5 个来源，并说明未找到的证据。

### 7.1 检索强度分层

```text
初筛靠广度。
精选靠评分。
深挖靠二次搜索和交叉验证。
```

| 阶段 | 检索目标 | 检索强度 |
|---|---|---|
| Raw / Pool | 覆盖更多方向，避免只看大公司和大融资 | 广度优先 |
| Structured / TOP 3 | 用评分和轻量二次搜索确认是否具备证据 | 评分 + 轻量二搜 |
| Deep Dive | 形成内参级机会判断 | 重搜索 + 交叉验证 |

### 7.2 精选 3 条：轻量二次搜索

每条入选 TOP 3 的 Signal，至少补充查 3 类信息：

1. `公司 / 产品官网`  
   确认它到底做什么，不只看媒体转述。

2. `融资或投资人信息`  
   确认金额、轮次、投资方，以及资本为什么关注。

3. `客户 / 案例 / 定价信息`  
   确认是否有真实商业化，不只停留在发布会。

只有完成以上三类补充，精选 Signal 才能被视为有证据。

### 7.3 深挖机会卡：重搜索与交叉验证

每天 1 张深挖机会卡至少做 5 类交叉验证：

1. `代表公司`  
   这个方向是否有 3 家以上公司在做。

2. `融资信号`  
   最近 6-12 个月是否持续融资。

3. `客户场景`  
   服务的是谁，客户是否真实付费。

4. `定价 / 商业模式`  
   是订阅、按席位、按量、按结果，还是服务费。

5. `中国迁移卡点`  
   分别检查监管、平台、数据、支付、客户预算、交付难度。

### 7.4 深挖证据链模板

每张机会卡最后必须增加：

```markdown
## 证据链

- 公司证据：
- 融资证据：
- 客户证据：
- 商业模式证据：
- 迁移可行性证据：

## 反向证据

- 哪些因素说明这个机会可能不成立：
```

证据链必须写具体事实、来源和链接，不得只写“有融资”“有客户”“可迁移”。

## 8. 趋势归类

所有结构化 Signal 都要写入趋势归类，不只写前台 3 条。

- `emerging`
- `warming`
- `diverging`
- `cooling`
- `risk`

- `trend_id`
- `trend_name`
- `status`
- `related_signal_ids`
- `related_opportunity_ids`

## 8A. Signal -> Trend 自动生成逻辑

核心原则：

```text
Signal 是点，Trend 是由多个高质量 Signal 反复验证出来的方向。
```

因此，不是每条 Signal 都能生成 Trend。无法支撑长期方向的 Signal，只能入库观察，不能污染趋势库。

### 8A.1 流程

```text
Signal 入库
↓
趋势标签识别
↓
相似信号聚类
↓
趋势强度评分
↓
判断：挂载旧 Trend / 新建 Trend / 仅入库观察
↓
更新趋势阶段、热度、置信度
```

### 8A.2 每条 Signal 入库后先打 6 个标签

每条 Signal 必须标注：

| 标签 | 候选值 |
|---|---|
| 场景标签 | 销售 / 客服 / Coding / 知识库 / 内容增长 / 法务 / 招聘 |
| 技术标签 | Agent / RAG / Voice AI / Workflow / AI Coding / Multimodal |
| 商业标签 | 融资 / 产品发布 / 客户案例 / 定价 / 并购 / 大厂动作 |
| 影响对象 | 老板 / 销售团队 / 客服团队 / 开发者 / 运营 / HR / 法务 |
| 价值维度 | 增收 / 降本 / 提效 / 替代岗位 / 新产品机会 |
| 迁移判断 | 中国可迁移 / 部分可迁移 / 暂不适合 |

这 6 类标签用于后续聚类。它们不是正式 Tags 字典的替代，而是趋势生成的中间判断字段。

### 8A.3 判断是否挂到已有 Trend

新 Signal 与已有 Trend 的匹配看 4 类相似度：

1. `场景相似`  
   例如 AI 销售自动化、AI 客服 Agent、AI Coding、企业知识库。

2. `工作流相似`  
   例如线索筛选、客户问答、代码生成、文档检索、销售跟进。

3. `商业信号相似`  
   例如融资、客户落地、大厂发布、开源项目爆发、VC 观点。

4. `关键词相似`  
   例如 AI SDR / sales agent / outbound automation，Voice AI / call center / customer support，vibe coding / AI coding / app builder。

如果相似度高，则挂载到已有 Trend，并更新该 Trend。

### 8A.4 新建 Trend 条件

新建 Trend 必须克制，避免趋势库膨胀。

满足以下条件才允许新建：

```text
过去 30 天内，至少出现 3 条相关高质量 Signal
且来自至少 2 类不同来源
且平均 Signal 评分 >= 7.5
且能说清楚“它正在重构哪个岗位 / 流程”
```

来源类型至少包含两类：

- 融资信号。
- 产品发布。
- Builder 观点。
- 客户案例。
- 大厂动作。
- 投资机构观点。
- 开源项目增长。

示例：

```text
11x.ai 融资
Artisan AI 增长
Salesforce 推 Agentforce
某 VC 讨论 AI SDR
```

可新建：

```text
Trend：AI Agent 正在重构销售前端
```

### 8A.5 Trend 强度评分（10 分制）

```text
Trend Score =
信号数量 × 20%
+ 信号质量 × 20%
+ 来源多样性 × 15%
+ 商业化强度 × 20%
+ 迁移可行性 × 15%
+ 增长速度 × 10%
```

评分解释：

| 维度 | 权重 | 判断 |
|---|---:|---|
| 信号数量 | 20% | 最近 30 / 90 天相关 Signal 越多，分越高 |
| 信号质量 | 20% | 高评分 Signal 越多，分越高 |
| 来源多样性 | 15% | 融资、产品、客户、大厂、Builder 多源验证比分散媒体更高 |
| 商业化强度 | 20% | 是否有真实客户、明确收费、企业采购、融资持续、ARR 增长 |
| 迁移可行性 | 15% | 是否能迁移到中国 / 华人市场或项目系统 |
| 增长速度 | 10% | 最近信号是否加速出现 |

### 8A.6 Trend 生命周期阶段

| 阶段 | 判断规则 |
|---|---|
| 萌芽期 | 少量 Builder 观点 / 开源项目 / 早期产品，商业化弱 |
| 起步期 | 出现种子轮 / A 轮融资，有明确产品，开始有客户试点 |
| 加速期 | 多家公司融资，大厂跟进，企业开始采购，媒体和 VC 持续讨论 |
| 爆发期 | 企业批量采用，头部公司收入增长明显，竞品快速增加，资本密集进入 |
| 成熟期 | 模式稳定，客户教育完成，竞争格局初步形成 |
| 衰退 / 泡沫期 | 融资热但客户留存差，产品同质化严重，监管 / 平台 / 成本问题暴露 |

### 8A.7 Signal 如何更新 Trend

每条新 Signal 入库后，对 Trend 产生 5 个更新动作：

1. 增加信号数量。
2. 更新热度。
3. 更新趋势阶段。
4. 更新相关机会卡。
5. 更新观澜判断。

示例：

```text
新 Signal：Salesforce 发布 Agentforce 2.0
↓
挂载 Trend：AI Agent 重构销售流程
↓
更新：
- 信号数 +1
- 大厂动作权重 +1
- 趋势阶段从“起步期”提升为“加速期”
- 关联机会卡：私域 AI 销售 Agent
- 新判断：大厂正在推动 AI 销售 Agent 标准化
```

## 8.1 Trend Classifier

每条 Signal 必须归入 1 个主趋势。只能选一个 primary trend，避免一条信号被贴多个方向导致判断发散。

候选趋势：

- `AI Sales Automation`
- `AI Customer Support`
- `AI Knowledge Base`
- `AI Workflow Automation`
- `AI Coding / Vibe Coding`
- `One-person Company`
- `AI Content Growth`
- `Other`，必须说明具体趋势名。

输出模板：

```markdown
## Trend Classification

- Trend name：
- Why this signal belongs here：
- Related past signals：
```

Data Agent 规则：

- 趋势归类要优先看客户流程，而不是公司所属行业。
- AI SDR、销售外呼、线索生成、销售跟进优先归入 `AI Sales Automation`。
- 客服、工单、退货、理赔、呼叫中心优先归入 `AI Customer Support`。
- 企业搜索、RAG、知识管理、内部问答优先归入 `AI Knowledge Base`。

- 编程 Agent、IDE、代码生成、vibe coding 优先归入 `AI Coding / Vibe Coding`。

- AI 内容、增长、SEO、投放和素材生产优先归入 `AI Content Growth`。

### 每日最低完成线

1. `01-raw` 30-50 条。

6. `06-insights` 至少 1 条。
7. `10-databases` 至少更新趋势库。

Raw 最低线补充：

- 每条 Raw 必须对应一个本地原文档案。
- 本地原文档案路径统一为 `06-content/01-raw/originals/YYYY-MM-DD/R-xxx-source-title.md`。
- Raw 列表中的 30-80 字是采集理由，不替代原文。
- 只有链接、没有本地原文档案的线索，不算完成 Raw 入库。

深挖内参不是每日硬凑项。证据不足时，正确输出是：

今日暂无足够证据支撑深挖内参。

## 11. 06-content 目录优化

`06-content/` 使用编号目录，按判断链路排序：

```text
00-inbox
01-raw
02-pool
03-structured-signals
04-selected-signals
05-trend-chain
06-insights
07-points
08-opportunities/deep-dive
09-mvp-validation
10-databases
11-content-distribution
12-feedback
_archive
```

目录治理规则：

- `00-inbox` 只放临时线索，不作为已入库内容；尚未保存本地原文的链接先留在这里。
- `01-raw` 到 `04-selected-signals` 是信号漏斗，不混放深挖文章。
- `01-raw/originals/YYYY-MM-DD/` 保存 Raw 本地原文档案，后续 Pool / Structured / Selected 都应引用该档案。
- `05-trend-chain` 只负责趋势挂载、评分、阶段和长期观察指标。
- `06-insights` 是观澜判断层，不写成新闻摘要。
- `07-points` 只收观点和 Builder 共识，不能冒充事实证据。
- `08-opportunities` 只放机会卡和深挖内参，机会标题不写公司名。
- `09-mvp-validation` 记录 7-14 天验证，不等同于开发需求。
- `10-databases` 分为 `trends`、`opportunities`、`scenarios`、`projects`、`risks`、`tag-map`。
- `11-content-distribution` 分为 `brief`、`moments`、`poster`、`short-video`。
- `12-feedback` 的反馈必须能回流到 Signal、Trend 反证或 Opportunity 修正。

## 12. 标签与标题体系融合规则

v2 使用两层标签，防止标签膨胀：

| 字段 | 作用 | 是否进入正式迁移 |
|---|---|---|
| `primary_title_family` | 对齐前台栏目：Signals / The Point / Opportunities / Trends / Daily Brief | 是 |
| `primary_trend_label` | 对齐标题和趋势理解，如 AI Sales Automation / AI Coding | 是，作为趋势归类字段 |
| `formal_tags` | 使用 `tag-taxonomy.md` 中已有 `tag_id` | 是 |
| `classification_labels` | v2 中间聚类字段：场景、技术、商业、影响对象、价值维度、迁移判断 | 否 |
| `candidate_tags` | 新词候选池 | 否，需复核后迁移 |

### 标题体系映射

| v2 趋势标题 | 优先映射到现有正式标签 |
|---|---|
| AI Sales Automation | `track-ai-marketing` + `function-sales` |
| AI Customer Support | `track-ai-customer-service` + `function-customer-service` |
| AI Knowledge Base | `track-enterprise-data` + `scenario-knowledge-base` |
| AI Workflow Automation | `track-enterprise-workflow` + `function-operations` |
| AI Coding / Vibe Coding | `track-ai-coding` + `function-engineering` |
| One-person Company | 先作 `primary_trend_label`，不立即新增 track |
| AI Content Growth | `track-ai-marketing` + `function-marketing` |

### 标签准入

- 单条 Signal 最多 6 个正式标签。
- 单条 Opportunity 最多 7 个正式标签。
- Point 不以人物姓名做 tag。
- 公司名、产品名、轮次名、情绪词、宽泛词不进入正式 tag。
- 新标签必须先进入 `candidate_tags`；满足 3 条以上资产、跨 2 个日期，或经 PM / Data 明确确认后，才允许进入正式字典。
- 同义词优先归并到现有 `tag-taxonomy.md`，例如 AI SDR 不新增 track，优先映射到 `track-ai-marketing` + `function-sales` + 候选场景。

迁移标准：

```text
能帮助搜索、筛选、关系网络和趋势沉淀的，才是 Tag。
只能帮助当天理解的，留在 classification_labels。
```
