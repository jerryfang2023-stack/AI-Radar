## 1. V2 设计定位

观澜 AI 是面向商业决策者的 AI 机会判断系统。页面要像高端商业内参和专业研究产品，而不是普通 SaaS 官网、资讯流、后台系统、工具站或 AI 科技氛围页。

核心气质：

- 克制、清晰、有留白。
- 有判断感和商业可信度。
- 像一份可持续阅读的商业情报产品。
- 让用户看见“信号 -> 判断 -> 机会 -> 边界”的路径。

设计不追求炫技，不用装饰证明科技感。每一屏只完成一个主要阅读任务，每个视觉元素都必须服务判断、证据、机会或边界。

## 2. 当前导航与栏目体系

V2 普通前台只采用四个一级导航：

```text
今日观察 / 商业信号 / 趋势追踪 / 商业内参
```

| 前台名称 | 内部承接 | 页面角色 | 更新节奏 |
|---|---|---|---|
| 今日观察 | Daily Observation / AIBriefDaily | 每日行情综述，不做新闻流 | 日更 |
| 商业信号 | ChangeSignal / CaseSignal | 每日高质量商业变化与案例信号 | 日更 |
| 趋势追踪 | TrendReport + TrendCard | 低频、少而精的趋势分析报告 | 证据足够才更新 |
| 商业内参 | AIBriefIssue + HeatCard | 会员 / 增值层周期判断 | 周度或月度 |

不进入普通一级导航：

- `The Point`：降级为 `观点校准`，嵌入今日观察、商业信号、趋势追踪和商业内参。
- `机会判断`：降级为趋势追踪和商业内参中的机会判断段落。
- `商业热力图`：作为商业内参核心模块和后台判断资产，不做公开榜单。
- `Scoring / Priority Engine`：后台化，不作为普通前台栏目。
- `Tags`：作为搜索、筛选和关系网络能力，不作为一线栏目。
- `Admin`：只属于管理员后台，不混入普通前台。

旧 `Daily Brief / Signals / Opportunities` 可作为内部英文语义和路径兼容，但前台显示必须使用 V2 中文命名。

## 3. VI / SVG / Token First

正式 VI 资产目录：

```text
docs/brand/wavesight-ai-vi/
```

站点引用副本：

```text
01-SiteV2/site/assets/brand/
```

强制规则：

- Logo 必须引用正式 SVG 文件，不允许用 CSS、图标库或手写 SVG 重绘。
- Logo 固定为香槟金地平线、三道主澜线、`观澜 AI` 中文字标和 `WAVESIGHT AI` 英文字标。
- V2 网站导航栏 Logo 以当前正在使用的版本为准：`01-SiteV2/site/assets/brand/logo-wavesight-reference-horizontal.svg`。后续不得轻易更换、重绘、替换为其他 Logo 版本或调整识别结构；如确需变更，必须另立任务并经 PM / UI / QA 复核。
- 正式横版 Logo 网页默认宽度不低于 `160px`；报告封面建议不低于 `220px`；小于 `120px` 宽时优先使用 symbol。
- 不得改变澜线数量、去掉香槟金地平线、添加信号点、雷达圈、眼睛、机器人、箭头、蓝紫科技渐变或新的品牌母题。
- 页面、符号、信息卡片和动效分镜优先引用 `executable-svg/` 的单项 SVG，不从规范源图裁切局部图像。
- 如需新增组件，必须基于现有 SVG 风格扩展，保持深海蓝、暖白、雾灰和少量香槟金。
- 页面动效使用 `motion-tokens.css`，保持轻、慢、稳，像水面微动，不做炫技。

站点 token 规则：

- `docs/brand/wavesight-ai-vi/brand-tokens.css` 是 `--gl-*` token 唯一来源。
- 页面 CSS 不得重复定义同名 `--gl-*` token。
- 站点可定义页面级别名，但必须映射到品牌 token，不得另造品牌色、品牌阴影或品牌圆角。

## 4. 色彩系统

以 `brand-tokens.css` 为准：

| Token / 色值 | 用途 |
|---|---|
| `--gl-bg-page` / `#FFFDF8` | V2 公开页面全局背景、导航融合底色 |
| `--gl-bg` / `#F7F4EF` | 品牌纸感底色、物料底色或局部背景 |
| `--gl-paper` / `#FFFFFF` | 卡片、报告页、浮层 |
| `--gl-paper-warm` / `#FFFDF8` | 暖白纸面容器 |
| `--gl-ink` / `#071827` | 主标题、Logo 字标、重点判断 |
| `--gl-deep` / `#0B1D2A` | 深色面、强调区域 |
| `--gl-blue` / `#0D355C` | 品牌主色、链接、图表线 |
| `--gl-gray` / `#6F7F8F` | 次级文字、说明信息 |
| `--gl-mountain` / `#A7ADB4` | 弱信息、边界、禁用态 |
| `--gl-gold` / `#C8A766` | 地平线、分隔线、会员标识、少量强调 |

公开页面背景统一使用 `#FFFDF8` / `--gl-bg-page`。导航栏应无边框、无阴影，背景与页面融合。`#F7F4EF` / `--gl-bg` 不再作为 V2 公开网站唯一全局背景，只用于品牌纸感物料、局部底色或需要弱分区的区域。

禁止：

- 大面积紫色 / 紫蓝 AI 渐变。
- 霓虹科技蓝和高饱和互联网蓝。
- 大面积纯黑压迫。
- 花哨装饰渐变、发光光束和通用科技海报背景。
- 大面积香槟金铺色。

### 标题治理硬规则

所有公开页面必须进入同一套标题层级，不允许新页面、新文章或新详情页自行放大字号。

| 类型 | 适用页面 | 桌面字号 |
|---|---|---:|
| 首页 Hero 标题 | 首页首屏专用 | `56px / 72px` |
| 栏目页标题 | 今日观察 / 商业信号 / 趋势追踪 / 商业内参 | `44px / 58px` |
| 详情页主标题 | 今日观察详情 / 商业信号详情 / 趋势报告详情 | `40px / 56px` |
| 模块标题 | 详情页正文区块、栏目页内容区块 | `28px / 40px` |
| 卡片标题 | 列表卡片、侧栏卡片、关联内容 | `20px / 30px` |

执行要求：

- 新增 HTML 页面必须给 `<body>` 加稳定页面类，例如 `page-signals`、`page-signal-detail`、`page-trend-report-detail`。
- 新增 CSS 不能绕过 `V2 Typography Governance` 自行写 `clamp(60px...)`、`6vw`、`100px` 等海报级字号。
- 标题可以换行，但不能被截断成半句话，也不能因为容器限制显示 `...` / `…`。
- 商业信号、趋势报告、商业内参详情页不是海报页，主标题不得超过 `40px / 56px` 的文章标题层级。

## 6. 页面宽度与布局系统

默认布局：

| 区域 | 桌面规范 | 移动规范 |
|---|---|---|
| 顶部导航 | 约 `72px` 高，Logo 左，四导航中，登录 / 会员右 | 约 `64px` 高，Logo 左，菜单右 |
| 主内容容器 | `1200-1280px` | 左右 `20px`，单列 |
| 报告阅读正文 | `760-860px` | 单列 |
| 详情侧栏 | `320-380px` | 折叠到正文后 |
| 模块间距 | `64-96px` | `48-72px` |

宽度冲突处理：

- `1200-1280px` 是默认主容器。
- `760-860px` 是详情页正文舒适阅读宽度。
- `1440-1520px` 只允许用于商业内参热力图、宽表、证据矩阵或经 UI / QA 明确说明的数据密集模块。
- 不再默认使用 `1520-1680px` 作为普通页面主宽度。

布局原则：

1. 先确定阅读路径，再设计卡片。
2. 页面区块使用全宽分区或无框布局，卡片只承载独立内容项、工具面板、模态框或可重复列表。
3. 公开前台不做后台字段表，不让筛选器、徽章、数字和按钮抢过标题和正文。
4. 首页首屏必须露出下一段内容，不做满屏孤岛。
5. 移动端优先单列、无横向溢出、无遮挡、按钮可点击。

## 7. 圆角、边框、阴影和卡片规则

以 `brand-tokens.css` 为准：

| Token | 用途 |
|---|---|
| `--gl-radius-card: 12px` | 内容卡片、报告样张、独立面板 |
| `--gl-radius-button: 8px` | 按钮、关键交互控件 |
| `--gl-radius-sm: 6px` | 标签、细小状态、轻量控件 |
| `--gl-radius-md: 10px` | 搜索、筛选、工具条 |
| `--gl-radius-lg: 12px` | 大卡片、内参样张 |
| `--gl-shadow-card` | 轻纸面阴影 |
| `--gl-shadow-hover` | 悬浮状态阴影 |

规则：

- 公开前台卡片半径控制在 `6-12px`，优先使用 token，不新增随意圆角。
- 不使用大圆角 Bento 作为默认风格；如有特殊样张必须由 UI / UE 说明理由。
- 不使用卡片套卡片。
- 阴影必须轻，不使用深色重阴影或霓虹外发光。
- 列表和报告结构优先使用细分割线、留白和字体层级，不靠边框堆叠制造秩序。
- 同一屏不要超过三种卡片风格、圆角或阴影系统。

## 8. 页面母版

### 8.1 Home

用户问题：观澜 AI 是什么，今天为什么值得继续读。

结构：

1. Hero：品牌主张、价值说明、两个克制动作。
2. 今日 3 条商业信号摘要。
3. 今日观察预览。
4. 最新 1-2 篇趋势追踪。
5. 商业内参样张与会员入口。

要求：

- 首屏主任务是说明价值和进入今日判断。
- 右侧判断样张必须像纸面内参，不像后台仪表盘。
- 不恢复旧抽象雷达图、通用 AI 氛围图、四张营销利益卡或 Trends / The Point 独立入口。
- CTA 建议使用 `查看今日观察`、`阅读趋势追踪`、`了解商业内参` 等克制表达。

### 8.2 今日观察

用户问题：今天有哪些 AI 变化值得花时间看，为什么。

栏目页结构：

- 日期与今日一句判断。
- 3-5 条要点。
- 今日 3 条商业信号。
- 观点校准摘录。
- 风险与边界。
- 历史日期索引。

详情页结构：

- 标题区：日期、阅读时长、当日判断标签。
- 核心判断：1 段总判断 + 3-5 条要点。
- 今日商业信号。
- 观点校准。
- 机会观察。
- 趋势背景。
- 反证与证据缺口。

验收重点：不能像新闻摘要合集；观点要克制，边界要具体，不输出行动指令。

### 8.3 商业信号

用户问题：哪些外部变化已经足够重要，值得进入判断资产。

栏目页结构：

- 今日精选 3-5 条。
- 历史检索：日期、行业、岗位、流程、关联机会。
- 卡片展示：事件、商业含义、来源数量、影响对象、反证摘要。

详情页结构：

- 标题必须是“事件 + 商业含义”。
- 一句话判断。
- 事实背景。
- 二次来源摘要。
- 机会解析 6 维。
- 商业解释。
- 观点校准。
- 反证与缺口。
- 关联内容。

验收重点：列表页不堆完整报告；详情页不把字段直接摊成卡片。

### 8.4 趋势追踪

用户问题：哪些方向经过足够搜索、论证和反证，值得写成深度机会分析。

栏目页结构：

- 低频、少而精的栏目说明。
- 最新报告。
- 观察中方向。
- 历史报告。
- 内参关联。

详情页结构：

- 报告标题、发布日期、阅读时长、证据状态。
- 一句话机会判断。
- 变化背景。
- 问题与对象。
- 流程变化。
- 价值来源。
- 趋势背景。
- 观点校准。
- 反证与限制。
- 成立边界。
- 建议关注变量。

验收重点：不输出落地方案，不写确定性机会，不用公司名做标题。

### 8.5 商业内参

用户问题：本周 / 本月哪些 AI 变化正在形成更高密度的商业判断。

栏目页结构：

- 最新一期：期号、标题、核心判断、会员状态。
- 本期热力摘要。
- 核心判断列表。
- 证据来源摘要。
- 往期内参。
- 会员提示。

详情页结构：

- 封面区：期号、周期、标题、会员标签、阅读时长。
- 本期总判断。
- 商业热力图。
- 高热三元组。
- 升温 / 降温 / 争议。
- 证据展开。
- 观察边界。
- 往期关联。

验收重点：像周期性商业决策文件，不像普通文章栏目、公开榜单或后台图表。

### 8.6 通用详情页

结构：

```text
顶部：类型 / 日期 / 来源状态
主栏：H1 + 一句话判断 + 正文报告节奏
侧栏：摘要、来源、关联 Signal / Trend / Opportunity / Brief
底部：相关内容
```

规则：

- 正文主栏宽度 `760-860px`。
- 侧栏宽度 `320-380px`。
- 字段必须转成报告阅读结构，不直接摊成卡片。
- 引用只出现一次，不重复截断。
- 侧栏只做摘要和导航，不抢主阅读区。

## 9. 嵌入模块规范

| 模块 | 出现位置 | 展示方式 | 禁止 |
|---|---|---|---|
| 观点校准 | 今日观察 / 商业信号 / 趋势追踪 / 商业内参 | 来源、立场、支持或质疑什么 | 做成独立频道、观点热榜或人物墙 |

| 反证与边界 | 所有详情页 | 风险、反例、证据缺口 | 空泛免责声明 |

## 10. 商业图表规则

图表服务判断，不服务装饰。任何图表都必须有明确问题、数据来源、时间范围和边界说明。

### 10.1 热力图

可用场景：

- 商业内参详情页。
- 会员层热力摘要。
- Admin 判断资产和质量治理。
- Home 只能展示极简样张或摘要，不做完整公开榜单。

禁止：

- 做成公开公司榜、岗位替代榜、新闻热榜。
- 用热力颜色制造确定性机会。
- 无来源、无时间范围、无反证说明。

### 10.2 雷达图

雷达图不是观澜 AI 品牌母题。禁止用于 Logo、首屏主视觉、背景装饰、品牌辅助图形或通用 AI 氛围。

仅在满足以下条件时可作为小型数据可视化：

- 用于比较 3 个以上维度。
- 每个维度有清晰定义和数据来源。
- 出现在报告正文、会员内参或 Admin 中。
- 视觉必须克制，不使用霓虹、发光、科技网格或大屏效果。

多数场景优先使用条形、矩阵、趋势线或文本化判断替代雷达图。

### 10.3 趋势图

可用于展示证据积累、时间变化、升温 / 降温 / 争议状态。

规则：

- 必须解释变化原因，而不是只画线。
- 线条使用深澜蓝、雾蓝灰和少量香槟金。
- 不使用股市 K 线堆叠、复杂金融终端视觉或彩虹趋势线。
- 数据不足时只写趋势背景，不强行画图。

## 11. 文案与标题规则

文案必须言之有物、简洁克制、精炼有力。不讲废话，不把内部流程语言写给外部用户。

栏目标题：

- 一级栏目标题必须保持同一套位置、字号、行高、留白和首屏节奏。
- 标题是一句话价值表达，不写解释型副标题、功能说明或内部使用提示。
- 标题背景默认与页面背景一致，不额外添加突兀色块、装饰块或孤立渐变。
- 如果标题过长，退回 Copy 压缩，不放大容器硬塞。

Signal 标题：

- 必须是“事件 + 商业含义”。
- 不能只是公司名、产品名或新闻标题。

Trend / 趋势追踪标题：

- 不能写公司名。
- 公司只能作为证据、案例或来源出现。
- 不写确定性机会、落地方案或执行建议。

公开前台避免：

```text
本页用于
入口
同步
字段
后台
JSON
编辑
恢复
证据链
强证据
来源明确
机会确定
下一步验证
立即行动
保证收益
赋能
颠覆
风口红利
```

允许的语气：

- 值得继续观察。
- 证据正在增强。
- 需要保留反证。
- 可能形成连续证据。
- 当前更像早期信号。

## 12. 前台 / Admin 边界

普通前台不能出现：

- Admin、JSON、同步、恢复、编辑、字段、后台、脚本、生成状态。
- Raw / Pool / Structured 等生产过程字段。
- 内部质量检查、入站、发布准备等流程语言。

Admin 可以高密度、低装饰，但必须与公开前台区分：

- 左侧窄导航。
- 顶部状态条。
- 高密度工作区。
- 列表 / 筛选 / 审批 / 编辑区。
- 模块框起点、模块标题起点和工具栏高度一致。

Admin 不是本文件前台母版的视觉延伸；后续 Admin 任务应另按后台设计规范验收。

## 13. UI/UE 审美标准和桌面截图验收

页面类任务必须经过 UI/UE 审美标准与桌面证据复核。评分既是创作要求，也是调度复核依据。

| 质检项 | 分值 | 重点 |
|---|---:|---|
| Style Purity / 风格一致性与纯净度 | 20 | 是否遵守澜线 VI，是否有异类视觉 |
| Proportion & Rhythm / 比例与韵律感 | 20 | 首屏、标题、模块节奏是否成立 |
| Color Sophistication / 色彩深度与平衡 | 20 | 暖白、墨海蓝、雾灰、香槟金比例是否克制 |
| Craftsmanship / 细节的艺术处理 | 20 | 圆角、线宽、hover、状态、对齐是否统一 |
| Emotional Resonance / 情感共鸣 | 20 | 第一秒是否像商业判断产品 |

通过线：

- 首页 / 全站母版 / 核心首屏 / 海报 / 视觉资产：`>=85`。
- 一级栏目页 / 详情页 / 会员页：`>=80`。
- Admin 工作台和后台模块：`>=75`。
- 任一单项低于 `14`、Squint Test 不通过或出现审美阻塞项，必须重做。

证据化质检必须包含：

- 设计基准引用。
- 桌面端截图路径。
- 移动端观察；如派发单明确要求移动端专项，则填写截图路径。
- 五项评分和逐项扣分原因。
- Squint Test 结论。
- 必须重做清单。
- 可延后优化清单。
- Dev 实现偏差清单。
- QA 复核建议。

页面实现后必须输出：

- 桌面截图：建议 `1440px` 宽。
- 移动观察：当前 V2.1 采用桌面端优先，移动端专项只有在派发单明确要求时才需要 `390px` 截图并作为硬阻塞。
- 横向溢出检查。
- 首屏重心说明。
- 普通用户、登录用户、会员、管理员四状态截图或说明。
- 与本母版不一致处的 Dev 偏差清单。

本任务只重写设计规范，不实现页面，因此不需要截图；后续页面任务必须补齐截图验收。

## 14. 审美阻塞项

以下情况不得进入 Dev 或不得验收通过：

- 页面粗糙、简陋、像模板页或临时拼装页。
- 首页继续使用旧雷达图、泛营销利益卡或通用 AI 氛围图。
- 栏目标题位置、字号、行高、模块起点不一致。
- 页面像后台组件堆叠，主次不明。
- 徽章、数字、按钮、摘要争抢注意力。
- 详情页把字段直接摊开，没有报告阅读路径。
- 文案过长破坏布局，但 UI 未退回 Copy 压缩。
- 普通前台出现 Admin、JSON、同步、编辑、恢复、字段、后台等痕迹。
- 派发单要求移动端专项时，移动端出现横向溢出、遮挡、标题断行难看或按钮拥挤。

- Apple：留白、克制、主标题冲击力。

## 18. UI / UE Agent 工作流程

每次修改页面前，UI / UE Agent 必须输出：

1. 页面类型和用户任务。
2. 使用的设计基准：观澜 VI、本文件、字体、token、SVG；可补充 Apple / Linear / Stripe 或有效 skill，并说明适配边界。
3. 页面阅读路径。
4. 要删除、弱化或合并的信息。
5. 桌面端验收点；如派发单明确要求移动端专项，再补移动端验收点。
6. UI/UE 审美评分预期和阻塞项。

每次修改页面后，必须检查：

- 是否使用了本文件和 VI 资产。
- 是否过度卡片化。
- 是否有过大留白或过度拥挤。
- 文字是否断行难看。
- 普通用户前台是否出现后台痕迹。
- 是否完成桌面截图验收；移动端专项按派发单另验。
