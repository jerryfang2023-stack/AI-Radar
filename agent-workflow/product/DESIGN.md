# 观澜AI｜WaveSight AI 设计规范

## 设计定位

观澜AI是面向老板、行业操盘者、资源方和投资观察者的 AI 商业情报产品。页面要像高端商业内参和专业研究产品，而不是普通 SaaS 官网、资讯流、后台系统或工具站。

核心气质：

- 克制
- 高级
- 清晰
- 有留白
- 有判断感
- 有商业可信度

## 参考方向

每次 UI/UE Agent 改页面前，必须从以下方向里选 1-2 个作为当次参考：

- Apple：留白、克制、主标题有冲击力。
- Linear：信息密度适中、模块干净、边界清晰。
- Stripe：商业产品感、图表和说明的专业平衡。
- Bloomberg / FT / Economist 内参式阅读：标题、导语、正文、侧栏有清楚层级。
- `awesome-design-md` 中的真实 `DESIGN.md`：用于补充具体页面结构和组件规则。

不允许只说“高级”“苹果风”，必须说明本次参考的是哪一种页面逻辑。

## 色彩

主色：

- 象牙白 / 米白：页面底色。
- 深石墨黑：标题和重点判断。
- 深墨绿：品牌底色、重点模块、雷达感。
- 低饱和青绿色：链接、趋势、轻提示。
- 铜棕色：少量 eyebrow 和强调。

禁止：

- 大面积紫色渐变。
- 霓虹科技蓝。
- 高饱和互联网蓝。
- 大面积纯黑压迫。
- 花哨装饰渐变。

## 字体与字号

原则：

- 大标题只用于真正的页面主判断。
- 卡片内标题必须克制，不得像首页 hero。
- 正文行高要舒适，中文建议 `1.6-1.85`。
- 字体不能靠无限放大制造高级感。

建议层级：

- 首页 hero：`clamp(48px, 6vw, 96px)`
- 栏目页标题：`clamp(42px, 5vw, 72px)`
- 详情页主判断：`clamp(30px, 3vw, 48px)`
- 卡片标题：`18-24px`
- 正文：`15-17px`
- 辅助信息：`12-14px`

禁止：

- 同一屏出现多个超大黑体标题。
- 小卡片标题超过 `28px`。
- 文本容器过窄导致中文频繁断行。
- 文本行宽过宽导致阅读疲劳。

## 布局原则

页面要先确定“阅读路径”，再设计卡片。

优先结构：

1. 页面判断 / 导语
2. 主阅读区
3. 侧栏或补充证据
4. 关联内容

避免：

- 卡片套卡片。
- 三栏内容完全等权但信息量不同。
- 太多边框、阴影、徽章同时出现。
- 每个元素都像按钮。
- 公开页面出现后台编辑、恢复数据、JSON 等痕迹。

## 栏目标题规范

一级栏目页标题必须保持统一的视觉锚点，避免每个栏目看起来像不同产品。

硬规则：

- Daily Brief、Signals、Opportunities、Trends 等一级栏目标题区必须使用同一套位置、字号、行高、留白和内容宽度节奏。
- 标题必须是一句话价值表达，优先一行显示；不在标题下追加解释型副标题、功能说明或内部使用提示。
- 标题背景默认与页面背景一致，不额外添加突兀色块、装饰块或孤立渐变。
- 新增或修改任一栏目标题后，必须对照至少两个既有栏目截图，检查标题左边距、顶部位置、字号、行高和首屏节奏是否一致。
- 标题不能为了“有设计感”而牺牲清晰度；如果文案过长，应先让 Copy Agent 压缩文案，而不是放大容器。

## 成品感与精致度

观澜AI不是临时数据看板，公开前台不能粗糙、简陋或像组件堆叠。

硬规则：

- 首屏必须有明确主次：主标题、核心信息、主要动作不能互相抢权。
- 页面不能用大面积占位、空洞指标或内部解释制造“功能区”；每一块都必须服务用户阅读和判断。
- 信息密度要像商业内参：克制、可扫读、有层级；不能像新闻瀑布流，也不能像后台字段表。
- 卡片、徽章、边框、阴影和分割线只作为层级辅助，不作为装饰堆砌。
- 详情页应像研究报告或商业笔记，优先用正文节奏、分隔线、侧栏摘要和少量链接组织信息，不把每个字段都包成大卡片。
- 桌面端和移动端都要检查：是否主次清楚、是否有过大留白、是否文字过长、是否横向溢出、是否按钮或徽章压过正文。
- 如果用户反馈“粗糙、简陋、主次不明、占位过大、文案像内部话术”，默认视为结构问题，先调整信息架构和阅读路径，再处理视觉细节。

## Daily Brief 规范

Daily Brief 是“每日商业内参”，不是数据列表。

列表页：

- 每天一张主卡。
- 卡片可以点击进入详情页。
- 卡片只展示三类内容：
  - 关键 Signals
  - 机会卡
  - 趋势观察
- 短期不展示机会优先级，因为评分需要长期积累才有说服力。
- 页面左右留白不能过大，桌面端主内容宽度建议接近 `1520-1680px`。

详情页：

- 用“主 Signal 阅读区 + 右侧机会/趋势侧栏”。
- 首条 Signal 要明显更强，像今日主线。
- 右侧侧栏只做摘要，不抢主阅读区。
- 文案必须克制，强调“证据关系”和“观察”，不替用户下最终判断。

禁止：

- 三栏平铺且每栏等权。
- 一屏内堆太多小盒子。
- 详情页标题只有日期，没有判断含义。
- 侧栏文字过长。

## 首页 Decision Brief 规范

首页 Decision Brief 是给目标客户的第一层判断入口。

第一栏必须优先展示 Signals，而不是短期评分。

推荐三栏：

- 今日关键 Signals
- 正在升温的趋势
- 值得继续观察的机会

每条内容必须有：

- 标题
- 一句解释
- 必要时有 Signal Score / 来源层级 / 赛道

禁止：

- 首页显示过多后台指标。
- 用“优先级评分”作为首页第一判断依据。
- 文案像内部工作流说明。

## Opportunities 规范

Opportunities 是机会库，不是公司库。

机会卡名称不能以公司为核心命名。公司只能作为证据、案例或来源出现。

机会卡应展示：

- 机会名称
- 一句话机会定义
- 适用行业 / 场景
- 证据等级
- 相关 Signals
- 相关 Trends

Priority Engine 可以出现在 Opportunities 中，但必须作为排序和复盘辅助，不作为前台独立主栏目。

## Signals 规范

Signal 标题必须是“事件 + 商业含义”，不能只是公司名。

Signal 卡片优先展示：

- 事件标题
- 内容简介
- Signal Score
- 来源层级
- 相关趋势 / 机会

Signals 栏目页补充规则：

- Signals 栏目页是商业信号筛选和进入详情的页面，不是完整报告页，也不是内部雷达说明页。
- 单条 Signal 的“指向机会”数量使用主 Opportunity 口径；前台不显示由弱标签、赛道或反向关系放大的机会数量。
- 列表行要让标题和事实摘要成为主信息，赛道、信源状态、机会数、趋势数和 CTA 只做辅助。
- 详情页承担完整来源、事实、商业含义、机会拆解和关联内容，栏目页不重复堆叠详情正文。

## 文案原则

文案要有观点，但不替客户下最终判断。

推荐语气：

- “值得继续观察”
- “证据正在增强”
- “需要保留反证”
- “可能形成连续证据”
- “当前更像早期信号”

避免：

- “一定会”
- “必须投”
- “立刻行动”
- “确定机会”
- “保证收益”

## UI Agent 工作流程

每次修改页面前，UI/UE Agent 必须输出：

1. 本次页面目标。
2. 使用的设计参考，例如 `frontend-design + Apple/Linear/内参阅读布局`。
3. 页面阅读路径。
4. 要删除或弱化的信息。
5. 桌面端和移动端验收点。

每次修改页面后，必须检查：

- 是否使用了本 `DESIGN.md`。
- 是否过度卡片化。
- 是否有过大留白或过度拥挤。
- 文字是否断行难看。
- 普通用户前台是否出现后台痕迹。
- 是否需要截图验收。
