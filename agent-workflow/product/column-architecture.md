# 栏目架构与边界

栏目按“用户要完成的判断任务”设计，而不是按“我们有什么数据”设计。

## 推荐前台导航

```text
首页 / Daily Brief / Signals / Opportunities / Trends
```

Admin 为后台入口，不进入普通用户导航。

Tags 不再建议长期作为前台一级导航。它更适合作为搜索、筛选和机会网络能力，在栏目内部使用。

Method 暂不作为对外栏目。方法论保留为内部产品资产，用于支撑首页、样例报告、销售沟通和未来信任内容，不进入当前前台导航。

## 栏目任务

| 栏目 | 用户任务 | 核心内容 | 不应承担 |
|---|---|---|---|
| 首页 | 判断这个平台是否值得注册/登录 | 品牌价值、核心卖点、代表判断、注册/登录入口 | 展示全部内容、后台说明 |
| Daily Brief | 每天先看结论 | 今日结论、关键 Signals、优先验证机会、趋势变化、行动建议、风险提醒 | 变成新闻列表 |
| Signals | 看发生了什么变化 | 事件、证据、商业含义、相关机会 | 只展示公司名 |
| Opportunities | 决定哪些机会值得优先验证 | 机会定义、优先级、证据等级、趋势状态、客户、场景、验证动作 | 用公司名当机会名 |
| Trends | 判断赛道势能 | 升温、分化、降温、风险区、证据阶梯、反证观察点 | 只展示趋势名称 |

## Scoring / Priorities / Opportunities 的关系

结论：对外不再保留独立 `Scoring` 或 `Priorities` 一级栏目，评分和优先级合并进入 `Opportunities`。

原因：

- Scoring 是内部方法，用户关心的是哪些机会值得优先验证。
- 评分对象应是机会方向、赛道或应用场景，不是公司榜单。
- 没有评分或优先级的内容，不应该作为正式机会展示。
- 评分数据应该服务每张机会卡，而不是孤立展示为排行榜。

产品结构：

- `02-Scoring/`：保留为本地数据源和判断引擎。
- `Priority Engine`：后台评分与优先级引擎。
- `Opportunities`：前台最终机会库，承接评分、证据等级、趋势状态和行动建议。

## 合并与降级规则

当栏目或功能产生重叠时，按以下规则处理：

1. 如果只是同一数据的不同排序方式，优先做成筛选或视图，不新增栏目。
2. 如果用户任务不同，保留独立栏目。
3. 如果只服务后台生产，不进入前台导航。
4. 如果对外解释不清，先降级为后台能力。
5. 如果无法关联到 Signals、Priority Engine、Trends、Opportunities 之一，暂不进入正式产品展示。
