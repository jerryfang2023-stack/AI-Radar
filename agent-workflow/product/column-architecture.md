# 栏目架构与边界

栏目按“用户要完成的判断任务”设计，而不是按“我们有什么数据”设计。

## V2 前台导航

```text
今日要点 / 关键信号 / 机会解码 / 商业内参
```

Admin 为后台入口，不进入普通用户导航。

Tags 不作为前台一级导航。它是搜索、筛选、关键词表和关系网络能力，在栏目内部使用。

The Point 降级为观点校准模块。Trends 降级为趋势背景和热力输入。Scoring 后台化为 Priority Engine。

Method 暂不作为对外栏目。方法论保留为内部产品资产，用于支撑首页、样例报告、销售沟通和未来信任内容，不进入当前前台导航。

## 栏目任务

| 栏目 | 用户任务 | 核心内容 | 不应承担 |
|---|---|---|---|
| 今日要点 | 用 5-8 分钟理解今天 AI 商业变化 | 今日判断、发生了什么、值得关注、趋势温度、机会观察、观点与思考、风险与边界、后续观察、关键词表、来源状态 | 变成新闻列表或内部流程看板 |
| 关键信号 | 精读少数真正值得跟踪的商业信号 | Front Signal 深度分析报告、高质量 Structured Signal、来源、图表、参数、关联趋势和机会 | 只展示公司名或短卡片 |
| 机会解码 | 阅读能辅助老板判断的商业机会深度报告 | Deep Dive、机会报告库、机会成熟度、机会地图、来源证据、风险边界 | 用公司名当机会名，或把机会写成短卡片列表 |
| 商业内参 | 阅读跨周期、跨栏目、可归档的组合判断报告 | AIBrief Issue、专题、周/月复盘、趋势与机会地图、观点分歧、来源证据 | 写成泛泛观点合集，或把单个机会 Deep Dive 直接当作商业内参 |

## 今日要点标准

今日要点不是普通日报，而是 AI 商业判断 Newsletter。它要帮商业决策者快速理解今天发生了什么、哪些值得关注、哪些只是讨论升温但证据不足。

固定内容构成：

1. `日期导航`：上一日、下一日、周视图、月视图；标记完整日、降级日、证据不足日和无深挖日。
2. `今日判断`：用 2-4 段给出当天主线，2500-4500 中文字总长。
3. `发生了什么`：压缩 5-8 个事实，不写内部 Raw / Pool 名称。
4. `值得关注`：列出当天最值得继续看的 3-5 个信号或主题。
5. `趋势温度`：展示热度上升、证据不足、继续观察；高热但未成信号的内容只放这里。
6. `机会观察`：连接机会解码，不写“立即行动”。
7. `观点与思考`：当天 Builder / VC / Founder / Research 的有价值观点都可以出现；能修正主判断时说明修正关系，不能修正时作为独立思考模块。
8. `风险与边界`：说明证据缺口、来源偏差、商业变量未验证处。
9. `后续观察`：7 / 30 / 90 天继续看什么。
10. `关键词表`：公司、产品、人物、赛道、场景、风险、来源类型和正式标签。
11. `搜索与筛选`：支持按日期、关键词、正式标签、来源等级、栏目类型检索。
12. `相关内容索引`：回链 Front Signal、Structured Signal、Trend、Opportunity、Point。
13. `来源与证据状态`：列出来源名、来源等级、原始外链和每个来源提供的增量事实。

趋势图建议：

- 7 / 30 / 90 天热度折线。
- 来源类型堆叠条。
- 赛道 x 场景热力图。
- 风险温度条。
- 信号簇关系图。

表达边界：

- 不对外展示 Raw、Pool、Heat Candidate、M/C 等内部词。
- 高热未成信号的内容表达为“热度上升 / 证据不足 / 继续观察”。
- M/C 线索只能表达为“讨论升温”或“线索来源”，不能写成事实结论。

## 关键信号内容层级

Structured Signal 与 Front Signal 不是同一种文章。

| 类型 | 角色 | 字数 | 内容要求 | 进入知识库 |
|---|---|---:|---|---|
| Structured Signal | 结构化信号卡 | 1200-2000 中文字 | 快速理解、入库、关联；必须有事件、商业变量、来源等级、证据边界、趋势候选 | 高质量且可复用者进入 `knowledge/01-Signals/` |
| Front Signal | 深度分析报告 | 3000-5000 中文字 | 值得精读、可分享、可进入商业内参；必须有数据、分析、图片或图表、关键参数、至少 3 个 S/A/B 原始来源 | 优先进入 `knowledge/01-Signals/`，并回链 Trend / Opportunity / Case |

所有引用、事实、数据、融资、客户案例、产品发布、监管、观点和参数必须附来源名、来源等级、原始外链和增量事实说明。

## Builders 观点入口

Builders 观点作为「关键信号」下的子入口，不作为独立一级栏目。后台仍可沿用 Point 数据模型，但前台表达应是“Builders 观点 / 观点与思考”。

栏目层级：

```text
关键信号
├─ Front Signals｜深度信号报告
├─ Structured Signals｜结构化信号卡
└─ Builders 观点｜建造者观点与判断变化
```

Builders 观点定位：

- 不是新闻源。
- 不是事实主证据。
- 是 Builder / Founder / VC / Research / Operator 的判断输入。
- 用于观察观点延续、修正、转向、冲突，以及这些变化如何影响 Signal / Trend / Opportunity 判断。

关键信号栏目页中的 Builders 模块应展示：

- 人物头像 / 姓名 / title。
- 最新观点摘要。
- 发言时间。
- 来源平台：X / Blog / Podcast / YouTube / Interview / Research。
- 关联 Signal / Trend / Opportunity。
- 观点状态：新观点、延续观点、修正观点、转向观点、与主判断冲突。
- 入口文案：查看观点变化。

Builder 详情页以人物、title 和发言时间线为基线。固定要素：

1. `人物信息`：姓名、title、公司 / 机构、角色类型、关注领域、相关标签。
2. `当前观点摘要`：近 7 / 30 / 90 天核心判断。
3. `发言时间线`：日期、当时 title、来源平台、原始外链、原文摘录或摘要、中文转述、观澜解读、关联内容、观点状态。
4. `观点变化`：过去怎么看、最近是否改变判断、变化发生在哪个变量上。
5. `关联内容`：Front Signal、Structured Signal、Trend、Opportunity、Deep Dive。
6. `来源与边界`：观点只能作为判断校准，不替代公司公告、客户证据、财报、监管文件等事实来源。

## 机会解码标准

机会解码不是机会卡列表，也不是公司案例库。它是面向老板的 AI 商业机会深度分析栏目，用于判断一条 AI 商业变化是否可能变成一门生意、谁会付钱、证据是否足够、风险在哪里。

边界：

- 不出现“中国迁移难点”作为固定栏目或硬性模块。
- 公司不能作为机会标题，只能作为代表案例或证据。
- 不写投资建议，不写“马上行动”。
- 证据不足时不硬凑 Deep Dive，明确写“今日暂无足够证据支撑深挖内参”。

机会解码栏目页固定要素：

1. `顶部判断区`：栏目名、简短副标题、日期切换、搜索和筛选。
2. `今日重点机会报告`：展示 1-2 篇最重要 Deep Dive；证据不足时允许为空。
3. `机会报告库`：展示历史 Deep Dive 和高质量机会报告，卡片只作为报告入口。
4. `机会成熟度分层`：早期观察、证据增强、值得重点跟踪、可小范围验证、风险升高、暂缓判断。
5. `机会地图`：客户 x 流程矩阵、趋势温度 x 证据等级矩阵、商业模式分布、风险类型分布、机会成熟度阶梯。
6. `关联信号区`：Front Signal、Structured Signal、Builders 观点、Trend、来源证据。
7. `来源与证据状态`：来源名、来源等级、原始外链、增量事实和证据缺口。

Deep Dive 详情页是 6000-10000 中文字的商业内参级长文。固定结构：

1. `标题区`：机会名称、一句话判断、日期、阅读时间、判断等级、证据等级、来源数量、S 级 / 一手来源数量、关联趋势、关联信号。
2. `老板先看`：3-5 条核心结论，直接回答这是不是值得继续看的机会。
3. `机会判断`：这个机会是什么、为什么现在出现、当前处于什么成熟度。
4. `为什么现在值得看`：技术、客户行为、成本、渠道、竞争、监管或合规变化。
5. `目标客户与付费方`：行业、角色、预算方、决策方、使用者、购买触发点。
6. `真实问题`：当前流程、痛点、成本、时间、质量、合规或收入问题。
7. `AI 替代或增强的流程`：输入、输出、改变的流程和商业变量。
8. `商业模式`：subscription、usage-based、outcome-based、service fee、enterprise license、marketplace / take rate。
9. `代表案例与证据`：公司只出现在这里，每个案例必须有来源名、来源等级、原始外链和增量事实。
10. `关联信号与趋势`：Front Signal、Structured Signal、Trend、Builders 观点、历史同类信号。
11. `风险与反证`：付费意愿、数据可得性、集成成本、交付成本、平台依赖、合规风险、竞争挤压。
12. `判断结论`：值得重点跟踪、可小范围验证、先观察证据、风险升高暂缓判断。
13. `后续观察`：7 / 30 / 90 天分别看什么证据。

## 商业内参标准

商业内参是高阶组合判断报告，不是单个机会 Deep Dive，也不是普通专题文章。

定位：

- 围绕一个主题、赛道、趋势或周期，把多个 Signal、Opportunity、Trend、Builders 观点和风险证据组织成一份商业判断报告。
- 可归档、可复盘、可付费阅读。
- 适合周度、月度、季度或重点专题输出。

它回答的问题：

- 一个方向是否正在形成结构性变化。
- 哪些机会只是短期热度，哪些可能进入长期判断。
- 哪些公司、客户、场景、资本、技术、监管和观点变量正在一起变化。
- 老板应如何理解这个方向的中期机会和风险。

与机会解码的关系：

- 机会解码 Deep Dive 聚焦一个机会方向，6000-10000 中文字。
- 商业内参聚焦一个主题、赛道、趋势或周期，可以包含多个机会方向。
- 单篇 Deep Dive 可以被商业内参引用，但不能直接等同于商业内参。

建议篇幅：

- 常规内参：8000-15000 中文字。
- 深度专题：15000-30000 中文字。
- 周 / 月复盘：6000-12000 中文字。

商业内参栏目页固定要素：

1. `最新内参`：最新重点内参、标题、一句话判断、覆盖周期、关联信号数量、关联机会数量、来源数量、风险状态、阅读时间。
2. `内参库`：按主题、赛道、客户场景、周期、风险类型筛选。
3. `主题追踪`：使用正式标签体系组织企业 Agent、AI Coding、AI 销售、AI 合规、多模态内容、AI 基础设施等主题。
4. `周期复盘`：7 天、30 天、90 天、月度专题、季度判断。
5. `趋势与机会地图`：趋势阶段、机会分布、风险升温、证据强度、Builder 观点分歧。
6. `来源与证据状态`：S/A/B 来源数量、一手来源数量、观点来源数量、证据缺口。

商业内参详情页固定结构：

1. `标题区`：标题、一句话判断、覆盖周期、阅读时间、主题标签、证据等级、风险状态。
2. `执行摘要`：5-8 条核心结论。
3. `主题背景`：这条主线过去发生了什么，为什么现在值得重新看。
4. `信号链`：关键 Front Signals、Structured Signals、历史同类信号，区分主线和热度。
5. `机会组合`：关联多个机会解码，说明重点跟踪、早期观察和风险升高的方向。
6. `趋势判断`：rising / splitting / cooling / emerging / mature / risk，7 / 30 / 90 天变化。
7. `Builders 观点分歧`：支持、谨慎、转向、冲突的观点。
8. `风险与反证`：客户不买单、平台依赖、成本结构、监管 / 合规、竞争挤压、技术瓶颈。
9. `判断结论`：结构性机会形成、证据增强但未成型、热度高但商业证据不足、风险升高、暂缓判断。
10. `后续观察`：7 / 30 / 90 天。
11. `来源附录`：来源名、来源等级、原始外链、增量事实、支撑段落。

## Scoring / Priorities / Opportunities 的关系

结论：对外不再保留独立 `Scoring` 或 `Priorities` 一级栏目，评分和优先级合并进入 `Opportunities`。

原因：

- Scoring 是内部方法，用户关心的是哪些机会值得继续观察、哪些证据仍不足。
- 评分对象应是机会方向、赛道或应用场景，不是公司榜单。
- 没有评分或优先级的内容，不应该作为正式机会展示。
- 评分数据应该服务每张机会卡，而不是孤立展示为排行榜。

产品结构：

- `02-Scoring/`：保留为本地数据源和判断引擎。
- `Priority Engine`：后台评分与优先级引擎。
- `Opportunities / 机会解码`：前台机会判断报告层，承接评分、证据等级、趋势状态和风险边界。

## 合并与降级规则

当栏目或功能产生重叠时，按以下规则处理：

1. 如果只是同一数据的不同排序方式，优先做成筛选或视图，不新增栏目。
2. 如果用户任务不同，保留独立栏目。
3. 如果只服务后台生产，不进入前台导航。
4. 如果对外解释不清，先降级为后台能力。
5. 如果无法关联到关键信号、Priority Engine、趋势温度、机会解码或商业内参之一，暂不进入正式产品展示。
