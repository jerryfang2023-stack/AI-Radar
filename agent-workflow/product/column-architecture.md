# 栏目架构与边界

栏目按“用户要完成的判断任务”设计，而不是按“我们有什么数据”设计。

## V2 前台导航

```text
今日观察 / 商业信号 / 趋势追踪 / 商业内参
```

Admin 为后台入口，不进入普通用户导航。

Tags 不作为前台一级导航。它是搜索、筛选、关键词表和关系网络能力，在栏目内部使用。

The Point 降级为前沿观点模块。Trends 降级为趋势背景和热力输入。Scoring 后台化为 Priority Engine。

Method 暂不作为对外栏目。方法论保留为内部产品资产，用于支撑首页、样例报告、销售沟通和未来信任内容，不进入当前前台导航。

## 栏目任务

| 栏目 | 用户任务 | 核心内容 | 不应承担 |
|---|---|---|---|
| 今日观察 | 用一篇文章理解当天 AI 商业行情 | 当日市场综述、热点解释、精选变化卡、观点露出、来源状态 | 变成新闻列表或内部流程看板 |
| 商业信号 | 精读少数真正值得跟踪的商业信号 | 变化卡、案例卡、来源、图表、参数、关联趋势和机会判断 | 只展示公司名或短卡片 |
| 趋势追踪 | 阅读能辅助老板判断的趋势深度报告 | 趋势快报、深度报告、变化簇、趋势卡、机会判断、来源证据、风险边界 | 卡片聚合页、公司案例库或执行方案 |
| 商业内参 | 阅读跨周期、跨栏目、可归档的组合判断报告 | AIBrief Issue、专题、周/月复盘、趋势与判断地图、观点分歧、来源证据 | 写成泛泛观点合集，或把单篇趋势深度报告 直接当作商业内参 |

## 今日观察标准

今日观察不是普通日报，而是一篇当天 AI 商业行情综述。它要帮商业决策者快速理解今天发生了什么、为什么选择这些变化、哪些只是讨论升温但证据不足。

固定内容构成：

1. `日期导航`：上一日、下一日、周视图、月视图；标记完整日、降级日、证据不足日和无深挖日。
2. `今日判断`：用 2-4 段给出当天主线，2500-4500 中文字总长。
3. `发生了什么`：压缩 5-8 个事实，不写内部 Raw / Pool 名称。
4. `值得关注`：列出当天最值得继续看的 3-5 个信号或主题。
5. `趋势温度`：展示热度上升、证据不足、继续观察；高热但未成信号的内容只放这里。若触发急件候选，可回链趋势快报。
6. `机会判断`：连接趋势追踪，不写“立即行动”。
7. `观点与思考`：当天 Builder / VC / Founder / Research 的有价值观点都可以出现；能修正主判断时说明修正关系，不能修正时作为独立思考模块。
8. `风险与边界`：说明证据缺口、来源偏差、商业变量未验证处。
9. `后续观察`：7 / 30 / 90 天继续看什么。
10. `关键词表`：公司、产品、人物、赛道、场景、风险、来源类型和正式标签。
11. `搜索与筛选`：支持按日期、关键词、正式标签、来源等级、栏目类型检索。
12. `相关内容索引`：回链 精选变化卡、变化卡、Trend Card、Trend Report、Point。
13. `来源与证据状态`：列出来源名、来源等级、原始外链和每个来源提供的增量事实。

趋势图建议：

- 7 / 30 / 90 天热度折线。
- 来源类型堆叠条。
- 赛道 x 场景热力图。
- 风险温度条。
- 信号簇关系图。

表达边界：

- 不对外展示 Raw、Pool、Heat Candidate、M/C 等内部词。
- 高热未成信号的内容表达为“热度上升 / 证据不足 / 继续观察”。
- M/C 线索只能表达为“讨论升温”或“线索来源”，不能写成事实结论。

## 商业信号内容层级

商业信号不是卡片列表，而是“少数值得精读的商业变化库”。首页只展示少数商业信号入口；商业信号栏目页负责筛选、浏览、比较；信号详情页负责完整讲清楚事件、原因、价值、证据、风险、同类案例、关联趋势和观点参照。它承接今日观察中被选出的变化，也为趋势追踪和商业内参提供可复盘的判断资产。

商业信号在首页的第一屏结构采用“今日精选 + 信号库入口”：

1. 左侧展示 1 条今日最值得精读的商业信号。
2. 右侧展示 2-3 条次级信号。
3. 下方提供赛道、场景、证据、来源、时间等筛选入口。
4. 再往下进入完整信号库。

首页商业信号不做新闻流，不做大而全数据库，也不重复今日观察下的精选变化卡。

变化卡 与 精选变化卡 不是同一种文章。

| 类型 | 角色 | 字数 | 内容要求 | 进入知识库 |
|---|---|---:|---|---|
| 变化卡 | 结构化信号卡 | 1200-2000 中文字 | 快速理解、入库、关联；必须有事件、商业变量、来源等级、证据边界、趋势候选 | 高质量且可复用者进入 `knowledge/01-Signals/` |
| 精选变化卡 | 深度分析报告 | 3000-5000 中文字 | 值得精读、可分享、可进入商业内参；必须有数据、分析、图片或图表、关键参数、至少 3 个 S/A/B 原始来源 | 优先进入 `knowledge/01-Signals/`，并回链 Trend / Trend Report / Case |

所有引用、事实、数据、融资、客户案例、产品发布、监管、观点和参数必须附来源名、来源等级、原始外链和增量事实说明。

## 前沿观点入口

前沿观点作为「商业信号」下的子入口，不作为独立一级栏目。后台仍可沿用 Point 数据模型，但前台栏目名称统一叫「前沿观点」。

栏目层级：

```text
商业信号
├─ 精选变化卡｜深度信号报告
├─ 变化卡｜结构化信号卡
└─ 前沿观点｜关键人物观点与判断变化
```

前沿观点定位：

- 不是新闻源。
- 不是事实主证据。
- 是 Builder / Founder / VC / Research / Operator 的判断输入。
- 用于观察观点延续、修正、转向、冲突，以及这些变化如何影响 Signal / Trend / Trend Report 判断。

商业信号栏目页中的前沿观点模块应展示：

- 人物头像 / 姓名 / title。
- 最新观点摘要。
- 发言时间。
- 来源平台：X / Blog / Podcast / YouTube / Interview / Research。
- 关联 Signal / Trend Card / Trend Report。
- 观点状态：新观点、延续观点、修正观点、转向观点、与主判断冲突。
- 入口文案：查看观点变化。

前沿观点详情页以人物、title 和发言时间线为基线。固定要素：

1. `人物信息`：姓名、title、公司 / 机构、角色类型、关注领域、相关标签。
2. `当前观点摘要`：近 7 / 30 / 90 天核心判断。
3. `发言时间线`：日期、当时 title、来源平台、原始外链、原文摘录或摘要、中文转述、观澜解读、关联内容、观点状态。
4. `观点变化`：过去怎么看、最近是否改变判断、变化发生在哪个变量上。
5. `关联内容`：精选变化卡、变化卡、Trend Card、Trend Report、趋势深度报告。
6. `来源与边界`：观点只能作为判断参照，不替代公司公告、客户证据、财报、监管文件等事实来源。

## 趋势追踪标准

趋势追踪不是卡片聚合页，也不是公司案例库。它是面向老板的 AI 商业趋势深度分析栏目，用于判断一组 AI 商业变化是否正在变成连续方向、谁会受影响、谁可能付钱、证据是否足够、风险在哪里。

趋势追踪承接两类资产：

- `Trend Card`：趋势状态记录，负责记录升温、分化、降温、被验证、被反证等状态。
- `Trend Report`：趋势判断文章，负责形成可阅读、可复盘的商业判断。

Trend Report 有两种前台形态：

| 类型 | 前台名称 | 作用 | 默认权限 |
|---|---|---|---|
| `kind = flash` | 趋势快报 | 某个方向突然升温时，当天或次日给出临时深读 | 公开或登录可读 |
| `kind = full` | 深度报告 | 周任务或快报升级后输出完整研究报告 | 会员重点内容 |

趋势快报混在趋势追踪页中，不单独开一级页面。它用 `趋势快报 / 继续观察 / 已升级为深度报告 / 判断被修正` 等自然标签表达，不暴露后台字段。

边界：

- 不出现“中国迁移难点”作为固定栏目或硬性模块。
- 公司不能作为机会标题，只能作为代表案例或证据。
- 不写投资建议，不写“马上行动”。
- 证据不足时不硬凑深度报告，明确写“今日暂无足够证据支撑深挖内参”。
- 单条新闻不能触发趋势快报，至少要多条信号汇聚。
- 每篇正式深度报告必须包含反证。没有反证，不进入正式报告。

趋势追踪栏目页固定要素：

1. `顶部判断区`：栏目名、简短副标题、日期切换、搜索和筛选。
2. `最新深度报告`：展示 1-2 篇最重要 Trend Report；证据不足时允许为空。
3. `正在升温`：展示趋势快报、观察中的 Trend Card 和变化簇。
4. `已升级`：展示已升级为深度报告的趋势快报，引导用户阅读对应深度报告。
5. `报告库`：展示历史深度报告；历史快报第一版弱展示或不展示，卡片只作为文章入口。
6. `趋势成熟度分层`：早期观察、证据增强、值得重点跟踪、可小范围验证、风险升高、暂缓判断。
7. `趋势地图`：客户 x 流程矩阵、趋势温度 x 证据等级矩阵、商业模式分布、风险类型分布、趋势成熟度阶梯。
8. `关联信号区`：精选变化卡、变化卡、前沿观点、Trend Card、来源证据。
9. `来源与证据状态`：来源名、来源等级、原始外链、增量事实和证据缺口。

趋势快报详情页是 2000-3500 中文字的临时深读。固定要求：

1. 用一个具体人、具体场景或具体冲突开头。
2. 说明今天为什么突然升温。
3. 说明它可能改变谁的流程、预算、岗位或竞争关系。
4. 说明市面上谁在动，包括同类产品、竞品或案例。
5. 只解释技术路线的商业含义，不写技术教程。
6. 明确反证、信息缺口和 30 天观察变量。
7. 至少 3 个独立来源，其中至少 1 个 S/A 或一手来源。

趋势深度报告详情页是 6000-10000 中文字的商业内参级长文。固定结构：

1. `标题区`：趋势方向、一句话判断、日期、阅读时间、判断等级、证据等级、来源数量、S 级 / 一手来源数量、关联趋势、关联信号。
2. `老板先看`：3-5 条核心结论，直接回答这是不是值得继续看的方向。
3. `机会判断`：这个机会是什么、为什么现在出现、当前处于什么成熟度。
4. `为什么现在值得看`：技术、客户行为、成本、渠道、竞争、监管或合规变化。
5. `目标客户与付费方`：行业、角色、预算方、决策方、使用者、购买触发点。
6. `真实问题`：当前流程、痛点、成本、时间、质量、合规或收入问题。
7. `AI 替代或增强的流程`：输入、输出、改变的流程和商业变量。
8. `商业模式`：subscription、usage-based、outcome-based、service fee、enterprise license、marketplace / take rate。
9. `代表案例与证据`：公司只出现在这里，每个案例必须有来源名、来源等级、原始外链和增量事实。
10. `关联信号与趋势`：精选变化卡、变化卡、Trend、前沿观点、历史同类信号。
11. `风险与反证`：付费意愿、数据可得性、集成成本、交付成本、平台依赖、合规风险、竞争挤压。
12. `判断结论`：值得重点跟踪、可小范围验证、先观察证据、风险升高暂缓判断。
13. `后续观察`：7 / 30 / 90 天分别看什么证据。

急件触发机制：

```text
urgent_trend_candidate -> rapid_review -> temporary_trend_report_run
```

满足 3 个条件可进入急件候选，但必须同时包含多源信号和商业影响：

- 24-48 小时内出现 5 条以上相关 Raw，且至少 3 个不同来源类型。
- 出现 S/A 来源。
- 明确影响客户预算、业务流程、竞争格局、成本结构、收入方式或采购决策。
- builders、投资人、行业媒体、客户社区中至少两类人群开始讨论同一方向。

急件候选先由系统复核，再进入人工批量复核。人只做 `放行临时报告 / 继续观察 / 降级为趋势卡`，不逐张卡反复确认。

writer 每次运行最多产出 1 篇：

- weekly mode：最多 1 篇深度报告。
- urgent mode：最多 1 篇趋势快报。

同一天多个急件候选时，按 `多源密度 > 观澜重点赛道 > 商业影响 > 来源质量 > 时间新鲜度` 排序。未选中的候选进入当天 `Deferred Candidates`，最多延后 2 次，第三次必须落状态。

没有足够证据写报告时，writer 必须产出 `no_report_decision`，不硬写、不沉默。该记录不进入前台索引。

## 商业内参标准

商业内参是高阶组合判断报告，不是单篇趋势深度报告，也不是普通专题文章。

定位：

- 围绕一个主题、赛道、趋势或周期，把多个 Signal、Trend Report、Trend Card、前沿观点和风险证据组织成一份商业判断报告。
- 可归档、可复盘、可付费阅读。
- 适合周度、月度、季度或重点专题输出。

它回答的问题：

- 一个方向是否正在形成结构性变化。
- 哪些机会只是短期热度，哪些可能进入长期判断。
- 哪些公司、客户、场景、资本、技术、监管和观点变量正在一起变化。
- 老板应如何理解这个方向的中期机会和风险。

与趋势追踪的关系：

- 趋势追踪深度报告聚焦一个趋势方向，通常 6000-10000 中文字；趋势快报是临时深读，不等同于完整深度报告。
- 商业内参聚焦一个主题、赛道、趋势或周期，可以包含多个机会方向。
- 单篇趋势追踪可以被商业内参引用，但不能直接等同于商业内参。
- 商业内参可以反向修正趋势快报或正式报告的状态，包括已升级、归档观察或判断被修正。

建议篇幅：

- 常规内参：8000-15000 中文字。
- 深度专题：15000-30000 中文字。
- 周 / 月复盘：6000-12000 中文字。

商业内参栏目页固定要素：

1. `最新内参`：最新重点内参、标题、一句话判断、覆盖周期、关联信号数量、关联趋势报告数量、来源数量、风险状态、阅读时间。
2. `内参库`：按主题、赛道、客户场景、周期、风险类型筛选。
3. `主题追踪`：使用正式标签体系组织企业 Agent、AI Coding、AI 销售、AI 合规、多模态内容、AI 基础设施等主题。
4. `周期复盘`：7 天、30 天、90 天、月度专题、季度判断。
5. `趋势与判断地图`：趋势阶段、趋势报告分布、风险升温、证据强度、Builder 观点分歧。
6. `来源与证据状态`：S/A/B 来源数量、一手来源数量、观点来源数量、证据缺口。

商业内参详情页固定结构：

1. `标题区`：标题、一句话判断、覆盖周期、阅读时间、主题标签、证据等级、风险状态。
2. `执行摘要`：5-8 条核心结论。
3. `主题背景`：这条主线过去发生了什么，为什么现在值得重新看。
4. `信号链`：关键 精选变化卡、变化卡、历史同类信号，区分主线和热度。
5. `机会组合`：关联多个趋势追踪中的机会判断，说明重点跟踪、早期观察和风险升高的方向。
6. `趋势判断`：rising / splitting / cooling / emerging / mature / risk，7 / 30 / 90 天变化。
7. `前沿观点分歧`：支持、谨慎、转向、冲突的观点。
8. `风险与反证`：客户不买单、平台依赖、成本结构、监管 / 合规、竞争挤压、技术瓶颈。
9. `判断结论`：结构性机会形成、证据增强但未成型、热度高但商业证据不足、风险升高、暂缓判断。
10. `后续观察`：7 / 30 / 90 天。
11. `来源附录`：来源名、来源等级、原始外链、增量事实、支撑段落。

## Scoring / Priorities / Trend Report 的关系

结论：对外不再保留独立 `Scoring`、`Priorities` 或旧英文机会字段对应的一级栏目，评分和优先级进入 Trend Report 的证据状态、观察优先级和风险边界。

原因：

- Scoring 是内部方法，用户关心的是哪些机会值得继续观察、哪些证据仍不足。
- 评分对象应是机会方向、赛道或应用场景，不是公司榜单。
- 没有评分、来源或反证的内容，不应该作为正式深度报告展示。
- 评分数据应该服务 Trend Report 和 Trend Card，而不是孤立展示为排行榜。

产品结构：

- `02-Scoring/`：保留为本地数据源和判断引擎。
- `Priority Engine`：后台评分与优先级引擎。
- `Trend Reports / 趋势追踪`：前台趋势判断报告层，承接变化簇、Trend Card、机会判断、证据等级和风险边界。

## 合并与降级规则

当栏目或功能产生重叠时，按以下规则处理：

1. 如果只是同一数据的不同排序方式，优先做成筛选或视图，不新增栏目。
2. 如果用户任务不同，保留独立栏目。
3. 如果只服务后台生产，不进入前台导航。
4. 如果对外解释不清，先降级为后台能力。
5. 如果无法关联到商业信号、Priority Engine、趋势追踪、机会判断或商业内参之一，暂不进入正式产品展示。
