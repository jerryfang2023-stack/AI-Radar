# Bad Monthly Report Page

Avoid these patterns:

- turning the monthly report into a brief overview card;
- placing raw counts, data boundary, route fields, or method notes in first-screen stat cards;
- using raw Markdown tables with dense borders;
- treating monthly report pages like weekly report pages without adapting the structure;
- hiding the full body behind only charts or summaries;
- adding redundant return buttons or report-center buttons after the user removed them;
- leaving mobile tables horizontally scrollable without a clear reason;
- updating project skill files without syncing `.skill-store` and rebuilding the registry.

Bad user-visible result:

> The page has a nice hero but only a short summary, then a few raw tables and backend method notes.

Why it fails:

The monthly page must be a full report-reading surface. It should preserve the monthly structural argument while improving readability.
