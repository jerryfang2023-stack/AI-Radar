# PM Module Governance Upgrade

## 1. 背景

用户明确要求 PM Agent 在构建和治理产品功能模块时秉持“宁缺毋滥”原则：不因想法存在就新增模块，也不因模块已经存在就默认保留，而要先判断商业价值、模块重复、用户接受度、体验顺畅度和战略贴合度。

本规则用于约束后续栏目、页面、功能入口、后台模块、关系视图、会员能力和内容形态的新增、保留、强化、合并、后台化、隐藏和淘汰。

## 2. 已补充能力

PM Agent 已新增“宁缺毋滥模块生命周期治理”：

1. 商业价值：模块是否提升判断质量、付费理由、复访动力、客户决策效率、销售线索或品牌可信度。
2. 与现有模块关系：是否与 Daily Brief、Signals、The Point、Opportunities、Trends、Priority Engine、Tags 或 Admin 重复。
3. 目标用户与付费意愿：商业决策者是否真的会用，是否愿意支付时间、注意力、团队协作成本或订阅费用。
4. 使用体验与逻辑顺畅：用户路径是否更清楚，信息进入、阅读、筛选、判断、回访和下一步动作是否顺畅。
5. 战略核心贴合：是否紧扣“从 AI 热点中筛出商业信号，从信号形成判断，从判断发现机会”。

## 2A. 外部参考与观澜AI适配

本轮联网参考了以下产品方法，但只吸收方法骨架，不照搬通用 SaaS 模板：

| 来源 | 可吸收方法 | 观澜AI适配 |
|---|---|---|
| Product Talk / Teresa Torres：Opportunity Solution Tree | 从业务结果出发，区分机会、方案和假设测试 | 新功能不能从“想做某个页面”开始，必须先说明它服务哪个观澜业务结果和用户判断任务 |
| SVPG / Marty Cagan：Four Big Risks | PM 负责价值和商业可行性，产品必须同时服务客户价值和业务价值 | PM 不只写功能清单，必须判断该功能是否增强付费、留存、销售可信度或商业化路径 |

## 2B. 新增功能门禁

PM Agent 已新增 WAVE 功能门禁。任何新增功能、页面、入口、视图、筛选、后台能力、会员能力、自动化产物或数据维度，必须先过门禁，未通过不得写 PRD，不得进入 Dev。

WAVE 评分：

| 维度 | 说明 | 通过线 |
|---|---|---:|
| W - Worth paying for | 是否增强付费理由、复访动力、销售可信度或定制服务线索 | >= 2 |
| A - Alignment | 是否紧扣“信号 -> 判断 -> 机会 -> 趋势/验证”的战略链路 | >= 2 |
| V - Validation | 是否来自用户场景、销售反馈、内容质量问题、数据证据或明确反复痛点 | >= 1 |
| E - Experience / Effort | 用户路径是否顺畅，且实现、维护、运营成本可控 | >= 2 |

任一项为 0，直接阻塞。

## 2C. 观澜AI需要的功能

观澜AI需要的功能必须至少属于以下一类：

1. 提升判断质量：让 Daily Brief、Signals、Opportunities 或 Trends 更有观点、证据、优先级、反证和验证动作。
2. 提升判断效率：让商业决策者更快完成“今天看什么、为什么值得看、下一步验证什么、方向是否持续成立”。
3. 提升证据可信度：让事件、来源、客户采用、融资、收入、招投标、监管、产品落地和反证更可追溯。
4. 提升机会验证能力：帮助用户比较机会优先级、证据等级、趋势状态、目标客户、适用场景和下一步验证路径。
5. 提升复访与付费理由：让用户有理由每天回来、愿意试读、愿意订阅，或愿意进入行业简报、闭门简报、资源连接服务。
6. 提升运营质量但保持后台化：去重、合并、标签治理、权限、发布状态、数据健康、错误日志等能力只能服务生产质量，不污染普通前台。
7. 提升商业化转化但不夸大承诺：注册、登录、试读、会员状态、样例报告、订阅入口、企业咨询入口可以做，但必须克制表达。

默认拒绝：

- AI 新闻流、热点瀑布流、工具导航、公司排行榜、项目黄页、开发者社区。
- 泛工作台、泛仪表盘、泛图表墙、泛 AI 助手。
- 只增加信息量、不提高判断质量的功能。
- 只服务内部生产却暴露到普通前台的功能。
- 需要大量人工维护但不能形成付费理由的功能。
- 解释成本高、用户任务不清、需要文案不断解释才懂的入口。

## 3. 新增 PM 输出

PM Agent 在涉及新增、调整或复盘功能模块时，必须输出“模块决策表”，包含：

- 模块名称
- 用户是谁
- 用户为什么需要
- 商业价值
- 与现有模块是否重复
- 是否可通过加强现有模块解决
- 用户是否愿意付费或持续使用
- 使用路径是否顺畅
- 是否贴合观澜战略核心
- 决策：新增 / 保留 / 强化 / 合并 / 优化已有 / 后台化 / 隐藏 / 延期复核 / 淘汰 / 不做
- 非目标
- 需要交接的 Agent

## 4. 调度含义

后续调度窗口派发产品功能类任务时，不能只写“PM 输出 PRD”。派发单必须要求 PM 先输出模块决策表。

如果模块决策表没有证明新增必要性，任务不得进入 Dev。

如果既有模块不再适合目标用户、商业价值有限、路径不顺或偏离战略核心，PM 必须提出优化、合并、后台化、隐藏、延期复核或淘汰方案。

如果需求与现有模块交叉，PM 必须优先提出加强、合并或淘汰现有模块的方案。

## 5. 需要写入长期记忆的规则

已写入 `agent-workflow/governance/agent-memory.md`：

- PM Agent 默认不新增模块，也不默认保留既有模块。
- 新增、保留或强化前必须检查商业价值、重复、用户接受与付费意愿、体验路径、战略贴合。
- 新模块任一项不通过，默认不新增；既有模块任一项长期不通过，优先优化、合并、后台化、隐藏、延期复核或淘汰。
- 用户反馈“功能太散、入口太多、解释成本高、像内部系统、不是商业决策者会用的东西”时，应判定为 PM 模块治理失效。
