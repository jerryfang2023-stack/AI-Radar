# WSD-20260522 趋势追踪页面规划与重设计规格 Closeout

## 0. 执行结论

本任务属于 `页面 / 文案 / Typography Harness`，只产出趋势追踪栏目规划与页面重设计规格，不改代码、不改 `site-content.json`、不生成页面、不进入 Build & Release。

结论：

- 趋势追踪应采用 `信号证据累积 / 趋势雷达型`，辅以少量成熟趋势报告，不采用旧版报告频道、资讯流、趋势榜单、标签墙或普通 SaaS dashboard。
- 栏目核心用户任务不是“看最新 AI 趋势”，而是判断哪些方向已经有连续证据，哪些仍只是观察方向。
- 当前数据中只有 1 条趋势报告样数据，且状态为“继续观察”；4 条趋势候选都缺正式成立证据。后续前台必须把 `观察方向`、`趋势候选`、`趋势快报`、`深度趋势报告` 分层展示。
- 今日观察保留每日主判断和叙事价值，商业信号保留单条事实与事实档案，趋势追踪负责跨时间窗口的证据累积、反证、观察窗口和正式报告，商业内参负责周期性融合判断。
- 阶段 2B 可以在用户确认本规格后启动；启动后只能按本文的信息架构、字段表、Copy-first 文案表、Typography 页面位置表和验收清单实现，不得回到旧版 `trend-tracking.html` / `trend-detail.html` 局部修补。

## 3. 设计方案选择说明

| 方案 | 适合点 | 不适合点 | 是否采用 |
|---|---|---|---|
| 趋势观察看板型 | 适合展示观察方向、证据状态、时间窗口和过滤检索；能让用户看见哪些方向正在形成 | 如果做成后台 dashboard，筛选器和数字会抢过判断 | 部分采用，用作栏目页工具层 |
| 成熟趋势报告型 | 适合少量高证据密度内容，能体现商业内参式阅读感 | 当前正式报告数量不足，强行做报告频道会把候选包装成趋势 | 作为成熟内容详情模板采用 |
| 信号证据累积 / 趋势雷达型 | 适合观澜 AI 的判断逻辑：先看多条信号是否持续指向同一客户、流程、预算或组织变化，再判断是否升级 | 需要清晰标注证据缺口，否则容易像趋势榜单 | 采用为主方案 |
| 内参式趋势档案型 | 适合趋势详情页，把支撑信号、观点参照、反证和后续变量组织成阅读路径 | 不适合栏目页首屏，会显得过重、低频且不便检索 | 详情页采用 |

最终方案：`趋势雷达 + 成熟报告`。

页面气质：不是趋势榜，不是报告频道，也不是 AI 新闻归档。栏目页像一个克制的趋势雷达，先告诉用户哪些方向还在观察、哪些证据正在增强、哪些已经可以进入报告；详情页像一份内参式趋势档案，围绕支撑信号、场景、观点参照、反证、时间窗口和后续变量组织。

不采用旧 `trend-tracking.html` / `trend-detail.html` 的报告封面、旧报告频道逻辑和“最新报告 + 观察清单 + 框架图”固定结构。后续 Build 应重做信息架构和组件，不在旧版面上修补。

## 4. 趋势追踪栏目边界表

| 内容类型 | 今日观察 | 商业信号 | 趋势追踪 | 商业内参 |
|---|---|---|---|---|
| 单条公司动作 | 只有在当天形成叙事主判断时引用 | 主展示对象，作为事实档案 | 只能作为支撑信号，不单独升级成趋势 | 可作为周期判断证据 |
| 单条融资 / 产品发布 | 可作为当天观察材料 | 主展示对象 | 作为观察方向的证据之一 | 可进入资金流向或市场热度判断 |
| 多条同类信号 | 可解释当天判断背景 | 仍按事实台账展示 | 核心展示对象，用来判断证据是否增强 | 用于周期性判断和热力变化 |
| 已评级观点卡 | 观点参照，校准当天判断 | 事实详情的观点参照 | 趋势观察的观点参照，不能替代事实证据 | 可作为内参判断修正材料 |
| 观察方向 | 不作为主内容，只可在趋势背景中提示 | 不展示为信号结论 | 主展示层，必须标注证据不足和观察窗口 | 可作为内参中的未定方向 |
| 趋势候选 | 不包装成今日主题 | 不作为单条信号 | 主展示层，强调还缺什么 | 可作为下期内参跟踪对象 |
| 趋势快报 | 可被今日观察引用 | 可链接相关事实 | 正式展示层，短报告形式 | 可进入周期判断 |
| 深度趋势报告 | 不在今日观察展开 | 只作为关联趋势入口 | 正式展示层，完整报告详情 | 可成为内参核心证据 |
| 机会判断段落 | 只作当天观察中的克制判断 | 不作为商业信号字段直出 | 进入趋势报告正文段落，不单独成栏目 | 内参中可进一步复盘 |
| Tags 聚合 | 工具，不做主视觉 | 筛选和搜索 | 筛选、时间窗口和同类趋势聚合 | 周期性主题索引 |

## 6. 趋势内容分层与呈现表

| 层级 | 进入条件 | 前台呈现 | 不得呈现 |
|---|---|---|---|
| 观察方向 | 至少有 1-2 条相关信号或观点参照，但缺多源证据、客户采用或同类案例 | 趋势观察卡，标注“观察中方向”、时间窗口、还缺什么 | 不得写成趋势、报告、机会判断或榜单排名 |
| 趋势候选 | 多条信号、场景或观点参照指向同一客户、流程、预算或组织变化，但反证和边界未补齐 | 候选卡 + 支撑证据折叠区 + 反证 / 缺口卡 | 不得用确定性标题，不得隐藏证据缺口 |
| 趋势快报 | 多个独立事实来源支持，边界清楚，可形成短报告，但尚不适合长篇深挖 | 短报告卡和短详情页，包含一句判断、支撑信号、观察窗口、限制 | 不得输出落地方案、执行建议或保证收益 |
| 深度趋势报告 | 多源证据、同类案例、时间延续和反证处理足够，能形成完整报告 | 正式趋势报告卡和完整详情页 | 不得用单家公司名做标题，不得把观点当事实主证据 |
| 机会判断段落 | 来自趋势快报、深度趋势报告或商业内参中的判断段落 | 在报告正文中作为“机会判断”或“建议关注变量”出现 | 不作为单独栏目、卡片流或一级导航 |

当前数据落位：

- `contentIndex.trends` 的 4 条内容全部应落入 `观察方向` 或 `趋势候选`。
- `TRD-WATCH-20260522` 虽在 `trendReports` 中，但 `stage` 和 `evidenceGaps` 明确证据不足，应以前台“观察中方向 / 趋势候选详情”呈现，不得称为正式深度趋势报告。

## 9. 日期 / 时间窗口与 tags 检索规格

## 10. 关系模型与关联展示

| 关系 | 起点 | 终点 | 前台展示方式 | 数据字段来源 | 不得展示 |
|---|---|---|---|---|---|
| 趋势观察 -> 支撑商业信号 | `trend.id` / tags / date | `contentIndex.signals` | 详情页“支撑信号”，栏目页卡片摘要 | `relations` 优先，fallback 到 tags / date / signalType | `sourcePath`、Raw / Pool、内部门禁 |
| 趋势观察 -> 案例 / 融资 / 产品服务卡 | `trend` | `signals.signalType` 分类对象 | 支撑证据区按类型展示 | `signalType / frontend / tags` | 空案例库伪装成案例 |
| 趋势观察 -> 观点卡 | `trend` | `contentIndex.points` | “观点参照”模块 | `relatedTrends / relatedTrendReports / tags`，并过滤评级 | `opinionTier` 原字段、评分、把观点当事实 |
| 趋势观察 -> 今日观察文章 | `trend` | `daily` 或日期文章 | 侧栏“相关今日观察” | date、tags、后续 relations | 内部同步语言 |
| 趋势观察 -> 商业内参 | `trend` | `brief` | 底部“进入周期判断” | brief.evidence、tags、后续 relations | 会员内部字段 |
| 趋势报告 -> 变化短专题 / 成熟趋势候选 | `trendReport` | change topic / trend candidate | 报告正文中的支撑背景和底部关联 | `sections / relations / tags` | candidate / draft 原状态 |
| tags -> 同类趋势方向聚合 | tag id | trends / trendReports / signals / points | 筛选结果、同类方向入口 | `tagTaxonomy`、对象 `tags` | Tags 一级栏目、标签墙主视觉 |
| 时间窗口 -> 证据增强 / 减弱 / 争议状态 | 7D / 30D / 90D | trends / signals / points | 时间窗口状态条、趋势详情观察轴 | date、后续 evidence state | 没有数据时画趋势线 |

关系展示原则：

- 显式关系优先。当前显式关系为空时，只能展示“同类内容”“相关材料”，不得写成“支撑关系已确认”。
- 观点卡必须先通过 `feature / sidebar` 和前台 lane 过滤。
- 趋势观察可以前台展示，但必须标注证据状态和缺口。
- 正式趋势报告必须有多源事实和反证处理，不能由单条信号或单个观点升级。
