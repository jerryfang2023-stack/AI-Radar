# PRD-007：Admin 管理后台独立升级

更新时间：2026-05-04  
owner：`pm`  
协作 agent：`ui-ue`、`copy`、`dev`、`qa`、`workflow`

## 1. 背景

Admin 后台当前承担内容同步、栏目编辑、用户权限和订阅记录管理，但仍是本地控制台原型。随着观澜AI进入持续运营，后台需要从“能操作”升级为“方便、快捷、清晰、低失误”的管理工作台。

本 PRD 将 Admin 从会员模块中拆出，作为独立后续开发任务。会员模块只负责普通用户的注册、登录、账户、订阅和阅读权限；Admin 模块负责运营管理、内容管理、权限管理、质量检查和发布准备。

本轮 PM 规划只输出需求，不修改 `04-Site/admin.html`，不接入真实云端权限、数据库或支付。

## 2. 产品目标

1. 让管理员更快完成每日更新、复核和发布。
2. 让内容、用户、订阅、质量检查集中在一个清晰后台中。
3. 降低误操作风险，尤其是删除、覆盖、发布、同步和权限修改。
4. 普通前台不暴露任何后台能力。
5. 为未来云端正式后台预留信息架构、权限边界和审计口径。

## 3. 当前 Admin 观察

基于 `04-Site/admin.html` 当前结构：

| 区域 | 当前能力 | 问题 |
|---|---|---|
| 顶部导航 | 仍使用普通前台导航，含首页、Daily、Signals、Priorities、Trends、Opportunities、Tags | 后台缺少独立信息架构；Admin 与前台视觉边界不够清楚 |
| 首屏 | 本地控制台、Ready for update | 缺少今日待办、同步健康、质量状态、用户权限概览 |
| 同步区 | 一键同步、同步说明 | 缺少同步前后状态、失败原因、数量差异和发布闸门 |
| 本机访问权限 | 开通本浏览器、查看同步说明、清除本机编辑缓存、编辑栏目入口 | 能力有用，但需要转成更明确的 Admin 工具区；高风险操作需二次确认 |
| 内容管理 | 选择栏目和条目，编辑 JSON，保存、复制、删除 | 不能长期只依赖纯 JSON；需要列表、搜索、状态、结构化字段编辑和变更预览 |
| 用户与阅读权限 | 默认注册有效天数、用户列表 | 需要用户筛选、权限批量查看、到期/延期操作记录 |
| 订阅记录 | 订单列表 | 需要套餐状态、支付状态、手动确认、订单与用户关联 |

## 4. 模块清单

### 4.1 今日工作台

目标：管理员打开 Admin 后立即知道今天能否发布。

核心模块：

- 今日内容状态：Daily、Signals、Priority Engine、Trends、The Point、Opportunities 的最新数量和更新时间。
- 同步状态：最近一次同步时间、结果、失败原因、数据文件写入状态。
- 质量状态：关系检查硬错误、软提醒、标签检查、The Point 质量检查。
- 待处理事项：缺字段、缺关联、待下架、待复核、到期用户、异常订单。
- 发布准备：发布 checklist、备份提示、回滚提示。

首屏原则：

- 优先展示“今天能不能发布”。
- 常用动作放在首屏：同步、质量检查、查看待处理、进入内容管理。
- 不使用对外营销语，全部面向后台操作。

### 4.2 内容管理

目标：承接 Daily Brief、Signals、Priority Engine、The Point、Trends、Opportunities 的日常管理。

能力清单：

- 按栏目查看内容：Daily、Signals、Priority Engine、The Point、Trends、Opportunities。
- 搜索和筛选：日期、状态、标签、赛道、关联状态、质量状态。
- 内容状态：draft、review、published、archived、needs_review。
- 结构化编辑：标题、摘要、来源、标签、关联实体、状态、发布时间。
- 详情编辑：允许展开高级 JSON，但不把 JSON 作为默认入口。
- 复制为新条目。
- 下架 / 恢复。
- 删除前二次确认。
- 保存结果清楚区分“当前浏览器缓存”和“网站数据文件”。

后台命名：

- Scoring 在后台统一称为 `Priority Engine`。
- Tags 不作为一线前台栏目，但 Admin 可作为筛选、治理和关系网络能力维护。

### 4.3 用户与权限

目标：管理注册用户、阅读有效期、订阅状态和管理员身份边界。

能力清单：

- 用户列表：邮箱 / 手机号、昵称或公司、注册时间、最近访问、当前状态。
- 状态筛选：有效、即将到期、已到期、订阅会员、管理员。
- 权限查看：阅读有效期开始 / 到期时间、套餐来源、手动调整记录。
- 权限操作：按周 / 月 / 季 / 年延长，手动设为到期，恢复默认有效期。
- 默认注册有效期设置：默认 7 天，可由 Admin 调整。
- 高风险操作：设为管理员、取消管理员、清空权限、缩短有效期，必须二次确认。

### 4.4 订阅与订单

目标：在未接真实支付前，先管理套餐展示、订单占位和手动确认。

能力清单：

- 套餐管理：月 / 季 / 年套餐是否显示、价格占位、说明文案。
- 订单列表：订单号、用户、套餐、金额、状态、创建时间、更新时间。
- 支付状态：待支付、已支付、失败、手动确认、已取消。
- 手动标记支付结果，并同步延长用户阅读权限。
- 异常订单提示：支付成功但权限未延长、用户不存在、重复订单。

非目标：

- 不接入真实支付网关。
- 不做发票、退款、企业采购流程。

### 4.5 质量检查

目标：把每日内容入站后的风险集中给管理员看。

能力清单：

- 关系检查：硬错误、软提醒、孤立内容、缺关联。
- 标签检查：未知标签、禁用别名、标签膨胀、建议合并。
- The Point 检查：来源、全文、译文、speaker/timecode、短链残留。
- Signal 检查：事件、来源、商业含义、机会拆解 6 点。
- Opportunity 检查：公司名标题、评分证据、关联 Signal / Trend。
- 质量报告入口：读取最近报告，不在普通前台暴露。

### 4.6 发布准备

目标：在云端部署前先形成后台发布闸门。

能力清单：

- 同步已完成。
- 质量检查硬错误为 0。
- 软提醒有处理结论。
- 数据文件已备份。
- 普通前台无后台入口。
- 关键页面可打开。
- 发布备注和回滚路径已记录。

### 4.7 系统设置

目标：集中管理本地配置和后续云端配置占位。

能力清单：

- 默认注册有效期。
- 内容路径配置说明。
- 本地管理服务状态。
- 缓存清理。
- 备份与恢复提示。
- 自动化任务状态说明。

非目标：

- 本阶段不做真实云端环境配置写入。
- 本阶段不改自动化任务时间线或运行口径。

## 5. 页面地图

### 5.1 P0 单页后台

第一阶段仍可保留 `admin.html` 单页，但必须形成后台专属信息架构：

```text
admin.html
├─ Admin 顶部 / 侧栏导航
│  ├─ 今日工作台
│  ├─ 内容管理
│  ├─ 用户与权限
│  ├─ 订阅与订单
│  ├─ 质量检查
│  └─ 系统设置
├─ 今日工作台
│  ├─ 今日状态总览
│  ├─ 同步与质量结果
│  ├─ 待处理事项
│  └─ 发布准备
├─ 内容管理
│  ├─ 栏目切换
│  ├─ 列表 / 搜索 / 筛选
│  ├─ 结构化编辑
│  └─ 高级 JSON 展开
├─ 用户与权限
│  ├─ 默认有效期
│  ├─ 用户列表
│  └─ 权限操作
├─ 订阅与订单
│  ├─ 套餐管理
│  └─ 订单列表
├─ 质量检查
│  ├─ 运行检查
│  ├─ 结果摘要
│  └─ 最近报告
└─ 系统设置
   ├─ 本地服务状态
   ├─ 缓存清理
   └─ 备份 / 恢复提示
```

### 5.2 P1 多页后台

后续云端版本可拆成多页：

```text
admin.html                 今日工作台
admin-content.html         内容管理
admin-members.html         用户与权限
admin-orders.html          订阅与订单
admin-quality.html         质量检查
admin-settings.html        系统设置
```

拆页条件：

- 单页滚动过长，常用操作需要频繁查找。
- 内容编辑、用户管理和质量检查出现相互干扰。
- 云端部署前需要更清楚的权限和审计边界。

## 6. 页面设计方向

Admin 是运营后台，不是前台栏目页。

设计原则：

- 信息密度高但不拥挤，优先支持重复操作和快速扫读。
- 使用后台专属导航，不沿用普通前台栏目导航。
- 首屏以状态和操作为主，不做大面积品牌展示。
- 卡片只用于摘要和列表项，不把页面堆成卡片墙。
- 高风险动作使用明确按钮层级、风险提示和二次确认。
- 桌面端优先完成主要管理操作，移动端至少可查看状态和处理轻量任务。
- Copy 面向管理员任务，不使用公开前台的商业化表达。

推荐布局：

- 左侧窄导航 + 顶部状态条 + 主工作区。
- 首屏四个状态块：同步、质量、内容、用户。
- 内容管理采用左侧列表 / 右侧编辑详情，或上方筛选 / 下方表格 / 侧边抽屉。
- 用户与订单采用表格优先，而不是大卡片优先。
- 质量检查采用状态摘要 + 问题列表 + 报告入口。

## 7. P0 / P1 开发批次

### P0：本地 Admin 工作台成型

目标：不改变底层权限和自动化代码前提下，让现有 Admin 变成可用的后台工作台。

需求：

1. 重构 `admin.html` 信息架构为后台专属导航。
2. 首屏新增今日工作台，展示同步、质量、内容、用户、订单关键状态。
3. 内容管理从纯 JSON 默认编辑升级为列表 + 结构化摘要 + 高级 JSON。
4. 用户与权限区支持更清楚的状态筛选、有效期显示和延期操作入口。
5. 订阅与订单区显示支付状态和手动确认入口。
6. 增加质量检查区，集中展示关系检查、标签检查和发布前 checklist。
7. 高风险操作增加二次确认文案和状态反馈。
8. 保持所有 Admin 能力只在 Admin 页面或显式 Admin 模式中出现。

验收：

- Admin 首屏能回答“今天能不能发布”。
- 管理员能在一个页面完成同步、查看质量状态、进入内容编辑、查看用户权限。
- 普通前台无 Admin 导航、编辑、同步、恢复、JSON 或质量检查入口。
- JSON 编辑仍可作为高级能力保留，但不是唯一管理方式。

### P1：云端后台准备

目标：为真实权限、数据库、支付和部署做结构预留，但不在本批次接入。

需求：

1. 拆分多页后台或建立可拆分路由结构。
2. 增加操作日志模型：谁在什么时候改了什么。
3. 增加备份、恢复、回滚提示和发布记录。
4. 为真实用户表、订单表、内容表预留字段映射。
5. 预留多管理员权限分级，但不实现复杂角色系统。
6. 与上线准备任务对齐部署、备份、回滚和权限方案。

验收：

- Dev Agent 可据此设计云端数据写入和权限方案。
- QA Agent 可按普通用户、有效用户、到期用户、管理员四种状态验收。
- 不影响 `ai-the-point`、`ai-2`、`ai-3` 自动化任务。

## 8. 非目标

- 不接入真实支付网关。
- 不接入真实云端数据库。
- 不做复杂 CRM、线索管理或企业销售流程。
- 不做多管理员权限分级实现。
- 不把 Admin 功能放入普通前台导航。
- 不把 JSON 编辑作为唯一管理方式。
- 不改每日雷达、The Point 或统一同步自动化的运行顺序。

## 9. 文案原则

推荐表达：

- `今日待处理`
- `同步最新数据`
- `保存到网站数据文件`
- `仅保存到当前浏览器`
- `关系检查通过`
- `仍有软提醒需要复核`
- `延长阅读权限`
- `发布前检查`
- `手动确认支付`

避免表达：

- `试读审批`
- `申请访问`
- `随便改一下`
- `内部临时入口`
- `看起来没问题`
- `用户一定会转化`

## 11. 验收标准

### 普通前台边界

- 普通页面无 Admin 导航。
- 普通页面无编辑、同步、恢复、JSON、发布、质量检查等后台入口。
- 只有 Admin 路径或显式 `?admin=1` 编辑路径出现后台工具。

### Admin 功能

- 管理员能在首屏看到同步、内容、用户、订单、质量检查关键状态。
- 用户权限管理可完成查看、延长、到期。
- 订阅订单可查看和手动标记支付结果。
- 内容管理可完成选择、编辑、复制、删除。
- 保存状态文案明确区分本地浏览器与网站数据文件。
- 高风险操作有二次确认或明确风险提示。

### Admin 体验

- 页面模块分区清楚。
- 常用操作不需要滚动太深。
- 文案面向管理任务，不使用对外营销语。
- 移动端可查看关键状态，桌面端可完成主要管理操作。

### 自动化影响

- 不改变 Markdown 文件命名、目录或 frontmatter。
- 不改变 Signal / Priority / Trend / Opportunity / Point 字段规则。
- 不改变 `sync-data.mjs`、`check-relations.mjs`、`check-point-quality.mjs`、`unified-site-sync.mjs` 的运行口径。
- 不改变 `ai-the-point`、`ai-2`、`ai-3` 的运行顺序。

## 12. 建议开发顺序

1. UI / UE Agent 输出 Admin P0 单页工作台结构。
2. Dev Agent 基于 P0 结构重构 `admin.html` 和必要样式 / 交互。
3. Dev Agent 接入或展示已有质量检查结果入口。
4. QA Agent 做 Admin 与普通前台边界验收。
5. PM Agent 根据 QA 结果判断是否进入 P1 云端后台准备。
