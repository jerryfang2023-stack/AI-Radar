# V2 Commercial Brief Depth Standard

## 1. 目标

每日内容不是把新闻改写成长文，而是把新闻翻译成老板能用的商业判断。

三类核心内容必须同时做到：

- 事实清楚：发生了什么、谁参与、产品或市场状态是什么。
- 商业清楚：这件事会改变谁的成本、收入、效率、风险或竞争位置。
- 边界清楚：哪些还只是线索，哪些已有一手证据，哪些需要继续观察。
- 表达清楚：不用专家黑话堆砌，不用模板句填充，不写“值得关注”这类空判断。
- 展示清楚：每篇都要有结构化展示元素，帮助普通老板快速看懂变化、位置和风险。

## 2. 读者口径

默认读者不是 AI 专家，而是需要做经营、产品、投资、采购或转型判断的普通老板。

写作要求：

- 先说人话，再说术语。术语必须跟一句白话解释。
- 不用“范式”“涌现”“多模态能力边界”“智能体编排”等词独立承担判断。
- 不把模型名、论文名、框架名当作结论；必须说明它改变了哪个业务动作。
- 少写抽象评价，多写可观察变化：价格、客户、渠道、交付方式、采购门槛、替代流程。
- 每段只讲一个意思，段落内要有事实、解释或判断，不能只有铺垫。
- 允许专业，但必须让非技术读者知道“这和我的公司有什么关系”。

禁用写法：

- “这标志着 AI 进入新阶段”但不说明新在哪里。
- “有望提升效率”但不说明提升哪个岗位、哪个流程。
- “商业化潜力巨大”但不说明谁付钱、为什么现在付。
- “生态正在形成”但不说明生态里有哪些角色、谁掌握议价权。
- 只罗列新闻标题、融资金额、产品功能，没有判断链。

## 3. 每日信号

定位：每日信号是“今天发生的一件事，对某类老板意味着什么”。

每条必须包含：

| 模块 | 必须讲清楚 |
|---|---|
| 一句话判断 | 这不是新闻摘要，而是商业变化判断。 |
| 发生了什么 | 用 3-5 句话讲清事实，不堆来源。 |
| 为什么现在出现 | 技术、成本、监管、客户需求或渠道变化中至少讲清 2 个原因。 |
| 谁会先受影响 | 写清行业、岗位、流程和公司类型，不写泛泛的“企业”。 |
| 商业变量 | 这件事影响成本、收入、交付速度、转化率、留存、风险还是采购门槛。 |
| 证据梯子 | 官方 / 产品 / 客户 / 媒体 / 社区线索各自提供了什么事实。 |
| 反证与边界 | 为什么它可能没那么快发生，缺哪类证据。 |
| 老板该看什么 | 未来 7-30 天该观察什么信号，不给经营指令。 |

展示元素至少选 3 个：

- 时间线：发布、融资、客户采用、竞品跟进、监管或诉讼节点。
- 影响坐标：行业、岗位、流程三列，标出影响位置。
- 证据梯子：S/A/B/C/M 来源分层，M 只显示为采集通道。
- 商业变量卡：成本、收入、效率、风险、采购五项中标出受影响项。
- 反证卡：尚未看到客户、价格、合规、规模化、留存中的哪些证据。
- 小型热力条：短期影响、中期影响、确定性、可迁移性。

## 4. 深挖机会卡

定位：深挖机会卡不是“推荐一个项目”，而是一份小型商业机会备忘录。

每篇必须包含：

| 模块 | 必须讲清楚 |
|---|---|
| 机会定义 | 不用公司名做标题，用一个可经营的机会命题。 |
| 具体痛点 | 谁现在花钱、花时间或承担风险，痛在哪里。 |
| 原有替代方案 | 今天客户不用 AI 时怎么解决，为什么不好。 |
| AI 带来的结构变化 | 是成本下降、交付变快、个性化、合规记录、还是入口变化。 |
| 买单者与预算 | 谁批准预算、预算从哪来、是新预算还是替代旧支出。 |
| 供给地图 | 代表公司、开源项目、大厂、垂直服务商、集成商各在什么位置。 |
| 商业模式 | SaaS、API、席位费、按量、项目制、交易抽成或混合模式，说明优劣。 |
| 证据链 | 至少 5 个来源分别证明市场、产品、客户、资金、风险中的哪一环。 |
| 反向证据 | 写清失败、成本、合规、客户不买单、渠道受限等风险。 |
| 落地约束 | 数据、合规、渠道、客单价、交付团队、平台依赖和竞争卡点。 |
| 机会分级 | 给出 S/A/B/C 机会级别，并说明升级或降级触发条件。 |

展示元素至少选 5 个：

- 机会地图：痛点 -> 解决方案 -> 买单者 -> 交付方式。
- 玩家分层图：大厂、创业公司、开源、集成商、渠道方。
- 商业模式表：收费方式、适合客户、毛利假设、规模化难点。
- 验证清单：客户、价格、复购、部署、合规、渠道六项。
- 雷达图：需求强度、付费能力、技术成熟度、竞争强度、落地难度、证据强度。
- 风险雷达：监管、数据、替代品、客户教育、毛利、交付复杂度。

## 5. 趋势链

定位：趋势链不是热点排行，而是记录“某个商业变化正在处于什么阶段”。

每条趋势更新必须包含：

| 模块 | 必须讲清楚 |
|---|---|
| 趋势定义 | 用行业、岗位或流程定义趋势，不用抽象技术名。 |
| 本期变化 | 和上一次相比，热度、证据或商业化阶段有什么变化。 |
| 阶段判断 | 萌芽、验证、早期商业化、规模化、降温、风险暴露。 |
| 信号簇 | 至少 3 条相关信号组成一个趋势，不用单点新闻当趋势。 |
| 推动力 | 成本、模型能力、客户需求、渠道、政策、资本中哪些在推动。 |
| 阻力 | 采购、合规、交付、数据、利润、替代方案中哪些在拖慢。 |
| 代表案例 | 用案例解释趋势，不用案例替代判断。 |
| 反趋势 | 什么迹象说明这个趋势可能被高估。 |
| 下期观察 | 下一次更新看什么指标或事件。 |

展示元素至少选 4 个：

- 趋势阶段条：从萌芽到规模化，标出当前阶段和上期阶段。
- 趋势线：过去 7 / 30 / 90 天信号数量或强度变化。
- 热力图：行业 x 流程，显示热度与变化方向。
- 信号簇图：3-5 个信号围绕同一趋势主题聚合。
- 推力 / 阻力双栏：让读者同时看到为什么会发生和为什么会变慢。
- 升降级触发器：达到什么证据升级，出现什么风险降级。

## 6. 第三方可借鉴方法

可借鉴的是方法，不是照搬栏目：

- McKinsey Technology Trends Outlook：用新闻、投资、论文、专利、人才等多指标判断趋势活跃度。
- Gartner Hype Cycle：用阶段判断避免把所有热点都写成“爆发”。
- CB Insights market maps：用公司、赛道和资金分布展示市场结构。
- a16z market thesis：用 why now、供需变化和商业模式讲机会。
- The Information / Stratechery：面向商业读者解释公司动作背后的战略和钱。

V2 采用方式：

- 每日信号借鉴新闻 + 商业变量解释。
- 深挖机会卡借鉴 market map + investment memo。
- 趋势链借鉴趋势阶段 + 多指标热度。

## 7. 入库前结构化要求

每日自动化在写正文前，必须先生成以下结构化字段：

```yaml
reader_question: 普通老板看完要解决的问题
one_sentence_judgment: 一句话商业判断
business_variables:
  cost:
  revenue:
  efficiency:
  risk:
  procurement:
evidence_ladder:
  s_sources:
  a_sources:
  b_sources:
  c_or_m_channels:
counter_evidence:
display_elements:
  timeline:
  heatmap:
  radar:
  trend_line:
  opportunity_map:
plain_language_notes:
  terms_to_explain:
  forbidden_jargon:
```

没有这些字段，不得直接生成长文。
