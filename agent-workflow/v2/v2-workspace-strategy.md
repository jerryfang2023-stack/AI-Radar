# V2.0 Workspace Strategy

日期：2026-05-07  
状态：recommendation  
owner：`dev` / `workflow`

## 1. 推荐策略

V2.0 仍在当前项目体系内推进，但开发阶段使用独立分支或 worktree 隔离。

重要口径：V2.0 是正式升级项目，不是测试环境项目。独立分支或 worktree 是为了保护 V1.0 基线、支持可审计迁移和回滚，不是为了把 V2 长期停留在实验区。

```text
01-WaveSight
  主项目、V1.0 归档、V2.0 知识库和调度看板

01-WaveSight-v2-lab
  可选：V2 正式升级开发 worktree，未验收前不覆盖 V1.0 主线
```

## 2. 为什么不新建孤立项目

- V1.0 的 agent-workflow、内容资产、自动化规则和质量闸门仍然有价值。
- V2.0 需要继承 Signal / Trend / Opportunity / The Point 的关系资产。
- 单一项目能避免后续同步、部署、内容迁移和交接割裂。

## 3. 为什么不直接在主工作树开发

- 当前工作树已有大量 V1.0 后续任务、测试管线、自动化报告和待验收内容。
- V2.0 会涉及 Agent、算法、VI、页面母版和可能的目录结构调整。
- 直接改主工作树会提高回滚和验收成本。

## 4. 建议 Dev 操作

由 `V2-5 / WSD-20260507-06-v2-dev-workspace-baseline` 负责执行：

- 确认当前 main / 本地工作树 / Netlify 版本差异。
- 经用户确认后创建 `v1.0-baseline` tag。
- 创建 V2 分支，例如 `codex/v2-planning`。
- 如需要页面或脚本原型，创建 worktree：`01-WaveSight-v2-lab`。
- 所有 V2 代码开发先在 V2 分支或 worktree 完成，验收后再按迁移方案合并或部署到生产主线。

## 5. 正式开发与测试环境边界

V2 分支 / worktree 的定位：

- 是正式升级开发区。
- 不是临时 demo。
- 不是 test-only 管线。
- 产物最终应进入 V2 正式版本，除非 QA 或用户验收判定不通过。

测试环境的定位：

- 用于验证算法、VI、内容源、页面和自动化迁移。
- 不替代最终生产交付。
- 不作为“完成”的定义。

完成定义：

- 通过对应硬闸门。
- 用户验收或调度中枢 accepted。
- 有回滚和部署记录。
- 合并或部署到正式 V2 目标环境。
