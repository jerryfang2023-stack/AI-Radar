# V2｜观澜AI商业内参与 AI商业热力图增值产品方案

## 0. 结论

`观澜AI商业内参 / AI商业热力图` 通过 PM 新增功能门禁，建议进入 V2 产品架构与 PRD，但不直接作为普通公开一级栏目开发。

推荐决策：

```text
AI商业内参：进入 V2 增值产品层，作为会员/增值产品入口。
AI商业热力图：作为 AI商业内参的核心模块与数据资产，不做普通公开榜单。
首期 MVP：周度 AI商业热力周报 + 本期高热三元组 + 可展开证据来源。
月度完整版：作为第二阶段内参产品，等待 V2-2 算法与 7 日测试稳定后推进。
```

一句话产品定义：

```text
观澜AI商业内参基于 Signals、Point、Opportunity 和 Trends 的证据，形成 AI商业热力图，帮助商业决策者判断 AI 正在影响哪些行业、岗位和工作流。
```

核心边界：

- 四栏目继续保留，负责日常证据输入。
- AI商业内参负责周期性判断输出。
- AI商业热力图是内参的判断资产，不是新闻榜单、公司榜单、岗位替代榜或泛仪表盘。
- 本任务只做产品规划，不修改 `04-Site/`、生产内容源、同步脚本或自动化本体。

## 1. 产品定位

### 1.1 所属产品层

| 层级 | 组成 | 作用 |
|---|---|---|
| 公开内容层 | Signals / Point / Opportunity / Trends | 日常证据、观点、机会和趋势输入 |
| 判断资产层 | HeatEvidence / HeatCards / HeatmapTriple | 把四栏目转成可计算、可追溯的热力证据 |
| 增值产品层 | 观澜AI商业内参 | 周度 / 月度输出高密度商业判断 |
| 核心模块 | AI商业热力图 | 呈现行业、岗位、流程和三元组的 AI 影响热力 |

### 1.2 用户任务

用户打开 AI商业内参，不是为了看更多 AI 新闻，而是为了在 30 秒内完成三个判断：

1. 本期哪些行业、岗位和工作流正在被 AI 明显影响。
2. 这些变化由哪些 Signals、Point、Opportunity 和 Trends 支撑。
3. 哪些方向值得继续观察、试点或进入内部讨论。

### 1.3 商业价值

AI商业内参强化观澜AI的付费理由：

- 从日更内容升级为周期性决策产品。
- 把零散信号转成可复盘的行业 / 岗位 / 流程判断。
- 为会员订阅、行业定制简报、闭门简报和企业服务提供样张。
- 增强销售沟通可信度：用户能看到“判断从哪里来、为什么升温、反证在哪里”。

## 2. 四栏目与 AI内参关系

| 栏目 | V2 中继续承担 | 转为 HeatEvidence 的重点 | 在内参中的位置 |
|---|---|---|---|
| Signals | 外部变化证据 | 事件、行业、岗位、流程、影响方式、可信度 | 证明“发生了什么变化” |
| Point | 一线实践校准 | builder 观点、实践反馈、争议、阻力、非共识判断 | 校准“是不是过热或被低估” |
| Opportunity | 商业机会解释 | 目标客户、核心需求、商业价值、风险限制 | 解释“为什么可能形成机会” |
| Trends | 长期热力沉淀 | 持续性、阶段、趋势分化、反证 | 判断“是否持续成立” |

关系定义：

```text
Signals / Point / Opportunity / Trends
-> HeatEvidence
-> IndustryHeatCard / JobHeatCard / WorkflowHeatCard / HeatmapTripleCard
-> AIBriefIssue
-> 观澜AI商业内参
```

四栏目不被热力图替代。热力图也不承接日常内容流，它只承接周期性聚合后的判断。

## 3. AI商业热力图数据对象

### 3.1 HeatEvidence

HeatEvidence 是四栏目进入热力图前的统一证据对象。它不是前台展示字段，而是 V2 判断资产层的基本单位。

必须包含：

| 字段组 | 字段 | 产品口径 |
|---|---|---|
| 来源 | `sourceType` / `sourceId` / `title` | 保证每个热力判断能反向追溯 |
| 标签 | `industryTags` / `jobTags` / `workflowTags` | 支撑行业、岗位、流程和三元组聚合 |
| 影响 | `impactModes` / `businessValueTags` | 判断影响收入、成本、效率、客户体验或管理难度 |
| 分数 | `evidenceScore` / `confidenceScore` / `heatContribution` | 用于热力贡献，不直接对外包装成确定判断 |
| 方向 | `heatDirection` | 支持升温、稳定、降温和混合信号 |
| 时间 | `period` / `publishedAt` | 支持周度、月度和跨期变化 |

进入热力图证据池的最低条件：

```text
至少命中 1 个行业标签
并且至少命中 1 个岗位标签或 1 个流程标签
并且能说明 AI 影响方式或商业价值标签
```

不满足条件的内容可保留在原栏目，但不进入本期热力计算。

### 3.1A 商业热力度维度

商业热力图的核心不是“行业 / 岗位 / 流程谁排名更高”，而是解释商业变化为什么成立。V2 商业热力度采用 6 个维度：

| 维度 | 回答什么 | 产品用途 |
|---|---|---|
| 影响坐标 | 哪个行业 / 岗位 / 流程被影响 | 定义热力图坐标 |
| 变化方式 | 替代、增强、压缩、重组、平台化或外包化 | 解释热力来源 |
| 证据密度 | 有多少信号、来源、案例、观点支撑 | 防止单点新闻制造热度 |
| 变化速度 | 本期相对上期是否明显升温或降温 | 支撑周度 / 月度变化判断 |
| 商业强度 | 是否影响预算、收入、成本、效率、交付或风险 | 区分信息热度和商业热度 |
| 争议与边界 | 是否存在反证、失败反馈、监管限制或集成难度 | 防止把高曝光误判为高机会 |

商业热力度建议表达为：

```text
商业热力度 = 影响坐标 × 变化方式 × 证据密度 × 变化速度 × 商业强度 × 争议边界
```

前台展示时不直接暴露公式，只解释“为什么升温 / 为什么有争议 / 边界在哪里”。

### 3.2 Heat Cards

| 对象 | 回答的问题 | MVP 是否需要 |
|---|---|---|
| IndustryHeatCard | AI 正在影响哪些行业 | 需要 |
| JobHeatCard | AI 正在影响哪些岗位 | 需要 |
| WorkflowHeatCard | AI 正在影响哪些流程 | 需要 |
| HeatmapTripleCard | 哪些行业 x 岗位 x 流程组合正在升温 | P0 核心 |

MVP 优先级：

```text
HeatmapTripleCard > WorkflowHeatCard > IndustryHeatCard > JobHeatCard
```

原因：三元组最接近商业决策，避免把热力图做成粗糙行业榜或岗位替代榜。

### 3.3 AIBriefIssue

AIBriefIssue 是内参一期的产品对象。建议支持：

| 字段 | 口径 |
|---|---|
| `issueType` | `weekly` / `monthly` |
| `period` | 例如 `2026-W19` 或 `2026-05` |
| `executiveSummary` | 3-5 条本期核心判断 |
| `topTriples` | 本期最值得看的三元组 |
| `risingHeatPoints` | 升温区域 |
| `coolingHeatPoints` | 降温区域 |
| `controversialHeatPoints` | 争议区域 |
| `evidenceSummary` | Signals / Points / Opportunities / Trends 的证据摘要 |
| `guanlanConclusion` | 克制结论，不替用户下最终经营或投资判断 |

## 4. 周度 / 月度 Issue 口径

### 4.1 周度版

名称建议：

```text
观澜AI商业热力周报
```

用户任务：

- 每周快速看清 AI 商业变化中正在升温的具体组合。
- 给内部周会、选题、投资观察、产品调研或客户访谈提供讨论素材。

建议结构：

1. 本期核心判断。
2. 本期 AI商业热力图总览。
3. 高热三元组。
4. 升温工作流。
5. 关键证据。
6. 争议与反证。
7. 观澜观察边界。

### 4.2 月度版

名称建议：

```text
观澜AI商业热力图｜月度完整版
```

用户任务：

- 判断一个月内哪些行业、岗位和工作流形成持续变化。
- 支撑定制简报、闭门交流、行业专题和付费销售。

建议结构：

1. 月度总判断。
2. 行业热力地图。
3. 岗位热力地图。
4. 工作流热力地图。
5. 高热三元组。
6. 跨期变化。
7. 争议期与降温区。
8. 证据索引。

### 4.3 推进顺序

| 阶段 | 版本 | 原因 |
---|---|---|
| MVP | 周度版 | 更新频率适中，能快速验证复访和付费感 |
| P1 | 月度完整版 | 需要更稳定的标签、证据和跨期比较 |
| P2 | 季度专题内参 | 适合商业化和企业服务，但不应先做 |

## 6. 模块决策表

| 决策项 | 结论 |

| 模块名称 | 观澜AI商业内参 |
| 用户是谁 | 商业决策者、投资人、创始人、产品负责人、业务负责人 |
| 用户为什么需要 | 需要把每天分散的 AI 信号转成周期性判断，减少噪音，支持内部讨论和方向筛选 |

| 使用路径是否顺畅 | 建议作为会员 / 增值入口，最新一期优先展示，往期归档次级展示 |

| 需要交接的 Agent | V2 Strategy & Product Architecture / V2 AI Brief Editorial / V2 VI / V2 Platform |

### 6.2 AI商业热力图

| 决策项 | 结论 |
|---|---|
| 模块名称 | AI商业热力图 |

## 7. MVP 范围

### 7.1 P0 必须做

1. 定义 HeatEvidence 产品口径。
2. 定义 Industry / Job / Workflow / Triple 四类 HeatCard 的最小字段。
3. 先做周度 `AIBriefIssue`。
4. 每期输出 3-5 条核心判断。
5. 每期输出 Top 5 高热三元组。
6. 每个三元组能反向展开 Signals / Point / Opportunity / Trends 证据。
7. 标记升温、稳定、降温、争议四类方向。
8. 使用 V2 VI：澜线、地平线、留白、报告式阅读，不做泛仪表盘。

### 7.2 P0 可简化

- 行业、岗位、流程榜单只做摘要，不做完整全量库。
- 热力评分可以先采用半结构化评分，等待 V2-2 正式算法定稿。
- 月度完整版先用周度数据累积生成样稿，不马上做自动生产。
- 公开页面只展示样例和摘要，完整体验放到会员 / 增值层规划。

### 7.3 暂不做清单

- 不新增 `04-Site` 页面代码。
- 不修改生产 Markdown frontmatter。
- 不替换 `ai-2` 或 `ai-3` 自动化。
- 不做完整支付、PDF 下载、企业后台和真实权限系统。
- 不做全量行业库、岗位库和流程库的复杂管理后台。
- 不做“AI 是否替代岗位”的简单风险榜。
- 不输出投资建议、经营命令或确定性机会。
- 不把热力图做成每日公开榜单。

## 8. 体验与视觉原则

AI商业内参和热力图必须吸收 V2 VI：

- 使用 `方案 A：极简澜线型` 的澜线、地平线、留白和文字秩序。
- 视觉像周期性商业决策文件，而不是资讯栏目或后台报表。
- 热力图像“商业变化判断地图”，不做彩虹热力表、金融 K 线、监控大屏或排行榜。
- 香槟金只作为期号、重点点位、分割线和会员标识，不大面积铺开。
- 每个热力点先给判断，再给证据，不堆字段。

建议页面层级由后续 V2-4 / V2-5 决定，当前只给产品口径：

```text
会员 / 增值入口
-> 最新一期 AI商业内参
-> AI商业热力图总览
-> 高热三元组详情
-> 证据来源展开
-> 往期内参
```

## 9. 自动化影响判断

本任务不修改任何自动化任务，因此本轮不直接影响：

- `ai-the-point`
- `ai-2`
- `ai-3`

但后续若将 HeatEvidence、HeatCards 或 AIBriefIssue 进入生产链路，会影响：

- 四栏目 Markdown 字段或解析规则。
- `sync-data.mjs` 数据结构。
- `check-relations.mjs` 关系检查范围。
- `unified-site-sync.mjs` 同步闸门。
- `ai-2` 每日雷达生成要求。
- `ai-the-point` 对 Point Fragment / Cluster / Insight 的输出要求。

因此后续必须单独派发自动化影响任务，不能在页面 Dev 或算法 Dev 中顺手修改生产口径。

## 10. 对后续任务的交接要求

### 10.1 交接给 V2-2

V2-2 必须明确：

- HeatEvidence 字段是否需要进入正式 schema。
- 行业 / 岗位 / 流程标签字典的最小可行范围。
- 四栏目到 HeatEvidence 的转换规则。
- 热力评分中哪些权重可先半自动，哪些必须人工复核。
- `AIBriefIssue` weekly / monthly 的最小输入。
- 争议期、反证和降温如何参与热力计算。

V2-3 已通过，本任务吸收其规则。后续任何内参 / 热力图视觉落地必须继续遵守：
