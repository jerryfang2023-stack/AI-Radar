# 观澜AI页面与文案质检 Skill

更新时间：2026-05-11  
状态：current / V2-only  
owner：QA / Acceptance Agent  
协作：UI / UE Agent、Copy Agent、PM Agent、Workflow Agent  
Skill Pattern：Reviewer + Inversion + Pipeline + Tool Wrapper

## 1. Skill 定位

本 Skill 用于独立审查观澜AI网站页面、栏目页、详情页、商业内参页、热力图页、小程序诊断页、会员入口页和页面级文案。

它不是错别字检查，也不是普通的“页面好看不好看”检查，而是回答四个问题：

1. 这是不是观澜AI？
2. 有没有商业判断？
3. 像不像真人写的？
4. 能不能让老板产生信任？

本 Skill 只做评审、打分、指出问题和生成修改指令，不直接修改页面代码、内容文件或 VI 资产。

## 2. 必读输入

执行独立质检时，默认读取：

1. `AGENTS.md`
2. `agent-workflow/governance/current-context.md`
3. 对应页面 / 文案开发任务派发单
4. 对应开发窗口 closeout
5. 页面截图、页面 URL、页面代码或文案文件
6. `agent-workflow/product/DESIGN.md`
7. `agent-workflow/product/COPY.md`
8. `docs/brand/wavesight-ai-vi/USAGE.md`
9. `docs/brand/wavesight-ai-vi/typography-guidelines.md`
10. `docs/brand/wavesight-ai-vi/brand-tokens.css`

涉及热力图、商业内参或机会判断时，补读对应设计规范、PRD、内容来源说明或数据 closeout。

如果缺少截图、页面 URL、开发 closeout 或关键规范，质检窗口必须标记为 `needs-input`，不得给出 accepted。

## 3. 前置原则

质检必须以当前可信源为准：

- 观澜项目定位：面向商业决策者的 AI 机会判断系统。
- 观澜 VI：深海蓝、暖白、雾灰和少量香槟金；Logo 引用正式 SVG；页面背景 `#FFFDF8` / `--gl-bg-page`。
- 观澜字体：中文标题用衬线，正文与界面用黑体，英文标签用 Inter 系，数字用 Mono 系。
- 观澜文案：冷静、有判断、有温度的商业内参风，避免 AI 味、模板句式、空泛正确和过度营销。
- 观澜内容：从新闻摘要升级为商业判断，围绕行业、岗位、流程、影响路径和机会边界展开。

当前尚未形成单独文件的规范，质检报告必须写明“待补文件”，不得假装已有。例如小程序诊断页若没有专门 PRD，应引用当前项目定位、`DESIGN.md`、`COPY.md` 和派发单作为临时依据。

## 4. 七维评分

每次质检按 7 个维度打分，每项 10 分，总分 70 分。

| 维度 | 检查重点 |
|---|---|
| 定位一致性 | 是否像观澜AI，而不是 AI 新闻站、AI 工具导航、科技媒体、SaaS 模板页或知识付费落地页；是否面向企业经营者、资源型合伙人和行业操盘手。 |
| 信息架构 | 阅读路径是否清晰，是否做到先判断、再事实、再影响、再机会、再行动入口；第一眼能否看懂页面价值。 |
| 商业判断 | 是否回答发生了什么、为什么重要、影响谁、影响哪个行业 / 岗位 / 流程、机会在哪里、风险在哪里、企业该如何判断。 |
| 文案自然度 | 是否有 AI 味、机械分点、重复句式、空泛正确、过度营销、内部话术和缺少真实项目感的问题。 |
| 视觉体验 | 是否符合观澜 VI、字体、字号、间距、卡片和留白规则；是否高级、克制、有内参感，避免科技感过度和模板感。 |
| 页面节奏 | 是否有轻重变化；是否存在卡片过多、模块重复、主次不清、占位过大、同屏元素争抢注意力。 |
| 可信度 | 判断是否有事实支撑；是否区分事实和观点；是否避免夸大、确定性结论、单一新闻推出大结论、海外案例直接等同国内机会。 |

转化路径不计入 70 分，但必须单独输出建议，检查主 CTA 是否清晰、是否需要低门槛入口、是否过早销售、是否牺牲克制感和专业感。

## 5. 评分解释

| 分数 | 判断 |
|---:|---|
| 9-10 | 可以上线，仅需微调 |
| 7-8 | 基本合格，但需要优化局部 |
| 5-6 | 方向可用，但问题明显 |
| 3-4 | 定位或表达严重跑偏 |
| 1-2 | 不符合观澜AI，需要重做 |

评分必须给出扣分原因。不得凭感觉打高分，不得只写“符合规范”。

## 6. 硬闸门

| 结论 | 标准 |
|---|---|
| `accepted` | 总分不少于 58 / 70；定位一致性、商业判断、文案自然度、可信度均不低于 8；其他维度不低于 7。 |
| `needs-revision` | 总分 49-57，或任一非核心维度为 6；必须退回 Dev / Copy 修改。 |
| `failed` | 总分低于 49，或定位一致性、商业判断、文案自然度、可信度任一项低于 7；必须重做关键模块。 |
| `needs-input` | 缺少截图、URL、closeout、核心规范或可检查页面，无法形成有效独立质检。 |

额外硬停顿：

1. 如果页面看起来不像观澜AI，即使语法、链接和截图通过，也不得 accepted。
2. 如果文案明显像 AI 模板，即使页面美观，也不得 accepted。
3. 如果公开前台出现 Admin、JSON、同步、恢复、编辑、字段、后台等痕迹，不得 accepted。
4. 如果 Opportunity 标题写公司名，或 Signal 标题只是公司 / 产品 / 新闻标题，不得 accepted。
5. 如果页面把趋势、融资热度、单一新闻或海外个案直接写成确定机会，不得 accepted。
6. 如果开发窗口使用本 Skill 给自己打 accepted，该结论无效。

## 7. 独立质检流程

后续页面 / 文案任务必须采用以下流程：

```text
页面 / 文案开发完成
-> 开发窗口写 closeout
-> 调度窗口派发独立质检任务
-> 质检窗口按本 Skill 打分并输出修改指令
-> 不达标则退回 Dev / Copy
-> 达标后调度窗口才可 accepted
```

流程规则：

1. 页面 / 文案开发窗口不得使用本 Skill 给自己打 accepted。
2. 本 Skill 默认由独立 QA / Acceptance 窗口执行。
3. 质检窗口只读页面、截图、代码、文案和 closeout，不直接修改页面。
4. 未通过质检的任务必须退回 Dev / Copy 修改，不得由调度窗口直接 accepted。
5. 质检报告必须保存为 UTF-8 Markdown。
6. 质检报告必须给出可执行修改提示词，而不是泛泛说“提升高级感”。
7. 当前 V2 桌面端优先阶段，桌面端视觉和页面节奏是硬闸门；移动端只做基础不崩坏 / 无严重横向溢出的非阻塞观察，除非派发单明确要求移动专项。

## 8. 输出格式

每次质检必须按以下结构输出。

### 1. 总体判断

一句话说明页面是否符合观澜AI定位，以及最大问题。

### 2. 维度评分

| 维度 | 得分 | 理由 |
|---|---:|---|
| 定位一致性 |  |  |
| 信息架构 |  |  |
| 商业判断 |  |  |
| 文案自然度 |  |  |
| 视觉体验 |  |  |
| 页面节奏 |  |  |
| 可信度 |  |  |
| 总分 |  |  |
| 结论 |  | `accepted` / `needs-revision` / `failed` / `needs-input` |

### 3. 主要问题

列出 3-7 个最影响页面质量的问题，按优先级排序。

### 4. 模块级优化建议

按页面模块逐项指出：保留、合并、弱化、删除、重写或强化。

### 5. 文案改写建议

指出有 AI 味、模板感、空泛感、营销感或内部话术的文案，并给出替换版本。

### 6. 视觉与节奏建议

指出视觉重复、页面太散、卡片过多、主次不清、留白不足、字体层级不稳等问题，并给出改法。

### 7. 转化路径建议

指出主 CTA 是否清晰，是否需要增加低门槛入口，是否过早销售，是否影响观澜AI的克制感。

### 8. Codex 执行指令

把以上建议整理成一段可直接交给 Codex 执行的修改提示词，必须包含修改范围、不可改范围、核心目标和验收重点。

## 9. 质检纪律

- 不参与实现。
- 不替 Dev 改代码。
- 不替 Copy 写整页最终稿，除非质检任务明确要求输出替换候选。
- 不把未运行的截图、浏览器检查或质量门禁写成通过。
- 不继承 failed / not accepted 任务的成功暗示。
- 不因开发任务耗时长、改动多或语法通过而放宽标准。
- 不用“主观高级感”代替可解释扣分。

## 10. 调度接入

页面 / 文案开发派发单必须增加：

```text
开发 closeout 后，调度窗口另派独立页面与文案质检任务。
开发窗口不得自验自收，不得使用 page-copy-quality-review-skill 给自己 accepted。
```

独立质检任务应使用：

```text
agent-workflow/execution/TASK-page-copy-quality-review-template.md
```

调度窗口验收时，只有在独立质检报告结论为 `accepted` 且硬闸门满足时，页面 / 文案开发任务才可进入 accepted。
