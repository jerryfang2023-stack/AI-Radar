# Agent Memory

更新时间：2026-05-03  
owner：`workflow` / `pm`  
状态：长期生效

## 1. 定位

`agent-memory.md` 是观澜AI多 agent 系统的长期组织记忆。

它记录项目中已经确认的规则、反复出现的问题、纠偏经验、禁用表达、数据边界和验收偏好。任何新会话、新窗口或上下文恢复时，长期 agent 都应把本文件作为补充记忆读取。

它不是 PRD，不替代产品战略；它用于防止同类错误反复发生。

## 2. 写入原则

以下情况必须写入本文件或由 Workflow Agent 汇总后写入：

1. 同一类问题出现第二次。
2. 用户明确纠正过某个表达、结构、字段或流程。
3. QA 发现可复用的验收规则。
4. 自动化任务失败并形成可复用降级经验。
5. 某个 agent 的边界被澄清。
6. 某个内容判断规则影响多个模块。

写入时要写清：

- 规则是什么。
- 适用范围。
- 反例是什么。
- 应由哪个 agent 负责执行或检查。
- 是否需要自动化检查。

## 3. 当前核心记忆

### 产品定位

- 观澜AI是面向商业决策者的 AI 机会判断系统。
- 不做 AI 新闻聚合站、AI 工具导航站、公司排行榜、开发者社区、内部 Markdown 管理系统。
- 前台表达必须强调判断、证据、机会、趋势、验证和风险边界。

### 前台与后台边界

- 普通前台不能出现 `Markdown`、`JSON`、同步、脚本、字段、恢复、编辑、后台流程等内部语言。
- Admin 是后台入口，不进入普通用户导航。
- Scoring 后台化为 Priority Engine，不作为普通前台栏目。
- Tags 不作为当前一线栏目，只作为搜索、筛选和关系网络能力。
- Method 暂不作为前台栏目，作为内部方法论资产。

### 今日观察

- 今日观察是 AI 商业判断 Newsletter，不是新闻列表。
- 固定结构为：日期导航、今日判断、发生了什么、值得关注、趋势温度、机会观察、观点与思考、风险与边界、后续观察、关键词表、相关内容索引、来源与证据状态。
- 不输出行动指令，不替用户下投资、经营或合作判断。
- 可以提示“值得观察”“需要验证”“证据增强”“仍需反证”，不能说“必须马上做”。

### 商业信号

- 商业信号必须同时包含事件、证据和商业含义。
- 商业信号标题必须是“事件 + 商业含义”，不能只是公司名或产品名。
- 没有商业含义的新闻不进入商业信号。
- 重复商业信号应合并或补充证据，不新增第二条。
- 进入前台的商业信号必须可追溯到 Raw / Pool / 变化卡，并保留观察边界；不再沿用旧每日雷达“每条 6 点机会拆解”作为当前硬条件。

### 机会判断

- 机会判断是可验证机会方向，不是公司档案，也不是 V2.1 一级导航。
- 机会判断标题不能写成具体公司名清单。
- 公司名只能作为代表案例、证据、来源或相关 Signal。
- 正式机会判断应有成熟度、证据等级、趋势状态、观察边界和后续观察。
- 没有评分、证据或关联关系的机会应进入观察或复核，不直接作为强机会展示。

### 趋势追踪

- 趋势追踪不是趋势名称列表，而是商业势能判断模型。
- 趋势必须解释为什么升温、分化、降温、成熟、风险增强或反证增强。
- 趋势应尽量关联商业信号、Priority Engine、机会判断和观点校准。
- 每个重要趋势都应保留反证观察点。

### Builders 观点 / The Point

- The Point 不作为 V2.1 一级栏目；前台归入 `商业信号 > 前沿观点` 和各栏目观点校准模块。
- Builders 观点追踪一线 AI 建造者观点，不做网红观点搬运。
- 原文和译文不能用摘要、观澜判断或截断句替代。
- X/Twitter 短链 `t.co` 不应出现在前台正文。
- YouTube speaker/timecode 标记不应混入观点正文。
- Blog / YouTube 全文入库需要明确来源与授权边界。
- 同一人物多观点可以合并展示，但每条观点必须保留独立来源链接。

### Intelligence Data

- `data-agent` 已升级为 Intelligence Data Agent。
- 它负责商业信号 / 趋势追踪 / 机会判断 / 前沿观点 / Relation / Quality Intelligence。
- 它可以判断内容是否具备进入系统的条件，但不替代 Strategy、PM、Copy、UI/UE、Dev、QA 或 Workflow。
- 关系检查硬错误必须修复；软提醒必须有处理路径。

### PM 模块治理

- PM Agent 构建和治理产品功能模块时必须秉持“宁缺毋滥”：默认不新增模块，也不默认保留既有模块。
- 新增、保留、强化、合并、后台化、隐藏或淘汰栏目、页面、功能入口、后台模块、关系视图、会员能力或内容形态前，必须输出模块决策表。
- 模块决策必须检查五项：商业价值、与现有模块是否重复、目标用户是否接受并愿意付费、使用路径是否顺畅、是否贴合观澜AI战略核心。
- 新模块任一项不通过，默认不新增；既有模块任一项长期不通过，必须提出优化、合并、后台化、隐藏、延期复核或淘汰方案。
- 与今日观察、商业信号、前沿观点、机会判断、趋势追踪、商业内参、Priority Engine、Tags 或 Admin 存在交叉的需求，必须先判断能否通过优化、合并或淘汰已有模块解决。
- 用户反馈“功能太散、入口太多、解释成本高、像内部系统、不是商业决策者会用的东西”时，调度中枢应优先判定为 PM 模块治理失效，而不是让 Dev 继续加页面。
- 产品功能类任务必须先过 PM 新增功能门禁，不得直接进入 Dev。门禁采用 WAVE 评分：Worth paying for、Alignment、Validation、Experience / Effort。
- 观澜AI需要的功能必须提升判断质量、判断效率、证据可信度、机会验证能力、复访与付费理由、后台运营质量或克制的商业化转化。
- 观澜AI默认拒绝 AI 新闻流、工具导航、公司排行榜、项目黄页、开发者社区、泛仪表盘、泛 AI 助手、只增加信息量但不提高判断质量的功能。

### 文案风格

- 文案必须言之有物、简洁克制、精炼有力。
- 不讲废话，不堆砌“赋能、颠覆、确定性、爆发、稳赚”等表达。
- 有观点，但必须保留证据边界和不确定性。
- 不替客户下最终经营、投资或合作判断。

### UI / UE

- 页面服务于高端商业判断，不服务于技术炫耀或信息堆砌。
- 2026-05-07 用户确认 V2 VI 以 `方案 A：极简澜线型` 为长期硬基线。以后官网、页面、可能新增的栏目、组件、符号、元素、商业内参视觉和动效，都必须按 `agent-workflow/v2/v2-vi-design-direction.md` 执行。
- 已给定的 Logo 与元素不得随意更改：一条远方地平线、三道流动澜线、`观澜 AI` 中文字标、`WAVESIGHT AI` 英文字标、商业内参符号系统、信息元素和动效规范均视为锁定资产。
- 后续执行窗口不得自行改变澜线数量、去掉金色地平线、添加信号点 / 雷达圈 / 眼睛 / 机器人 / 箭头 / 蓝紫科技渐变，或将 Logo、符号和动效重画成另一套品牌母题。确需调整时，必须先由 PM / UI / Verification 评审并判断是否需要用户确认。
- 每屏只完成一个主要任务。
- 首页只承载品牌价值、核心判断入口和少量代表内容。
- 详情页标题、字号、正文、卡片和移动端布局应保持一致。
- 普通页面不出现后台痕迹。
- `ui-ue-agent` 已升级为 `UI / UE Design Director`，不新增第九个长期 Agent。
- Design Director 不是单页修补者，而是全站视觉总监。全站 UI、首页、栏目、Admin、海报任务必须先定 Art Direction、页面母版、字体/间距系统和视觉资产规则，再进入 Dev。
- 用户反馈“不高级、栏目不紧凑、海报不符合网站调性、排版杂乱”时，默认判定为 Art Direction 和设计母版缺失，不应继续局部调 CSS。
- 海报或首屏视觉必须表达商业判断样张、情报桌面或信号到机会关系，不得只是抽象 AI 氛围图、赛博背景、机器人头像或通用 SaaS 插画。
- UI/UE 每一次页面设计都必须给出审美标准，包含 Style Purity、Proportion & Rhythm、Color Sophistication、Craftsmanship、Emotional Resonance 与 Squint Test。
- 风格美观评分是创作参考和调度复核依据；总分低于 70、任一单项低于 10、Squint Test 不通过或出现审美阻塞项时，必须重做，不得进入 Dev 或不得标记 `accepted`。
- 页面类派发单和 closeout 必须包含 UI/UE 审美标准与桌面证据表；调度中枢不得代补评分，不得接受缺少桌面证据的页面任务。
- 当前 V2.1 采用桌面端优先：页面已实现或已有可访问页面时，必须包含桌面截图、逐项扣分原因、必须重做清单和 Dev 实现偏差清单；移动端只有在派发单明确要求专项时才作为硬阻塞。
- 页面类型通过线：首页 / 全站母版 / 核心首屏 / 海报 / 视觉资产 >=85；一级栏目页 / 详情页 / 会员页 >=80；Admin 工作台和后台模块 >=75；任一单项低于 14、Squint Test 不通过或有审美阻塞项时必须重做。
- UI/UE 不得只自评；没有桌面截图、扣分原因、重做清单或 Dev 偏差清单的风格美观质检无效，调度中枢必须退回补齐或重做。

### 自动化

- 旧 The Point 自动化不再作为当前生产线；V2.1 当前自动化以 `guanlan-daily-monitor` 六线程链路为准。
- 自动化失败时不得生成空文件，不得覆盖上一版有效数据。
- 自动化异常必须记录失败原因、尝试过的路径和降级结果。
- `needs_review` 内容不得进入首页精选。

### 调度中枢窗口

- 用户已确认建立“当前窗口分配任务、独立窗口执行、回到当前窗口收口”的机制。
- 调度中枢窗口只做任务拆分、派发、收口验收和进度回填，不吞掉大任务执行。
- 每个独立执行窗口必须先读派发单，只处理派发单范围，结束前写 UTF-8 closeout 文件。
- 收口文件默认路径为 `agent-workflow/reports/<TASK-ID>-closeout.md`。
- 调度中枢收到 `收口：<path>` 后，必须核对派发单、改动范围、Quality Gates、自动化影响和进度回填，再标记任务通过。
- 常用口令为 `派发：<任务描述>`、`收口：<closeout 文件路径>`、`状态`、`下一批`、`阻塞：<task-id> <原因>`。
- 2026-05-04 已升级为任务看板驱动：`dispatch-board.md` 预生成任务卡和派发单，用户可直接说 `执行：P0-1` 领取任务。
- 页面类任务必须在派发单中加入 UI/UE 页面规范表、Dev 按表实现、QA 独立截图/测量验收三段硬规则；缺少任一项不得在调度中枢标记为 `accepted`。
- 首页、栏目页、详情页、Admin 和会员页任务不得只写“读取 DESIGN.md”；必须写清标题位置、字号、行高、模块起点、间距、组件克制规则、桌面截图验收和阻塞标准。移动端专项只有在派发单明确要求时才作为硬阻塞。
- 用户反馈“页面粗糙、简陋、栏目间距不一、标题位置不一、字体大小随意、排版丑陋”时，调度中枢应优先判定为页面类派发闸门失效，而不是单点样式问题；后续同类任务必须退回 UI/UE 规范表和 QA 独立验收。
- 页面类任务只要涉及可见文案，也必须加入 Copy 文案规范表、Dev 按表落地、QA 禁用语/判断边界/容器适配验收；缺少任一项不得标记 `accepted`。
- 文案类任务不得只写“读取 COPY.md”；必须写清用户任务、目标理解、替换前问题、替换后文案、禁用语检查、判断边界、标题长度与 UI 容器适配。
- 用户反馈“文案不规范、内部话术、说服感太强、标题难看、口号化、空泛”时，调度中枢应优先判定为文案派发闸门失效；后续同类任务必须退回 Copy 文案规范表和 QA 文案验收。
- 2026-05-10 已新增 `dispatch-state-reconciliation.md`：当看板、feature、progress、handoff 和 closeout 状态冲突时，按“用户最新裁决 -> 最新 closeout -> dispatch-board -> feature_list -> handoff 顶部 -> progress 顶部 -> 历史章节”判定。
- `failed / not-accepted / closed` 任务不得继续用原 Task ID 派发；stage summary、截图、reference mockups、local-site-quality-pass 和局部实现只能作为失败样本或复盘材料。
- 被 failed 任务合并过的子项不得自动视为已解决，必须改为 `review / needs-successor-after-failed-merge` 或新建继任任务。
- `dispatch-board.md` 的“使用建议”必须随收口刷新，不得继续推荐已经 completed / accepted、failed、abandoned、stopped、void 或被 failed 任务吞并的任务。
- V2-only 阶段，页面、样式、脚本、内容生成和截图验收默认入口是 `01-SiteV2/`；旧 `04-Site/` 已从当前仓库移除，不再作为参考源。

### 长期 Agent 能力训练

- 八个长期 Agent 的能力边界必须同时说明“应该擅长什么”和“不该做什么”，避免用一个岗位吞掉 Strategy、PM、Data、UI/UE、Copy、Dev、QA、Workflow 的职责。
- 每个 Agent 接到任务后必须先读本岗位说明、通用 handoff、相关治理文件和对应产品规范；不能只凭对话记忆执行。
- 每个 Agent 的输出必须包含可交接信息：输入依据、完成事项、改动或建议、验收标准、未决风险、下一步 owner。
- 涉及重大栏目、权限、数据模型、自动化、部署、商业化或长期 agent 职责变化时，必须先走 Plan-first，不直接进入实现。
- QA 必须保持独立验收；Dev 可以运行自检，但不能替代 QA 的发布判断。
- Workflow 负责把训练规则、收口文件和后续任务沉淀到 `agent-workflow`，不得只停留在聊天回复。

- UI/UE Agent 使用 `taste-skill` 时必须先适配观澜AI：克制商业情报调性优先，不默认采用强动效、持续动画、大圆角 Bento、玻璃拟态、Framer Motion、GSAP 或 ThreeJS。
- `taste-skill` 可吸收的长期规则包括：依赖先检查、移动端避免不稳定全屏高度、优先 CSS Grid、避免 AI 紫蓝渐变、避免过度卡片化、补齐 loading / empty / error / active 状态、动画只用 transform 和 opacity。
- 外部 skill 安装或学习前后，Workflow / QA 应按 `skill-vetter` 或安全扫描规则做静态检查；不得执行被审查 skill 的脚本或远程命令。

## 4. 更新流程

1. 任一 agent 发现可复用经验。
2. 写入本文件对应章节，或交给 Workflow Agent 汇总。
3. 如果经验影响 PRD、产品模型或检查脚本，需要同步更新对应文件。
4. 如果经验可自动化，交给 Dev Agent 增加检查。
5. QA Agent 在后续验收中检查该规则是否被执行。

## 5. 禁止事项

- 不把一次性偏好写成长期规则。
- 不记录没有上下文的口号。
- 不把内部争议写成已确认事实。
- 不用本文件替代 `strategy-single-source.md`、PRD 或正式产品模型。
