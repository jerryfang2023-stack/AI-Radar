# 栏目决策记录

## 2026-05-02：设立 PM Agent

决策：

- 观澜AI项目需要固定 PM Agent。
- PM Agent 负责栏目规划、产品功能规划、PRD、路线图和验收。

原因：

- 项目已经从网站页面优化升级为长期运营系统。
- 需要避免栏目越做越多、内容边界混乱、开发任务无法验收。

## 2026-05-02：Scoring 对外升级为 Priorities

状态：已被后续决策取代。

决策：

- `Scoring` 不再作为前台一级栏目名。
- `Scoring` 保留为后台评分模型和数据源。
- 对外使用 `Priorities` 表达“机会优先级判断”。

原因：

- 用户关心“哪个机会值得优先验证”，而不是内部评分过程。
- 评分对象应从公司名转为机会方向、赛道或场景。

后续任务：

- 修改前台导航和页面文案。
- 修复评分数据中的公司榜单化表达。
- 建立 Priority 与 Opportunity 的关联。

## 2026-05-02：Priority Engine 合并进入 Opportunities

决策：

- 前台不再保留独立 `Scoring` 或 `Priorities` 一级栏目。
- `02-Scoring/` 保留为后台数据源。
- `Priority Engine` 作为后台评分和优先级判断能力。
- `Opportunities` 承接评分、优先级、证据等级、趋势状态和下一步验证动作。

原因：

- 用户关心最终可验证的机会，而不是单独阅读评分榜。
- 没有评分、证据等级或验证动作的内容，不应作为正式机会展示。
- 前台栏目越少，越能突出 Daily Brief、Signals、Opportunities、Trends 的决策闭环。

后续任务：

- 旧 `scoring.html` 需要迁移、隐藏或降级为后台/兼容入口。
- Opportunity Card 字段需要补齐评分、证据等级、趋势状态和验证动作。
- 文档中残留的前台 `Priorities` 表达需要统一清理。

## 2026-05-02：Method 暂不作为前台栏目

决策：

- Method 不进入当前前台导航。
- 方法论作为内部产品资产保留，用于支撑样例报告、销售沟通、产品说明和未来信任内容。

原因：

- 当前最重要的是建立付费级 Daily Brief、机会库和趋势判断，而不是先解释方法论。
- 方法论可以成为未来信任内容，但不应分散第一阶段导航。

## 2026-05-02：Admin 独立隐藏

决策：

- Admin 不进入普通用户导航。
- 普通端不出现编辑、同步、恢复原始数据、访问状态等后台痕迹。

原因：

- 面向商业决策者的前台必须保持高端、克制、可信。
- 后台操作会破坏信任感和产品边界。
