## 机会摘要

- **一句话机会定义：** 面向 AI Agent 的网页执行基础设施，让 Agent 能够稳定地浏览网页、检索资料、执行多步骤研究、处理表单和业务系统，解决"Agent 能对话但不能执行"的瓶颈。
- **为什么是现在：** 4-29 雷达中 Parallel Web Systems（前 Twitter CEO 创办）完成 1 亿美元 B 轮、估值 20 亿美元，已有 10 万开发者使用，被 Harvey 等 AI 原生公司采用。这说明资本开始从"前台 Agent 应用"向"Agent 基础设施"下注，Agent 需要底层网页执行、权限管理、任务监控和结果验证能力。
- **这是谁的问题：** AI 原生创业公司、企业 AI 团队、法律科技公司、保险公司、研究机构——任何需要 Agent 在真实网页环境中执行复杂任务的开发者和企业。
- **不做的代价：** 基础设施方向技术门槛高、商业转化周期长，不适合轻资产创业者。但如果你在构建依赖网页任务执行的 Agent 应用（如情报采集、竞品监测），底层执行能力将成为关键瓶颈。

## 问题与解决方案

- **原始痛点：** AI Agent 在真实网页环境中执行复杂任务时极不稳定——反爬机制、动态渲染、登录墙、多步骤表单、跨站点数据融合等问题导致成功率低、成本高。通用浏览器自动化工具（Playwright/Puppeteer）需要大量工程配置，不适合 AI Agent 直接调用。
- **当前替代方案：** 通用浏览器自动化工具（Playwright/Puppeteer/Selenium）需要大量工程配置；人工执行（效率低、成本高）；RPA 工具（不是为 AI Agent 设计，缺乏灵活性）。缺少专门为 AI Agent 设计的网页执行基础设施。
- **AI解决方案类型：** AI Agent 网页执行基础设施（浏览器自动化 + 反爬对抗 + 任务编排 + 结果验证）
- **需要什么数据/系统前提：** 需要稳定的浏览器运行时环境和代理网络；需要处理目标网站的登录、反爬和动态渲染；需要任务定义和结果验证体系。
- **典型实施路径：** 定义 Agent 需要执行的网页任务类型 → 搭建浏览器运行时环境 → 处理反爬/认证/登录 → 任务编排和状态管理 → 结果提取和验证 → 异常处理和重试机制。

## ROI假设

- **可能节省的成本：** 替代人工执行网页研究和数据采集任务，可节省 70-90% 的人力成本。
- **可能提升的收入：** 底层基础设施的商业模式是 API 调用费和企业订阅，定价空间取决于执行任务的复杂度和成功率。
- **可能缩短的时间：** 一个需要人工 2-3 小时的网页研究任务，Agent 可在 5-15 分钟内完成。
- **回本周期假设：** 基础设施方向商业转化周期较长（12-18 个月），需要先积累开发者规模和任务执行数据。
- **适合先做试点还是直接规模化：** 不适合从零做底层基础设施，更适合基于现有工具（agent-browser、Playwright）构建 Agent 网页任务的应用层服务。

## 代表性案例

- **案例公司：** Parallel Web Systems（前 Twitter CEO 创办，1 亿美元 B 轮，估值 20 亿美元）
- **发生了什么：** Parallel 完成 1 亿美元 B 轮融资，由 Sequoia Capital 领投，Kleiner Perkins、Index Ventures、Khosla Ventures 参与。公司为 AI Agent 提供面向网页任务、深度研究、保险理赔、法律分析等场景的基础设施，已有 10 万开发者使用，Harvey 等 AI 原生公司为其客户。
- **关键数据：** 20 亿美元估值、10 万开发者、头部 AI 原生公司采用。
- **成功前提：** 开发者体验和任务成功率是核心壁垒；需要持续投资反爬对抗和浏览器环境维护；企业客户需要 SLA 和合规保障。
- **失败风险：** 中国开放网页环境的反爬和合规限制更严格；基础设施方向需要持续资金支持（烧钱期长）；大厂（Google、Microsoft）可能推出类似能力；一旦被大厂覆盖，独立基础设施公司的生存空间会被压缩。

## 典型供给方

- **海外代表：** Parallel Web Systems（20亿美元估值）、Browserbase（浏览器基础设施）、Stagehand（Agent 浏览器 SDK）
- **国内代表：** 无直接对标。agent-browser（开源工具）、Playwright（底层框架）、RPA 厂商（来也、弘玑）——但都不是 Agent 原生的网页执行基础设施
- **自建/采购/合作建议：** 不建议自建底层基础设施。如果需要在应用层使用网页 Agent 能力，建议基于 agent-browser（开源）或 Playwright 封装，或关注 Parallel/Browserbase 的 API 服务。
- **供应商筛选标准：** 评估 API 稳定性、任务成功率、反爬能力、定价模式和 SLA。

## 风险与反证

- **最大风险：** 中国互联网环境的反爬、登录墙、验证码和合规要求比海外更严格，Agent 网页执行成功率可能大幅下降。大厂可能通过浏览器内核层封锁 Agent 行为。
- **哪类企业不适合：** 不需要网页任务执行的纯内容/工具类 Agent；对网页执行稳定性和成功率要求极高的金融交易场景。
- **反对观点：** "Agent 网页执行不稳定，不如 API 可靠"——但对没有开放 API 的网站来说，网页执行是唯一选择。问题不是要不要用，而是如何提升成功率和可观测性。
- **需要进一步验证的问题：** 在国内主流网站（企微后台、抖音、小红书、美团商家后台）上测试 Agent 网页执行的成功率。如果核心场景的成功率低于 70%，应用层产品体验将难以保障。

## 行动建议

- **建议优先级：** 中
- **适合谁发起：** 不推荐单人/小团队从零做底层基础设施。如果你在构建依赖网页 Agent 的应用（如情报采集、竞品监测），建议先用 agent-browser 或 Playwright 快速验证，再评估是否需要定制基础设施。
- **下一步最小验证动作：** 选 3 个国内目标网站，用 agent-browser 测试 Agent 网页任务的执行成功率。如果核心场景成功率 > 80%，可考虑封装为行业垂直服务；如果 < 50%，建议等待基础设施成熟或选择有 API 的替代方案。
- **建议预算范围：** 技术验证 + 场景测试 1-2 万。
