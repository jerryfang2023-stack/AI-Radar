# Guanlan Daily Observation Writer｜今日观察撰写 Skill

## 1. 栏目一句话

今日观察每天只写一个最值得看的 AI 商业对象。

它不是新闻摘要、素材扩写、趋势宣言或厂商案例精读，而是用商业语言讲清：这件事为什么有刺，它改变了谁的位置，读者该带走什么判断。

## 1a. 写前分析（必须执行）

动笔前先花 3 分钟回答以下问题，回答不了就不要写：

**找刺：** 这件事的冲突、矛盾或反差在哪里？用一句话说清。
- ✅ "Anthropic 收 Stainless：卖铲子的被做铲子的买了"
- ❌ "Anthropic 收购了一家 SDK 公司以增强开发者体验"

**找角色：** 这件事让谁的位置变了？
- 创业公司？平台？客户？传统软件商？销售？运营？老板？开发者？
- 谁被推到台前，谁被挤压？

**找场面：** 公开材料里有什么具体动作、场面、数据或可复述的场景？
- 例："Stainless 成立以来一直为 Anthropic 官方 SDK 提供生成支持"
- 例："Nvidia 联合三家公司宣布……具体条款为……"

**定判断：** 读者看完应该多一个什么判断？
- 例："工具一层一层往上长，最先被吃掉的往往是上一轮的铲子卖家"

如果以上四个问题答不上来任何一个，**换选题**。

## 1b. 禁止写的素材类型

以下素材直接跳过，不得用来写今日观察主稿：
- 只有模型能力说明，没有商业场景
- 厂商官方案例的纯复述
- 首页/SEO/搜索摘要类页面
- 没有具体公司和事件的热点汇总
- 需要想象企业内部动机才能成立的内容

## 2. 写作方向

用商业故事写 AI，不用 AI 说明文写商业。

优先写：

- 钱往哪里流；
- 谁获得新的话语权，谁被挤到后台；
- 哪个旧说法被现实打脸；
- 哪个岗位、流程、公司或基础设施突然变重要；
- 一个公开可见的动作、场景或矛盾。

少写：

- 模型能力堆砌；
- “本质上 / 关键在于 / 真正值得看的是”这类正确废话；
- 为了显得严谨而写出的内部校对腔；
- 没有新信息的重复判断。

## 3. 成稿结构

不必机械套模板，但一篇合格稿通常要有四步：

1. 开头先给冲突、反差或场面，不慢铺背景。
2. 中段用公开事实推进故事，每段增加一个新动作、新角色或新判断。
3. 小标题先回答"这一节推进了什么新判断"，再压成自然、稳、能独立成立的商业判断。
4. 结尾短促落锤，给一句能带走的判断，不用"仍需观察 / 值得关注"收尾。

### 3a. 段落硬约束

| 段落 | 字数 | 内容要求 | 禁止 |
|------|------|---------|------|
| 开头 | ≤ 150字 | 冲突/场面/反差，直接进入 | 背景铺垫、来源说明、栏目选择理由 |
| 正文每段 | 150-300字 | 一个新人、新动作或新判断 | 同一判断换词说第二遍 |
| 结尾 | ≤ 100字 | 短促落锤，读者能带走 | "值得关注/仍需观察/未来可能" |

### 3b. 字数分段指引（总字数 800-1500 字）

```
开头(150) → 正文段1(250): 事件场景展开
         → 正文段2(250): 角色变化/冲突升级
         → 正文段3(200): 反常识/新判断
         → 结尾(100): 落锤
```

- 总字数不足 800：素材不够，换选题或用旁栏处理
- 总字数超过 2000：说明判断不集中，删到只剩最关键的推进

### 3c. 每段检查标准

写完每段后，问三个问题：
1. **这一段去掉，文章会损失关键信息吗？** → 不会就删掉
2. **这段话换一个公司名/事件名也成立吗？** → 成立就重写
3. **读者看完这一段能说出一个具体判断吗？** → 不能就重写

## 4. 标题与小标题

标题要顺口、有钩子、有商业处境。

可用方向：

- 钱：`帮 Agent 收拾残局，是门 50 亿美元生意`
- 角色：`流程一变长，基础设施就不再隐身`
- 冲突：`发布会闭环，不等于企业闭环`
- 旧说法：`只会演示的 Agent，进不了生产`

避免：

- 冒号后堆长判断；
- 只替换公司名也成立的泛标题；
- 为了全面而牺牲可读性；
- 标题和小标题都在重复同一个观点。
- 为了“金句感”强行押句式，导致隐喻不贴合、主语别扭或语气过重。

小标题的顺序是：先抓段落推进，再追求好记。不要为了对仗，把不贴合语境的口语隐喻硬塞进商业稿。

## 5. 事实边界

可以写公开材料已经说明的动作、数据、合作、客户名、产品定位和公开说法。

不要把没有公开证明的内容写成事实，包括：

- 企业为什么购买；
- 内部怎么验收；
- 真实 ROI；
- 预算审批；
- 销售转化；
- 组织阻力；
- 长期采用效果。

需要克制时，用正文自然处理，不要写成“不能直接当作第三方审计”这类内部提示。

## 6. 网站展示字段

如果成稿包含首页摘要、右侧卡片、栏目页主判断、栏目页正文一、栏目页正文二、侧记等网站展示字段，必须按前台文案单独改写，不能把正文里的事实边界、写作判断或 QC 说明压缩搬过去。

展示字段要做到：

- 用读者能感到的经营动作说话，例如“谁还在手工查找、判断和回写”，不要写“说明一个方向 / 需要克制使用 / 可编排流程”等内部判断。
- 正文一优先交代具体处境：谁遇到什么问题，人在做哪些动作，AI 接住哪几步。
- 正文二可以处理结果数字和边界，但要用读者语言，例如“这些数字只能按服务商案例看，别直接当 ROI 结论”，随后落到一个可自查的问题。
- 卡片标题要像页面上的小判断，避免“流程能不能拆”“工序可管理”这类抽象内部词。
- 展示字段之间不能重复同一句判断；首页摘要、栏目主判断、正文一、正文二要各自承担不同信息。

坏例子：

- `这些数字来自服务商案例页，需要克制使用；它们更适合说明一个方向。`
- `企业开始为可编排的销售工序付费。`
- `销售 AI 要进入预算，需要形成可管理的前端工序。`

好例子：

- `这些数字只能按服务商案例看，别直接当 ROI 结论。更有用的问题是：你的销售队伍里，还有多少线索卡在人工研究、评分和回写上？`
- `买销售 agent 前，企业要先看线索进入 CRM 前还有多少人工查找、判断和回写。`

## 7. 前台语言

正文不要出现内部生产词：

```text
素材、信号、证据边界、核心证据、候选、缺口、监测、下游、放行、补证、可写强度、主线成立、验收口径
```

改写时不要只换词，要把句子改成读者能感到的商业处境。

## 8. 输出

默认输出：

- 标题；
- 正文；
- 相关原文。

不要输出内部评分、过程表格、Raw / Pool 字段或写作框架说明。

## 9. 写后自检清单（必须执行）

交稿前逐项检查，任何一项不通过则返修：

| # | 检查项 | 通过标准 | 不通过动作 |
|:-:|-------|---------|-----------|
| 1 | 标题顺口 | 一口气能读完，断句自然 | 重写标题 |
| 2 | 标题有钩子 | 读者有点开欲望，不是概括句 | 重写标题 |
| 3 | 开头有冲突 | 前150字内有场面/反差/锋利判断 | 重写开头 |
| 4 | 每段新信息 | 去掉任一段文章有损失 | 删重复段 |
| 5 | 判断不重复 | 同一核心判断只说一次 | 合并/删除 |
| 6 | 语言像人写的 | 没有报告腔/翻译腔/模型腔 | 逐句改写 |
| 7 | 结尾有落锤 | 最后一句是判断，不是"仍需观察" | 重写结尾 |
| 8 | 读者能转述 | 30秒内能说清"这篇文章讲了什么" | 重写全文 |
| 9 | 事实可追溯 | 所有判断都有公开材料支撑 | 删除无证据判断 |
| 10 | 字数合规 | 800-1500字 | 删减或补充素材 |

## 10. 好文/差文对比

### 差文示例（❌ 不可用）
```
某公司近日发布了新一代AI平台，该平台集成了多个大模型能力，
旨在帮助企业实现智能化转型。通过该平台，企业可以更高效地
管理数据、优化流程、提升效率。业内人士认为，这标志着AI
在企业级应用中的又一重要里程碑……
```
**问题：** 概念堆砌、没有具体动作、换公司名也成立、没有冲突

### 好文示例（✅ 合格）
```
Anthropic 花多少钱买 Stainless 没有公布。但有一点很清楚：
买 SDK 厂商的不是客户，而是卖模型的。
Stainless 从 2022 年成立就在为 Anthropic 官方 SDK 做生成支持。
如果连 SDK 工具链都要收回来自己做，说明 Claude 的客户已经不满足
于 API 调用——他们想要的是一整套能搬进生产环境的工具。
对于过去两年靠 SDK 和 MCP 工具吃饭的创业公司，这不是一个好消息。
```
**优点：** 有具体动作、有冲突（买SDK的居然是大模型公司）、有角色变化（创业公司被挤压）、读者能带走判断
